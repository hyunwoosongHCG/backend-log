Dirty Tracking과 Partial Writes

레슨 020 changed? · attribute_changed? · partial_writes ~15분
🎯 레슨 19의 "updated_at은 저장할 때마다 자동 갱신된다"에 예외가 있다는 걸 완성한다 — MISSION.md

질문에서 시작한다

레슨 19에서 update를 호출하면 updated_at이 항상 자동으로 갱신된다고 배웠다. 그런데 ppback의 check_ins_controller.rb에는 이런 코드가 있다. 

def create_check_in_revision(check_in_details)
  data = check_in_details.each_with_object([]) do |check_in_detail, arr|
    key_result = @objective.key_results.find(check_in_detail[:key_result_id])
    next if key_result.value == check_in_detail[:value]   # ← 이 줄
    ...
  end
end

"값이 같으면 건너뛴다"는 가드가 왜 필요할까? 그냥 update를 호출해도 되지 않나? 이 질문의 답이 Dirty Tracking이다.

파트 1: 값이 실제로 안 바뀌면 저장 자체가 일어나지 않는다

ActiveRecord는 레코드의 각 속성(attribute)에 대해 "저장 전 값과 지금 값이 다른가"를 항상 추적하고 있다. 이걸 Dirty Tracking이라 부른다. 

key_result = KeyResult.find(1)
key_result.value          # => 50

key_result.value = 50     # 기존 값과 동일한 값을 대입
key_result.changed?       # => false  ← "바뀐 게 없다"

key_result.value = 80     # 실제로 다른 값
key_result.changed?       # => true   ← "바뀐 게 있다"

그리고 changed?가 false인 상태에서 save/update를 호출하면, ActiveRecord는 UPDATE SQL 자체를 DB에 보내지 않는다. (이 최적화를 partial_writes라고 부른다 — 바뀐 컬럼만 골라서, 그리고 바뀐 게 없으면 아예 안 보낸다.) 

key_result.update(value: 50)  # 기존 값과 동일
# → SQL: 아무것도 안 나감
key_result.updated_at         # → 그대로! (레슨 19의 자동 갱신이 "안 일어남")

key_result.update(value: 80)  # 실제로 다른 값
# → SQL: UPDATE key_results SET value = 80, updated_at = '...' WHERE id = 1
key_result.updated_at         # → 방금 시각으로 갱신
프론트에서 폼을 다룰 때 "실제로 값이 바뀐 필드만 서버에 diff로 보낸다"는 것과 같은 원리다.

if (formValue !== initialValue) { submit() }

Rails는 이 비교를 레코드 저장 단위로 자동으로 해준다 — 그래서 "저장을 호출했다"와 "실제로 DB에 UPDATE가 나갔다"는 서로 다른 이야기다.
Q1. key_result.value가 이미 50인 상태에서 key_result.update(value: 50)을 호출했다. 어떻게 되는가?

파트 2: 그래서 저 가드 코드가 있는 것이다

다시 처음 코드로 돌아가 보자.

key_result = @objective.key_results.find(check_in_detail[:key_result_id])
next if key_result.value == check_in_detail[:value]

이 가드가 없다면, 사용자가 이미 50인 값을 다시 50으로 "체크인"했을 때도 CheckInRevision(승인 대기 이력)이 생성되고, 매니저에게 승인 알림이 가고, History 레코드가 쌓인다 — 아무 의미 없는 노이즈가 계속 생기는 것이다.

Dirty Tracking이 updated_at 자동 갱신을 스킵해주는 것과 별개로, "애초에 의미 없는 변경이면 이력 자체를 만들지 말자"는 판단은 프레임워크가 대신해주지 않는다. 그래서 == 비교를 개발자가 직접 짜둔 것이다.

① 사용자가 체크인 값 입력 (예: 50 → 50, 안 바뀜)
↓
② next if key_result.value == check_in_detail[:value] → 여기서 걸러짐
↓ (만약 이 가드가 없었다면)
③ CheckInRevision 생성 → 매니저 승인 대기 → History 기록... 전부 의미 없이 발생
Q2. Dirty Tracking이 있는데도 check_ins_controller.rb가 value == check_in_detail[:value]를 직접 비교하는 이유는?

알아두면 좋은 관련 메서드

메서드 의미
record.changed? 저장 전, 어떤 속성이든 바뀐 게 있는가 (bool)
record.value_changed? 저장 전, value 속성만 바뀌었는가
record.saved_change_to_value? 저장 후, 방금 저장에서 value가 바뀌었는가
