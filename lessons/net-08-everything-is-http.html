모든 것이 HTTP

핵심 한 줄

오늘날 인터넷에서 주고받는 거의 모든 데이터는 HTTP 위에서 흐른다. HTML 문서뿐 아니라 이미지·영상·JSON·파일·심지어 서버 간 통신까지, HTTP는 현대 인터넷의 공용어(lingua franca)가 됐다.

HTTP(HyperText Transfer Protocol)의 역사

HTTP는 이름 그대로 처음에는 하이퍼텍스트(HyperText), 즉 HTML 문서를 링크로 연결해 전달하기 위해 만들어졌다. 팀 버너스리(Tim Berners-Lee)가 1989년 CERN에서 제안한 이 규격은 "서로 다른 컴퓨터의 문서끼리 링크로 이어지게 하자"는 단순한 아이디어에서 출발했다.

그런데 시간이 지나면서 HTTP는 문서 전달 그 이상으로 확장됐다. 지금은 아래 모든 것이 HTTP를 통해 오간다.

  • HTML, TEXT — 웹 페이지와 일반 텍스트
  • IMAGE, 오디오, 영상 — 미디어 파일
  • 파일 — 업로드·다운로드
  • JSON, XML — API 응답 (프론트가 매일 쓰는 바로 그것)
  • 서버 → 서버 통신 — 마이크로서비스, 내부 API 호출
프론트에서 fetch()나 axios.get()으로 서버를 호출할 때, 그 안에서 흐르는 것이 전부 HTTP다. JSON을 주고받든, 이미지를 다운로드하든, 파일을 올리든 — 전부 같은 프로토콜. "HTTP = 웹 페이지 다운로드"라는 좁은 이미지를 버리고 "인터넷의 공용 운반 컨테이너"로 다시 이해하면 된다.

HTTP 버전별 기반 프로토콜

HTTP는 버전마다 아래 전송 계층(Transport Layer)이 다르다.

버전 기반 프로토콜 특징
HTTP/1.1 TCP 현재 가장 널리 쓰임. 요청·응답 순서 보장
HTTP/2 TCP 한 연결에서 여러 요청을 동시에(멀티플렉싱). 성능 향상
HTTP/3 UDP (QUIC) UDP를 애플리케이션 레벨에서 재설계. 더 빠른 연결 수립

TCP와 UDP — 딱 한 줄씩

TCP(Transmission Control Protocol): 데이터가 순서대로, 빠짐없이 도착했는지 확인하는 "신뢰형" 프로토콜. 3-way handshake(SYN → SYN-ACK → ACK)로 연결을 맺고 나서야 데이터를 보낸다. 안전하지만 그만큼 연결 수립에 시간이 든다.

UDP(User Datagram Protocol): 확인 없이 일단 던지는 "빠른 전송형" 프로토콜. 신뢰성은 낮지만 속도가 빠르다. HTTP/3는 UDP 위에 QUIC이라는 레이어를 올려 "빠른 연결 + 신뢰성"을 모두 애플리케이션 단에서 직접 구현했다.

현재 현실

아직은 HTTP/1.1이 주류지만, HTTP/2와 HTTP/3 비중이 꾸준히 늘고 있다. Chrome DevTools의 Network 탭 "Protocol" 컬럼을 보면 h2(HTTP/2), h3(HTTP/3)를 실제로 확인할 수 있다.

정리

  • HTTP = HTML 전용이 아니라 인터넷의 범용 운반 수단
  • JSON API 응답, 이미지, 파일, 서버 간 통신 — 모두 HTTP
  • HTTP/1.1·2는 TCP 기반, HTTP/3는 UDP(QUIC) 기반
  • 현재는 HTTP/1.1이 주류, HTTP/2·3 점유율 증가 중
