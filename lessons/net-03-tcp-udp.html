🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · 인터넷 네트워크 — IP의 한계를 TCP가 어떻게 극복하는지, UDP는 언제 쓰는지 이해한다

핵심 한 줄

TCP(Transmission Control Protocol)는 IP 위에 올라타서 연결 확인 → 전달 보증 → 순서 보장을 추가한 신뢰성 있는 프로토콜이고, UDP(User Datagram Protocol)는 그 보장을 모두 포기하는 대신 빠른 속도를 얻은 경량 프로토콜이다. 현재 대부분의 웹 통신은 TCP를 사용한다.

왜 TCP/UDP가 필요한가

앞 강의에서 IP의 세 가지 한계를 배웠다: 비연결성, 비신뢰성, 포트 구분 불가. TCP/UDP는 애플리케이션 계층의 데이터(payload)에 TCP/UDP 정보를 추가(캡슐화)해서 이 한계를 보완하는 트랜스포트 계층 프로토콜이다.

IP를 "주소만 적힌 택배 봉투"라 하면, TCP는 "수령 확인 서명 + 분실 시 재발송 보장 + 순서 번호가 붙은 등기 택배"다. 반면 UDP는 "주소는 있지만 수령 확인도 순서도 없는 일반 엽서" — 대신 수수료가 저렴하고 빠르다.

TCP (전송 제어 프로토콜)

TCP의 특징

  • 연결 지향 — 데이터 전송 전 3-way handshake로 양측 연결을 먼저 확인
  • 데이터 전달 보증 — 수신 측이 "받았다(ACK)"를 응답, 없으면 재전송
  • 순서 보장 — 패킷이 순서가 틀리면 재전송 요청
  • 신뢰 가능 — 위 세 가지 덕분에 현재 대부분의 웹 통신이 TCP 사용

3-way handshake (3방향 악수)

TCP는 데이터를 보내기 전, 반드시 이 3단계 과정으로 가상 연결을 확립한다. 실제 물리 전용선이 생기는 게 아니라 "내가 보내면 네가 받을 수 있는지 확인" 수준의 약속이다.

1. SYN (접속 요청)
클라이언트 → 서버: "나 연결하고 싶어, 들려?"
↓
2. SYN + ACK (요청 수락 + 역방향 요청)
서버 → 클라이언트: "들려! 나도 연결할게, 너도 들려?"
↓
3. ACK (역방향 수락)
클라이언트 → 서버: "들려! 이제 양측 연결 확인됨"
(최근 구현에서는 이 ACK와 함께 데이터를 같이 전송하기도 함)
↓
4. 데이터 전송
양측 연결이 확인됐으므로 실제 데이터 전송 시작

참고: 3-way handshake는 클라이언트와 서버 양측의 연결만 확인한다. 중간 노드(라우터 등)가 잘 연결돼 있는지는 알 수 없고 알 필요도 없다 — 양측이 "보내고 받는다"는 결론만 중요하다.

React에서 useEffect 안에서 API를 호출하기 전 "컴포넌트가 마운트됐는지 확인"하는 패턴처럼, TCP는 데이터를 보내기 전에 "상대방이 받을 준비가 됐는지 먼저 확인"한다. Promise가 resolve되기 전까지 데이터가 없는 것처럼, TCP handshake가 완료되기 전까지는 데이터 전송이 시작되지 않는다.

TCP가 IP의 한계를 해결하는 방법

IP 한계 TCP 해결 방법
비연결성 — 상대 상태 미확인 3-way handshake로 연결 먼저 확인
비신뢰성 — 유실·순서 미보장 ACK 응답 + 순서 번호로 전달 보증·순서 보장
포트 구분 불가 TCP 헤더에 출발 포트 · 목적 포트를 포함

UDP (사용자 데이터그램 프로토콜)

UDP의 특징

  • 연결 지향 아님 — handshake 없이 바로 전송
  • 전달 보증 없음 — 패킷 유실돼도 재전송 요청 없음
  • 순서 보장 없음 — 도착 순서가 뒤섞여도 그대로
  • 기능이 적어 빠름 — IP에 PORT·체크섬(Checksum) 정도만 추가
  • 부족한 기능은 애플리케이션 레이어에서 직접 구현

체크섬(Checksum)은 전송 자료의 무결성 검사용 값이다 — 데이터가 깨졌는지 확인하지만, 깨졌다고 재전송하지는 않는다.

언제 UDP를 쓰나

신뢰성보다 속도가 더 중요한 상황에 쓴다. 실시간 화상통화나 게임에서 한 프레임이 유실되는 것보다 화면이 굳는 게 더 나쁘다. 최근에는 HTTP/3(QUIC 프로토콜)이 UDP 기반으로 TCP보다 빠른 재전송을 직접 구현하는 방식을 채택했다.

사용 사례 이유
실시간 화상통화 (Zoom, WebRTC) 약간의 화질 저하보다 지연이 더 나쁨
온라인 게임 (위치·입력 정보) 오래된 위치 정보는 재전송해도 의미 없음
DNS 질의 짧은 요청·응답, 빠른 처리 우선

TCP vs UDP 비교

TCP UDP
연결 확인 O (3-way handshake) X
전달 보증 O (ACK + 재전송) X
순서 보장 O X
속도 상대적으로 느림 빠름
신뢰성 높음 앱이 직접 구현

정리

  • TCP = 연결 지향·전달 보증·순서 보장. IP의 한계를 보완하는 신뢰성 프로토콜
  • 3-way handshake: SYN → SYN+ACK → ACK → 가상 연결 확립 후 전송
  • UDP = 기능 최소화·빠른 속도. 실시간성이 중요하거나 앱이 신뢰성을 직접 구현할 때
  • 현재 대부분의 HTTP 웹 통신은 TCP 기반
  • HTTP/3은 UDP 기반 QUIC으로 TCP의 지연을 줄이는 방향으로 발전 중
