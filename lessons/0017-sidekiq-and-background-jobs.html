왜 백그라운드 잡이 필요한가

HTTP 요청-응답 사이클은 짧아야 한다. 사용자는 보통 2초 이상 기다리지 않는다. 그런데 어떤 작업은 시간이 오래 걸린다:

  • 이메일 발송 (외부 메일 서버 API 호출)
  • PDF 생성 (리뷰 결과 리포트 만들기)
  • 대량 알림 발송 (100명에게 Slack 메시지)
  • 외부 API 연동 (Slack, Google Sheets 등)

이런 작업을 HTTP 요청 안에서 직접 처리하면 응답이 10초, 30초씩 걸린다. 해결책은 "나중에 처리해"라고 큐에 던져두고 즉시 응답하는 것이다.

식당에서 주문을 받는 것과 같다:

웨이터(컨트롤러)는 주문을 받고 즉시 "알겠습니다"라고 답한다.
실제 요리(무거운 작업)는 주방(Worker)이 따로 처리한다.
웨이터가 요리가 완성될 때까지 손님 앞에 서 있지 않는다.

구조 — Sidekiq + Redis

① 컨트롤러: EmailWorker.perform_async(user_id) 호출
↓ 잡 정보를 직렬화해서
② Redis 큐에 잡을 저장 (메모리 DB, 초고속)
↓ 컨트롤러는 즉시 200 응답 반환
③ Sidekiq 프로세스가 Redis를 지속적으로 감시
↓ 잡 발견 시
④ EmailWorker#perform(user_id) 실행
↓
⑤ 이메일 발송 완료 — 사용자는 이미 응답을 받은 상태

Rails 앱과 Sidekiq은 별도의 프로세스다. Rails는 HTTP 요청을 받고, Sidekiq은 잡을 실행한다. 둘 다 Redis를 공유 큐로 사용한다.

Q1. EmailWorker.perform_async(user_id)를 호출하면 이메일이 즉시 발송되나?

Worker 작성법

Worker는 Ruby 클래스다. 두 가지만 지키면 된다:

# app/workers/review_notification_worker.rb
class ReviewNotificationWorker
  include Sidekiq::Worker                      # Sidekiq Worker임을 선언

  sidekiq_options queue: :default, retry: 3    # 옵션 (생략 가능)

  def perform(review_id, recipient_id)         # 실제 작업 메서드
    review = Review.find(review_id)
    recipient = User.find(recipient_id)
    NotificationService.send_review_ready(review, recipient)
  end
end

두 가지 규칙:

  • include Sidekiq::Worker — 필수. 이 클래스가 Worker임을 알린다.
  • def perform(...) — 실제 작업을 여기에 쓴다. Sidekiq이 이 메서드를 호출한다.

호출 방법 세 가지

# 즉시 큐에 등록 (가능한 빨리 실행)
ReviewNotificationWorker.perform_async(review.id, recipient.id)

# n초 후에 실행
ReviewNotificationWorker.perform_in(5.minutes, review.id, recipient.id)

# 특정 시각에 실행
ReviewNotificationWorker.perform_at(Time.zone.parse('2026-07-01 09:00'), review.id)

인자로 객체를 넘기지 않고 ID를 넘기는 것이 관용이다. Redis에 저장될 때 직렬화(JSON)되는데, ActiveRecord 객체는 직렬화가 안 되기 때문이다. Worker 안에서 ID로 다시 DB 조회한다.

큐와 재시도

큐 종류

큐는 우선순위 채널이다. Sidekiq은 높은 우선순위 큐를 먼저 처리한다. 

# config/sidekiq.yml
:queues:
  - [critical, 3]    # 가중치 3 — 중요한 작업 (결제, 인증)
  - [default, 2]     # 가중치 2 — 일반 작업
  - [low, 1]         # 가중치 1 — 덜 급한 작업 (리포트, 집계)

실패와 재시도

class ReviewNotificationWorker
  include Sidekiq::Worker
  sidekiq_options retry: 5    # 실패 시 최대 5번 재시도 (기본: 25번)

  def perform(review_id, recipient_id)
    # ...
  end
end

잡이 예외를 발생시키면 Sidekiq은 자동으로 재시도한다. 재시도 간격은 지수적으로 늘어난다 (15초 → 1분 → 10분 → ...). 모든 재시도가 실패하면 Dead Jobs 큐로 이동한다.
