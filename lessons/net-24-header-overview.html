HTTP 헤더 개요

🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 일반 — HTTP 메시지에서 헤더가 무엇이고 어떤 역할을 하는지 전체 구조를 파악한다

핵심 한 줄

HTTP 헤더(Header)는 전송에 필요한 모든 부가 정보를 담는 곳이다. 메시지 본문(body)에는 실제 데이터를, 헤더에는 그 데이터를 어떻게 처리할지에 대한 설명(메타데이터)을 담는다.

헤더 필드 형식

HTTP 헤더 한 줄의 생김새는 단순하다.

field-name ":" OWS field-value OWS

OWS는 Optional White Space — 공백이 있어도 되고 없어도 된다는 뜻이다. field-name은 대소문자를 구분하지 않는다.

실제 HTTP 요청/응답에서 볼 수 있는 헤더 예시다.

Content-Type: text/html;charset=UTF-8
Content-Length: 3423
Authorization: Bearer eyJhbGc...
Accept-Language: ko-KR,ko;q=0.9

헤더의 분류

HTTP 명세에 정의된 수십 가지 표준 헤더가 있고, 각각의 역할로 분류하면 크게 네 가지다.

  • General 헤더: 요청·응답 양쪽에서 쓸 수 있는 헤더 (예: Date)
  • Request 헤더: 요청에만 쓰이는 헤더 (예: User-Agent, Accept)
  • Response 헤더: 응답에만 쓰이는 헤더 (예: Server, WWW-Authenticate)
  • Representation 헤더: 본문 데이터의 표현 방식을 설명 (예: Content-Type, Content-Length)

임의 헤더도 추가할 수 있다

표준 헤더 외에도 커스텀 헤더를 마음대로 추가할 수 있다. 실무에서는 서비스마다 고유한 추적 헤더나 버전 헤더를 쓰는 경우가 많다. 

X-Request-Id: a1b2c3d4
X-Service-Version: 2.0
HELLO_WORLD: catsbi

관례적으로 커스텀 헤더 앞에 X-를 붙이던 시절이 있었지만, RFC 6648(2012)부터 해당 관례는 공식적으로 폐기됐다. 현재는 의미 있는 이름을 쓰면 된다.

React fetch()에서 두 번째 인자로 넘기는 headers 객체가 바로 이것이다. { headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer ...' } } 본문 데이터가 무엇인지, 누가 보냈는지를 서버에 알려주는 역할을 한다. 헤더가 없으면 서버는 본문을 받아도 어떻게 처리해야 할지 알 수 없다.

RFC 역사: 엔티티 헤더 → 표현 헤더

예전 명세(RFC 2616, 1999)에서는 엔티티 헤더(Entity Header)와 엔티티 본문(Entity Body)이라는 용어를 썼다. 2014년 RFC 7230~7235로 개정되면서 이 용어가 표현 헤더(Representation Header)와 표현 데이터(Representation Data)로 바뀌었다.

오래된 문서나 블로그에서 "엔티티 헤더"라는 말을 보게 되면 같은 개념이라고 보면 된다. 다음 강의부터 표현 헤더를 하나씩 살펴본다.

정리

  • 헤더 = 메타데이터. 본문을 어떻게 처리할지 서버·클라에게 알려준다
  • 형식: field-name: field-value (공백 허용, 대소문자 무관)
  • 표준 헤더 외에 커스텀 헤더를 자유롭게 추가할 수 있다
  • RFC 2616의 "엔티티" → RFC 7230+의 "표현(Representation)"으로 용어 변경
