SQL JOIN: 테이블을 합치는 법

레슨 004 SQL · PK · FK · JOIN · INNER JOIN · LEFT OUTER JOIN ~25분
🎯 미션 연결: Sentry 에러 Unknown column 'appraisal_result_formal_objection.status' in subquery를 직접 디버깅하면서 배운 개념 — 워크로그

먼저 알아야 할 것 — PK와 FK

기본키 (Primary Key, PK)

테이블에서 각 행을 유일하게 구별하는 값이다. 이름이 같은 사람이 두 명일 수 있지만, id는 절대 겹치지 않는다. Rails에서는 id 컬럼이 자동으로 PK가 된다. 

appraisal_appraisees
┌──────┬──────┐
│  id  │ name │
├──────┼──────┤
│   1  │ 김철수│  ← id=1은 이 행만을 가리킴
│   2  │ 이영희│  ← id=2는 이 행만을 가리킴
│   3  │ 박민준│
└──────┴──────┘
   ↑
   PK

외래키 (Foreign Key, FK)

다른 테이블의 PK를 참조하는 컬럼이다. "이 행이 저 테이블의 어떤 행에 속하는지"를 기록하는 연결고리다. 

appraisal_result_formal_objections
┌────┬──────────────────────┬────────┐
│ id │ appraisal_appraisee_id│ status │
├────┼──────────────────────┼────────┤
│  1 │          1           │ review │  ← "appraisee id=1의 이의신청"
│  2 │          2           │ submit │  ← "appraisee id=2의 이의신청"
└────┴──────────────────────┴────────┘
              ↑
              FK — appraisal_appraisees.id(PK)를 가리킴
React에서 자식 컴포넌트가 parentId prop을 받아 부모를 참조하듯, FK는 "나는 저 테이블의 id=1 행에 속해요"라고 기록하는 컬럼이다.

왜 JOIN이 필요한가

데이터베이스는 정보를 여러 테이블에 나눠서 저장한다. 평가 대상자 정보는 appraisal_appraisees에, 이의신청 정보는 appraisal_result_formal_objections에 따로 있다. 두 정보를 같이 보려면 테이블을 합쳐야 한다 — 이것이 JOIN이다.

컴포넌트 A의 state와 컴포넌트 B의 state를 한 화면에 보여주려면 상위에서 합쳐야 하듯, DB도 테이블 두 개를 JOIN으로 합쳐야 한 행(row)에서 같이 볼 수 있다.

JOIN은 FK ↔ PK가 일치하는 행끼리 붙이는 것이다:

appraisal_appraisees                  appraisal_result_formal_objections
┌────┬──────┐                         ┌────┬──────────────────────┬────────┐
│ id │ name │                         │ id │ appraisal_appraisee_id│ status │
├────┼──────┤                         ├────┼──────────────────────┼────────┤
│  1 │ 김철수│                         │  1 │          1           │ review │
│  2 │ 이영희│                         │  2 │          2           │ submit │
│  3 │ 박민준│  ← 이의신청 없음          └────┴──────────────────────┴────────┘
└────┴──────┘
         ↑
appraisal_appraisee_id 컬럼이 연결고리

INNER JOIN — 양쪽에 모두 있는 것만

매칭되지 않는 행은 결과에서 완전히 제외된다.

SELECT appraisees.name, objections.status
FROM appraisal_appraisees AS appraisees
INNER JOIN appraisal_result_formal_objections AS objections
  ON objections.appraisal_appraisee_id = appraisees.id

결과:
┌──────┬────────┐
│ name │ status │
├──────┼────────┤
│ 김철수│ review │  ← 매칭 ✓
│ 이영희│ submit │  ← 매칭 ✓
└──────┴────────┘
박민준은 결과에 없음 — 이의신청 행 자체가 없으니 매칭 불가

ActiveRecord에서는:

AppraisalAppraisee.joins(:appraisal_result_formal_objection)

오늘 버그 수정에서 objection_status 필터에 INNER JOIN을 쓴 이유: 이의신청이 있고 특정 상태인 사람만 보겠다는 의도이므로, 이의신청 없는 사람은 결과에 나오면 안 된다. 

# app/queries/appraisees_query.rb
def objection_status(option)
  return relation if option.nil?

  relation
    .joins(:appraisal_result_formal_objection)           # INNER JOIN
    .where(appraisal_result_formal_objection: { status: option })
end

LEFT OUTER JOIN — 왼쪽은 전부, 오른쪽은 있으면 붙이고 없으면 NULL

왼쪽 테이블의 모든 행이 결과에 포함된다. 오른쪽에 매칭되는 행이 없으면 NULL로 채운다.

SELECT appraisees.name, objections.status
FROM appraisal_appraisees AS appraisees
LEFT OUTER JOIN appraisal_result_formal_objections AS objections
  ON objections.appraisal_appraisee_id = appraisees.id

결과:
┌──────┬────────┐
│ name │ status │
├──────┼────────┤
│ 김철수│ review │  ← 매칭 ✓
│ 이영희│ submit │  ← 매칭 ✓
│ 박민준│  NULL  │  ← 이의신청 없지만 행은 살아있음
└──────┴────────┘

ActiveRecord에서는:

AppraisalAppraisee.left_outer_joins(:appraisal_result_formal_objection)

is_objection=false (이의신청 없는 사람 필터)에 LEFT OUTER JOIN이 필요한 이유: 찾으려는 대상이 바로 오른쪽 테이블에 없는 행이기 때문이다. INNER JOIN을 쓰면 그 대상이 JOIN 단계에서 이미 사라진다. 

# app/queries/appraisees_query.rb
def is_objection(option)
  return relation if option.nil?

  if option
    # 이의신청 있는 사람만 → INNER JOIN으로 충분
    relation.joins(:appraisal_result_formal_objection)
  else
    # 이의신청 없는 사람만 → LEFT OUTER JOIN + NULL 조건
    relation
      .left_outer_joins(:appraisal_result_formal_objection)
      .where(appraisal_result_formal_objection: { id: nil })
  end
end

JOIN 없이 WHERE를 쓰면 어떻게 되나

JOIN 없이 다른 테이블의 컬럼을 WHERE 조건에 쓰면 MySQL이 그 테이블을 알 수 없어서 에러가 난다. 단순 쿼리에서는 운 좋게 동작하는 경우도 있지만, 서브쿼리 안에서는 반드시 에러가 발생한다. 

-- 서브쿼리 안에서 JOIN 없이 컬럼 참조 → 에러
SELECT * FROM appraisal_appraisees
WHERE id IN (
  SELECT id FROM appraisal_appraisees
  -- appraisal_result_formal_objections가 JOIN되지 않았음
  WHERE appraisal_result_formal_objections.status = 'review'
  --    ↑↑↑↑ MySQL: "이 테이블이 어딨죠?"
)

-- Mysql2::Error: Unknown column 'appraisal_result_formal_objection.status'
--   in 'IN/ALL/ANY subquery'

오늘 Sentry에서 발생한 에러가 정확히 이 패턴이었다.

정리

JOIN 종류 결과 쓰는 경우
INNER JOIN 양쪽 모두 있는 행만 연관 데이터가 반드시 있어야 할 때
LEFT OUTER JOIN 왼쪽 전부 + 오른쪽은 NULL 가능 연관 데이터가 없는 행도 포함할 때
