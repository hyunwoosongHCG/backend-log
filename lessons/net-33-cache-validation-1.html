🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 캐시 — 캐시가 만료됐을 때 자원이 그대로라면 다시 안 받는 방법을 배운다

핵심 한 줄

캐시 유효 시간이 지났더라도 서버의 자원이 변하지 않았다면, Last-Modified / If-Modified-Since 헤더로 서버에 확인하고 304 Not Modified를 받아 캐시를 그대로 재활용한다.

문제: 만료됐는데 자원이 그대로라면?

강의 35에서 본 것처럼 캐시가 만료되면 서버에 재요청한다. 그런데 서버의 이미지 파일이 60초 사이에 전혀 바뀌지 않았다면, 완전히 동일한 파일을 또 전송하는 것은 낭비다.

이를 해결하는 것이 검증 헤더(Validation Header)와 조건부 요청(Conditional Request)이다. 서버에게 "내가 가진 것이 최신이 맞나요? 다르면 새로 주세요."라고 묻는 방식이다.

Last-Modified / If-Modified-Since

날짜 기반 검증 방식이다. 서버는 자원의 최종 수정 시각을 Last-Modified 헤더로 알려주고, 브라우저는 재요청 시 그 날짜를 If-Modified-Since에 담아 보낸다.

전체 흐름

[최초 요청] 브라우저 → 서버: GET /logo.png
↓
서버 → 브라우저: 200 OK
Cache-Control: max-age=60
Last-Modified: Tue, 01 Jul 2026 10:00:00 GMT
+ 바디(이미지 전체)
↓
브라우저: 이미지 + Last-Modified 날짜를 캐시에 저장
↓
[60초 이내] 재요청 → 서버에 묻지 않고 캐시에서 바로 사용
↓
[60초 경과 — 캐시 만료] 브라우저 → 서버:
GET /logo.png
If-Modified-Since: Tue, 01 Jul 2026 10:00:00 GMT
↓
서버: "내 파일의 최종 수정일이 그 날짜 이후인가?"를 비교
↓ (변경 없음)
서버 → 브라우저: 304 Not Modified (바디 없음)
Cache-Control: max-age=60
↓
브라우저: 304 수신 → 캐시의 자원을 그대로 사용, 유효기간만 갱신

변경됐을 때

서버: "파일이 If-Modified-Since 이후에 수정됐다" → 200 OK + 새 바디
Last-Modified: Tue, 01 Jul 2026 11:30:00 GMT (갱신)
↓
브라우저: 새 파일로 캐시 갱신

304 Not Modified의 의미

304는 "요청한 자원은 변경되지 않았으니, 네가 가진 캐시를 써도 된다"는 신호다. 바디(Body)가 없다 — 헤더만 오기 때문에 전송량이 극히 적다. 자원 크기에 상관없이 헤더 몇 백 바이트만 주고받으면 된다.

Last-Modified / If-Modified-Since의 한계

  • 1초 단위 정밀도: 1초 이내에 여러 번 수정된 경우 변경을 감지하지 못할 수 있다.
  • 수정 후 원복: 파일을 수정했다가 원래대로 되돌려도 날짜는 바뀌어 불필요하게 재전송한다.
  • 날짜 비교의 한계: 서버 로직이 수정 날짜를 정확히 관리해야 한다.

이 한계를 보완하는 방식이 다음 강의(검증 헤더와 조건부 요청2)의 ETag / If-None-Match다.

프론트 개발자라면 Chrome DevTools → Network 탭에서 304 Not Modified를 본 적이 있을 것이다. 그게 바로 이 흐름의 결과다. 크기가 0B 또는 "from disk cache"로 표시되며, 서버 왕복은 했지만 바디는 받지 않았다는 뜻이다. If-Modified-Since를 요청 헤더에서 직접 확인할 수도 있다.

정리

  • 캐시 만료 후 재요청 시 If-Modified-Since에 기존 날짜를 담아 보낸다
  • 서버가 비교해서 변경 없으면 304 Not Modified(바디 없음)를 반환한다
  • 브라우저는 304를 받으면 캐시 자원 그대로 사용, 유효기간만 갱신한다
  • 변경됐으면 서버가 200 OK + 새 바디 + 새 Last-Modified를 반환한다
  • 날짜 기반이라 1초 이하 변경과 수정-원복 케이스는 감지하지 못하는 한계가 있다
