DLQ(Dead Letter Queue)

레슨 029 DLQ · Dead Letter Queue · mark_as_consumed · 재시도 ~15분
🎯 "메시지 하나가 계속 에러를 내면 뒤에 쌓인 메시지들은 어떻게 되나?" — 실패가 파이프라인 전체를 막지 않게 하는 법을 이해한다 — MISSION.md

왜 DLQ가 필요한가

카프카 토픽은 메시지를 순서대로 처리한다. 만약 특정 메시지 처리 중 예외가 나서 Consumer가 계속 재시도만 한다면, 그 메시지 뒤에 쌓인 모든 메시지는 처리되지 못하고 무한정 대기하게 된다. 잘못된 데이터 하나가 파이프라인 전체를 멈춰 세우는 것이다.

DLQ(Dead Letter Queue)는 이 문제를 푸는 패턴이다: 몇 번 재시도해도 계속 실패하는 메시지는 원래 흐름에서 빼내 별도 토픽으로 옮기고, 나머지 메시지는 계속 처리를 이어간다.

프론트에서 실패한 API 요청을 다루는 방식과 비슷하다. 보통 fetch가 실패하면 몇 번 재시도(retry with backoff)하다가, 그래도 안 되면 에러 토스트를 띄우고 다음 사용자 액션으로 넘어간다 — 그 요청 하나 때문에 앱 전체가 멈추지 않는다.

DLQ는 여기서 한 걸음 더 나아간 것이다: 에러 토스트만 띄우고 끝내는 게 아니라, "실패한 요청"을 별도 채널(Sentry, 로그, DLQ 토픽)에 기록해서 나중에 사람이 들여다볼 수 있게 남겨둔다.

theplus-back의 DLQ 구조

theplus-back의 거의 모든 토픽은 -fail이 붙은 짝 토픽을 가진다: 

# theplus-back/karafka.rb
topic :'record-hr-organization-code' do
  consumer HrmOrganizationInfoConsumer
  dead_letter_queue(
    topic: 'record-hr-organization-code-fail',
    max_retries: 2,
    dispatch_method: :produce_sync
  )
end
topic :'record-hr-organization-code-fail' do
  consumer HrmOrganizationInfoFailConsumer
end

흐름은 이렇다:

① HrmOrganizationInfoConsumer#consume에서 예외 발생
↓ mark_as_consumed를 호출하지 않고 그대로 raise
② Karafka가 같은 메시지를 max_retries(2)번까지 재시도
↓ 그래도 실패하면
③ 메시지를 record-hr-organization-code-fail 토픽으로 옮김 (DLQ)
↓
④ HrmOrganizationInfoFailConsumer가 몇 번 더 재시도 후, 계속 실패하면 Sentry.capture_message + mark_as_consumed로 종료

핵심은 ④번이다 — Fail Consumer는 "포기할 때 조용히 사라지지 않고, Sentry에 남긴 뒤 명시적으로 mark_as_consumed를 호출한다". 이걸 안 하면 이 메시지도 계속 큐에 남아 똑같은 문제를 반복한다.

Q1. Consumer가 예외를 계속 던지기만 하고 DLQ 설정이 없다면 어떻게 되는가?

이번 조사에서 발견한 "구멍 난 DLQ"

theplus-back의 system-account-updated 토픽은 DLQ 대상(system-account-updated-fail)이 선언은 되어 있는데, 정작 그 -fail 토픽을 처리할 Consumer/라우팅이 karafka.rb에 없다. 즉 DLQ로 옮겨지긴 하는데, 거기서 처리해줄 사람이 없는 상태 — "죽은 편지함이 하나 더 있는" 셈이다. 실전 코드도 완벽하지 않을 수 있다는 걸 보여주는 실제 사례라 기록해둔다.

Q2. DLQ 토픽으로 옮겨진 메시지를 아무도 구독하지 않으면?

심화 — 모든 실패가 똑같이 재시도할 가치가 있는 건 아니다

PR #1294는 이 레슨이 다룬 "일단 max_retries번 재시도하고 안 되면 DLQ"보다 한 단계 더 정교한 구조를 도입했다. KafkaErrorClassifier로 에러를 두 종류로 나눈 것이다:

분류 예시 처리
Transient (일시적) DB 커넥션 타임아웃, 락 경합 re-raise → Karafka가 재시도 → 그래도 안 되면 DLQ
Non-transient (구조적) InvalidPayloadError, ResourceNotFoundError 같은 데이터 자체의 문제 재시도 없이 바로 DLQ 직행

잘못된 형식의 페이로드는 100번 재시도해도 여전히 잘못된 형식이다 — 재시도는 "곧 나아질 수도 있는 문제"에만 의미가 있다. 이 구분이 없으면, 구조적으로 절대 성공할 수 없는 메시지 때문에 max_retries번만큼 브로커에 불필요한 부하를 주고 DLQ 도달까지의 시간만 늘어난다.

또 하나 바뀐 건 실패 기록 방식이다. 원래는 Consume 서비스 안에서 에러가 날 때마다 Sentry.capture_message를 직접 호출했는데, 이러면 재시도할 때마다 같은 메시지로 알림이 중복 발송된다. PR #1294는 이걸 DLQ에 도달한 시점 딱 한 번으로 모으고, 실패 이력을 Sentry(휘발성 알림)뿐 아니라 KafkaFailedMessage라는 DB 테이블에도 남긴다 — 나중에 개발자가 검색하고, 수기로 처리 완료 표시(resolve!)까지 할 수 있는 영구 기록으로 바꾼 것이다.

Q3. 페이로드 형식 자체가 잘못된 메시지(예: 필수 필드 누락)를 만났을 때 가장 합리적인 처리는?

핵심 요약

개념 역할
DLQ 계속 실패하는 메시지를 별도 토픽으로 옮겨, 나머지 처리를 막지 않게 하는 패턴
max_retries 원래 토픽에서 몇 번 재시도한 뒤 DLQ로 넘길지 정하는 값
Fail Consumer DLQ 토픽을 받아 최종적으로 로그를 남기고 mark_as_consumed로 마무리하는 쪽
Transient vs Non-transient 일시적 에러만 재시도, 구조적 에러는 바로 DLQ — 에러를 분류해야 재시도가 의미 있다
