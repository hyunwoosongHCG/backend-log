after_save, after_create, after_commit의 차이 — 트랜잭션 커밋 시점

🎯 코드를 보고 "이 요청이 어떤 경로로 흘러가는지" 머릿속에 그릴 수 있다 — MISSION.md

레슨 19에서 배운 것 — 콜백은 저장 시점에 걸린다

레슨 19에서 before_save, after_create 같은 콜백이 "레코드가 저장될 때 자동으로 실행되는 코드"라는 걸 배웠다. 그런데 History 모델을 보니 처음 보는 콜백이 있었다. 

# app/models/history.rb
after_commit :notify, on: :create
#     ^^^^^^ after_create가 아니라 after_commit

이름이 비슷해서 after_create와 같은 건 줄 알았는데, 실행 시점이 다르다는 걸 알게 됐다.

트랜잭션이란 무엇이었나 — 복습

레슨 24(트랜잭션과 원자성)에서 "여러 DB 작업을 하나의 단위로 묶어서, 전부 성공하거나 전부 실패하게 만드는 것"이 트랜잭션이라고 배웠다. 

ActiveRecord::Base.transaction do
  history = History.create!(...)   # ① INSERT 실행 (아직 커밋 전)
  some_other_record.update!(...)   # ② 이 줄에서 에러가 나면?
end
# ③ 여기까지 왔을 때 비로소 COMMIT (또는 에러 시 ROLLBACK)

①에서 INSERT가 실행돼도, ②에서 에러가 나면 트랜잭션 전체가 롤백되면서 ①도 취소된다 — DB에는 아무것도 남지 않는다.

after_save는 또 뭔가 — "트리거 조건"과 "실행 시점"은 다른 축이다

after_create 옆에는 after_save라는 비슷하게 생긴 콜백도 있다. 이름만 보면 헷갈리기 쉬운데, 이 둘의 차이는 after_create와 after_commit의 차이와는 완전히 다른 축이다.

  • after_create — 레코드가 새로 생성(INSERT)될 때만 실행
  • after_save — 레코드가 생성(INSERT)되든 수정(UPDATE)되든 저장될 때마다 실행

즉 after_create vs after_save는 "어떤 동작에 반응하는가" (트리거 조건, 범위)의 차이다. 반면 after_create/after_save vs after_commit은 "언제 실행되는가"(트랜잭션 안이냐 밖이냐, 타이밍)의 차이다. after_save도 after_create와 마찬가지로 SQL 실행 직후, 트랜잭션이 커밋되기 전에 실행된다 — 타이밍은 둘이 똑같다.

콜백 비교 — 트리거 조건 × 실행 시점

콜백 트리거 조건 실행 시점 트랜잭션이 나중에 롤백되면?
after_create 생성(INSERT)될 때만 트랜잭션 안 (커밋 전) 이미 실행된 콜백은 되돌릴 수 없다 — 부수효과(알림 발송 등)가 이미 일어난 뒤 DB만 롤백되는 불일치 발생
after_save 생성 또는 수정될 때 (둘 다) 트랜잭션 안 (커밋 전) — after_create와 동일 위와 동일하게 위험 — 트리거 범위만 넓을 뿐, 타이밍은 after_create와 같다
after_commit 기본은 생성+수정+삭제 전체, on: :create/:update/:destroy로 범위 지정 가능 트랜잭션이 실제로 COMMIT된 후 롤백되면 콜백 자체가 실행되지 않는다 — 안전

History#notify는 알림 push, Slack 메시지 발송 같은 "외부로 나가는 부수효과"를 실행한다. 만약 after_create나 after_save였다면, History.create!가 더 큰 트랜잭션 안에서 호출됐을 때 그 트랜잭션이 나중에 실패해서 롤백돼도 — 알림은 이미 나가버린 뒤라 취소할 수 없다. 존재하지 않게 된 History에 대한 알림이 사용자에게 도착하는 것이다. after_create를 after_save로 바꾼다고 이 문제가 해결되지 않는다 — 둘 다 커밋 전에 실행되는 건 똑같기 때문이다. after_commit은 "DB에 진짜로 반영된 게 확정된 후에만" 실행되므로 이 문제가 없다.

React의 낙관적 업데이트(optimistic update)와 반대되는 상황이라고 생각하면 된다. 
// 낙관적 업데이트: 서버 응답 전에 먼저 UI/부수효과를 실행 (after_create와 비슷한 위험)
setLiked(true);
trackAnalyticsEvent('liked');   // 서버 요청이 실패하면? 이미 이벤트는 발행됨

// 커밋 후 실행: 서버가 성공을 확정한 뒤에만 부수효과 실행 (after_commit)
await likeMutation();
trackAnalyticsEvent('liked');   // mutation이 실패하면 이 줄 자체가 실행되지 않음
await mutation()이 성공적으로 끝난 뒤에만 다음 줄을 실행하는 것처럼, after_commit은 "DB 트랜잭션이 끝까지 성공했다"는 게 확정된 뒤에만 실행된다.

규칙: 외부로 나가는 부수효과는 after_commit

이메일 발송, 알림 push, Slack 메시지, 외부 API 호출, Sidekiq job enqueue처럼 DB 트랜잭션 밖의 세계에 영향을 주는 코드는 after_commit에 두는 것이 안전하다. 반대로 같은 트랜잭션 안에서 다른 레코드를 만들거나 값을 계산하는 로직(레슨 19에서 본 것들)은 굳이 커밋을 기다릴 필요가 없으니 after_create/ after_save/before_save로 충분하다. after_create와 after_save 중 어느 걸 쓸지는 "생성 때만이냐, 수정 때도냐"라는 트리거 범위만 보고 고르면 된다 — 롤백 안전성과는 무관한 선택이다.
