핵심 한 줄

표현 헤더(Representation Header)는 HTTP 본문(body)에 담긴 데이터가 어떤 형식·인코딩·언어·길이인지 수신측에 알려주는 헤더다. 서버와 클라이언트는 이 헤더 없이는 본문을 어떻게 해석해야 할지 알 수 없다.

왜 "표현(Representation)"인가

같은 데이터라도 보내는 형식은 다를 수 있다. 회원 정보 하나를 application/json으로도 보낼 수 있고, application/xml로도 보낼 수 있다. 동일한 리소스를 여러 가지 방식으로 "표현"할 수 있기 때문에 RFC 7230+에서는 이것을 "표현(Representation)"이라 부른다.

Content-Type: 표현 데이터의 형식

본문이 어떤 미디어 타입(Media Type)인지 알려준다. 타입/서브타입 형식이다.

Content-Type: text/html;charset=UTF-8
Content-Type: application/json
Content-Type: image/png
  • text/html;charset=UTF-8 — HTML 문서, UTF-8 인코딩
  • application/json — JSON 데이터 (기본 인코딩 UTF-8)
  • image/png — PNG 이미지

브라우저는 이 값을 보고 렌더링 방법을 결정한다. HTML 파일을 application/json으로 보내면 브라우저는 렌더링하지 않고 JSON 텍스트 그대로 출력해버린다.

Content-Encoding: 표현 데이터의 인코딩

보내는 쪽이 본문을 압축한 경우, 어떤 방식으로 압축했는지 알려준다. 받는 쪽은 이 헤더를 보고 압축을 해제한다. 

Content-Encoding: gzip
Content-Encoding: deflate
Content-Encoding: identity
  • gzip — 가장 많이 쓰이는 압축 방식
  • deflate — zlib 포맷 압축
  • identity — 압축하지 않음 (기본값)

Content-Language: 표현 데이터의 자연 언어

본문이 어떤 언어로 작성됐는지 알려준다.

Content-Language: ko
Content-Language: en
Content-Language: en-US

Content-Length: 표현 데이터의 길이

본문의 크기를 바이트 단위로 알려준다. 주의: Transfer-Encoding: chunked를 사용하면 본문을 조각(chunk)으로 나눠 보내기 때문에 전체 길이를 미리 알 수 없다. 이 경우에는 Content-Length를 함께 쓰면 안 된다. 

Content-Length: 3423
React에서 fetch()로 POST 요청을 보낼 때 이렇게 쓴다. { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(data) } Content-Type: application/json이 없으면 Rails 서버는 body를 JSON으로 파싱하지 않고 plain text로 처리해버린다. 서버가 어떻게 읽을지 "설명"해주는 것이 바로 표현 헤더다.

정리

  • Content-Type: 본문의 미디어 타입 (무엇인가)
  • Content-Encoding: 본문의 압축 방식 (어떻게 압축했나)
  • Content-Language: 본문의 자연 언어 (어떤 언어로 쓰여 있나)
  • Content-Length: 본문의 크기 (몇 바이트인가)
  • 이 네 헤더가 없으면 수신측은 본문을 어떻게 해석할지 알 수 없다
