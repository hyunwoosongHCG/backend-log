인터넷 통신

네트워크 · 인터넷 네트워크 강의 4 01:51
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · 인터넷 네트워크 — fetch() 한 줄 뒤에 숨어 있는 "인터넷"의 실체를 본다

핵심 한 줄

인터넷 통신은 두 컴퓨터가 직접 연결돼 주고받는 게 아니라, 중간의 수많은 노드(라우터·케이블)를 거쳐 메시지가 전달되는 것이다. 그래서 "누구에게, 어떤 규칙으로 보낼지"에 대한 약속(프로토콜)이 필요하고, 그 약속을 계층(레이어)으로 쌓아올린 것이 TCP/IP 프로토콜 스택이다.

출발점: fetch() 한 줄 뒤에서 무슨 일이?

프론트에서는 이 한 줄이면 지구 반대편 서버의 데이터가 손에 들어온다. 

const res = await fetch('https://api.example.com/users');
const users = await res.json();

그런데 이 fetch가 요청을 보내는 순간, 내 브라우저와 그 서버 사이에서 실제로 무슨 일이 벌어질까? 둘은 케이블 하나로 직접 이어져 있을까? 아니다 — 그 사이에는 어마어마하게 복잡한 길이 있다.

fetch()가 Promise 하나로 "네트워크 요청"의 온갖 복잡함을 감춰주듯, "인터넷"이라는 단어 자체가 그 아래의 거대한 노드망을 감춰주는 추상화다. 평소엔 몰라도 되지만, 요청이 느리거나 끊길 때 원인을 짚으려면 이 아래층을 알아야 한다.

두 컴퓨터는 직접 연결돼 있지 않다

클라이언트(내 컴퓨터)와 서버는 1:1 전용선으로 묶여 있지 않다. 메시지는 중간의 여러 노드를 거쳐서(hop by hop) 목적지까지 전달된다.

클라이언트 (내 브라우저)
↓
공유기 · 집/회사 네트워크
↓
ISP(통신사) 라우터
↓
인터넷 백본 — 수많은 라우터 · 해저 광케이블
↓
대상 서버 측 ISP · 데이터센터
↓
서버

중간 노드가 수천 개일 수도 있고, 물리적으로는 대륙을 잇는 해저 케이블을 지나기도 한다. 이 복잡한 경로 전체를 뭉뚱그려 우리는 그냥 "인터넷"이라고 부른다.

그래서 "규칙"이 필요하다 — 프로토콜 스택

노드가 이렇게 많은데 아무 약속도 없다면, 메시지를 다음 노드에게 "어디로, 어떤 형식으로" 넘겨야 할지 알 수 없다. 그래서 인터넷의 모든 노드가 공통으로 지키는 약속 = 프로토콜(protocol)이 필요하다.

그 약속을 역할별로 나눠 쌓은 것이 프로토콜 스택(protocol stack)이다. 학문적으로는 OSI 7계층, 실제 인터넷에서는 TCP/IP 4계층을 쓴다.

OSI 7계층 vs TCP/IP 4계층

OSI 7계층 TCP/IP 4계층 대표 프로토콜 담당
7 애플리케이션
6 표현
5 세션		 ① 애플리케이션 계층 HTTP, FTP, DNS 앱이 사용하는 데이터 표현
4 트랜스포트 ② 트랜스포트 계층 TCP, UDP 경로 지정 후 데이터 전송 방법
3 네트워크 ③ 인터넷 계층 IP 논리 주소로 경로 선택
2 데이터링크
1 물리		 ④ 네트워크 액세스 계층 이더넷, Wi-Fi 랜카드·드라이버 등 물리 영역
React 컴포넌트도 "UI 레이어 → 상태 레이어 → API 레이어"로 역할을 나눈다. TCP/IP 계층도 같은 원리다 — 상위 계층은 하위 계층이 뭘 하는지 모른 채 자기 일만 한다. HTTP는 "데이터를 어떻게 표현할까"만 신경 쓰고, IP가 "어디로 보낼까"를 어떻게 해결하는지 알 필요가 없다. 계층을 분리하면 각 계층을 독립적으로 개발하고 교체할 수 있다(객체지향처럼).

인터넷 통신 6단계 과정 요약

fetch('https://google.com/users')를 눌렀을 때 실제로 벌어지는 일을 계층 순서로 따라가보면:

1. DNS 조회
브라우저가 google.com 도메인으로 DNS 서버에 질의 → DNS가 매칭되는 IP 주소 응답 (예: 142.250.196.46)
↓
2. 데이터를 패킷(Packet)으로 분할
보낼 데이터를 적절한 크기의 블록(payload)들로 나눔
↓
3. TCP 정보 추가 (트랜스포트 계층)
출발 포트·목적 포트·순서 번호·검증 정보를 각 패킷에 붙임
↓
4. IP 정보 추가 (인터넷 계층)
출발 IP · 목적지 IP · 관리 정보를 붙여 목적지를 지정
↓
5. TCP 3-way handshake (TCP인 경우)
SYN → SYN+ACK → ACK 로 양측 연결을 먼저 확인
↓
6. 데이터 전송 및 응답
패킷들이 노드를 hop by hop으로 이동 → 서버 수신 → 응답 반환

이 6단계 각각을 뒷받침하는 규칙이 다음 강의들의 주제다 (DNS → 강의 8, 패킷 → 강의 5 IP, TCP → 강의 6).

정리

  • 인터넷 통신 = 중간 노드를 거쳐 전달되는 간접 통신 (직접 연결 아님)
  • "인터넷"은 복잡한 노드망 전체를 감추는 추상화된 이름
  • 수많은 노드가 협력하려면 공통 규칙(프로토콜)이 필수
  • 그 규칙을 역할별로 쌓은 것이 TCP/IP 4계층 스택
  • 한 번의 HTTP 요청 뒤에는 DNS 조회 → 패킷 분할 → TCP/IP 포장 → handshake → 전송의 6단계가 있다
  • 가장 아래 규칙부터 → 다음 강의 IP로 이어진다
