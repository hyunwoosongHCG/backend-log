N+1 문제

레슨 009 N+1 · preload · includes · eager_load · 성능 ~20분
🎯 미션 연결: ppback 컨트롤러가 preload(:appraisal_result_formal_objection)를 쓰는 이유. — 워크로그

N+1이 뭔가?

목록 1번 쿼리 + 각 항목마다 연관 데이터 조회 N번 = 총 N+1번 쿼리가 나가는 문제다. N이 100이면 쿼리 101번, N이 1000이면 1001번이 된다. 

# N+1 발생 패턴
appraisees = AppraisalAppraisee.where(...)  # 쿼리 1번 — appraisees 100개 로드

appraisees.each do |appraisee|
  puts appraisee.appraisal_result_formal_objection&.status
  # ↑ 매 반복마다 쿼리 1번씩 추가 발생
  # SELECT * FROM appraisal_result_formal_objections WHERE appraisal_appraisee_id = 1
  # SELECT * FROM appraisal_result_formal_objections WHERE appraisal_appraisee_id = 2
  # SELECT * FROM appraisal_result_formal_objections WHERE appraisal_appraisee_id = 3
  # ... 100번
end

# 총 101번 쿼리
JS에서 API를 반복문으로 100번 호출하는 것과 같다:

const users = await fetchUsers() // 1번
for (const user of users) {
  const detail = await fetchDetail(user.id) // N번
}

한 번에 묶어서 보내면 되는데 반복문 안에서 개별 요청을 날리는 비효율이다.

해결 — preload

preload는 쿼리 2번으로 처리한다: ① appraisees 전체 로드 → ② 해당 ID들의 이의신청 전체 로드. 이후 Ruby에서 매칭해 메모리에서 연결한다. 

appraisees = AppraisalAppraisee
  .where(...)
  .preload(:appraisal_result_formal_objection)

-- 실행되는 SQL (딱 2번)
SELECT * FROM appraisal_appraisees WHERE ...
SELECT * FROM appraisal_result_formal_objections
  WHERE appraisal_appraisee_id IN (1, 2, 3, ..., 100)
appraisees.each do |appraisee|
  puts appraisee.appraisal_result_formal_objection&.status
  # 추가 쿼리 없음 — 이미 메모리에 있음
end

preload / includes / eager_load 차이

세 가지 모두 N+1을 방지하지만 내부 동작이 다르다:

메서드 SQL 방식 WHERE에 연관 컬럼 사용
preload 별도 SELECT 쿼리 2번 불가
includes 상황에 따라 preload 또는 eager_load 자동 선택 가능 (eager_load로 전환)
eager_load LEFT OUTER JOIN 한 방에 가능

ppback이 preload를 명시적으로 쓰는 이유: WHERE 조건은 joins로 이미 처리했고, 렌더링용 데이터 로딩만 필요하기 때문이다. includes를 쓰면 Rails가 내부적으로 어떤 방식을 선택할지 예측하기 어려워진다.

joins와 preload를 같이 쓰는 패턴

# 이 패턴이 왜 맞는가
AppraisalAppraisee
  .joins(:appraisal_result_formal_objection)    # WHERE status = ? 를 위한 SQL JOIN
  .where(appraisal_result_formal_objection: { status: :review })
  .preload(:appraisal_result_formal_objection)  # 결과 렌더링을 위한 메모리 로딩

같은 연관관계를 joins와 preload 양쪽에 쓰는 것이 어색해 보일 수 있지만, 목적이 다르다:

  • joins = SQL 필터링 조건 추가
  • preload = Ruby에서 데이터 접근할 때 추가 쿼리 방지
