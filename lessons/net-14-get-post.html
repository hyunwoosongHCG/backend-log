GET, POST

네트워크 · HTTP 메서드 강의 17 13:05
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 메서드 — GET과 POST의 "진짜 차이"를 백엔드 관점으로 이해한다

핵심 한 줄

GET은 조회 전용이라 서버 상태를 바꾸지 않고, POST는 "서버가 처리해 줘"라는 요청이라 등록부터 프로세스 실행까지 모두 담당한다.

GET — 리소스를 조회한다

GET은 서버에서 리소스를 읽어(Read) 오는 것이다. 서버 상태를 변경하지 않는다.

데이터를 어떻게 전달하는가?

GET 요청에서 서버에 전달할 데이터는 쿼리 스트링(Query String)으로 보낸다. 요청 바디(Body)에 데이터를 담는 것이 HTTP 명세상 가능하긴 하지만, 지원하지 않는 서버·프록시가 많아 실무에서는 쓰지 않는다. 

# 검색어와 언어를 쿼리 스트링으로 전달
GET /search?q=hello&hl=ko HTTP/1.1
Host: www.example.com

쿼리 스트링은 URL에 노출되기 때문에 비밀번호 같은 민감한 정보는 GET으로 보내면 안 된다.

GET 요청 전체 모습

GET /members/1 HTTP/1.1
Host: api.example.com
Accept: application/json

바디 없이 헤더만으로 요청한다. 응답에서 서버가 200 OK와 함께 JSON을 돌려준다.

POST — 서버가 처리하게 한다

POST는 단순히 "데이터를 만들어 줘"가 아니다. 더 넓은 의미로 "메시지 바디의 데이터를 서버가 처리하도록 요청"하는 것이다.

POST의 두 가지 역할

1) 새 리소스 생성 (Registration)

서버가 아직 식별하지 않은 새 리소스를 만든다. 클라이언트는 URI를 지정하지 않고, 서버가 ID를 결정해 반환한다. 

# 요청: 회원 등록 — ID는 클라이언트가 지정하지 않는다
POST /members HTTP/1.1
Content-Type: application/json

{
  "name": "김철수",
  "email": "kim@example.com"
}

# 응답: 서버가 생성한 URI를 Location 헤더로 알려준다
HTTP/1.1 201 Created
Location: /members/100

2) 프로세스 처리 (Process Execution)

단순한 생성을 넘어 어떤 처리 흐름을 시작할 때도 POST를 쓴다. 새 리소스가 만들어지지 않을 수도 있다. 

# 주문의 배달을 시작시키는 POST — 새 리소스를 만드는 게 아니라 프로세스를 실행한다
POST /orders/55/start-delivery HTTP/1.1

POST는 만능 메서드

다른 메서드(GET, PUT, PATCH, DELETE)로 처리하기 애매한 경우는 모두 POST로 해결한다. 조회라도 JSON 바디가 꼭 필요하다면 POST를 쓰기도 한다.

GET vs POST — 핵심 차이 비교

구분 GET POST
목적 리소스 조회(Read) 처리 요청(Create / Execute)
데이터 위치 쿼리 스트링(URL) 메시지 바디(Body)
서버 상태 변경 변경 없음 변경 있음
멱등(Idempotent) 멱등 (같은 요청 = 같은 결과) 비멱등 (반복 시 중복 발생 가능)
캐시 가능 (브라우저·CDN 캐시) 원칙상 가능하나 실무상 안 함
응답 상태코드 200 OK 201 Created / 200 OK
React에서 fetch('GET')는 화면에 데이터를 뿌리는 용도, fetch('POST')는 폼을 제출하는 용도로만 써왔을 것이다. 백엔드 관점의 차이는 멱등성이다 — GET은 100번 호출해도 서버 상태가 달라지지 않지만, POST로 "회원 등록"을 3번 호출하면 회원이 3명 생긴다. 이 차이가 재시도 안전성(retry safety)에 직결된다 — 네트워크 에러 후 자동 재시도를 할 때 GET은 안전하고 POST는 위험하다.

정리

  • GET = 조회 전용. 데이터는 쿼리 스트링으로. 바디 사용은 지양
  • POST = 처리 요청. 데이터는 메시지 바디로. 등록·프로세스 실행 모두 담당
  • POST는 서버가 ID를 결정한다 (클라이언트가 URI를 지정하지 않음)
  • 다른 메서드로 표현 어려우면 POST로 해결한다
  • GET은 멱등·캐시 가능, POST는 비멱등·재시도 위험
