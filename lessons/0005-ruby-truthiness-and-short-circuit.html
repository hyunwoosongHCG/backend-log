레슨 5 — Ruby의 ? ! &.

2026-06-29 실무 출처: ppback PR #5479 디버깅 이전: 레슨 4 — SQL JOIN
오늘 실제로 겪었다. setting.user?를 setting.user?에서 setting&.user?로, 다시 원점으로 돌아온 과정이 이 세 가지 개념으로 설명된다.

1. ? — "이게 맞아?" 라고 묻는 메서드

Ruby에서 메서드 이름이 ?로 끝나면 boolean을 반환한다는 약속이다. true 또는 false (또는 truthy/falsy)를 돌려준다는 신호다.

TypeScript에서 isAdmin(), hasPermission()처럼 is/has prefix를 붙이는 관습과 같다. Ruby는 그냥 끝에 ?를 붙인다. 
# 표준 라이브러리
"".empty?          # => true
[1,2].include?(3)  # => false

# 이 레포에서 실제로 쓴 것들
comment.completed?           # 코멘트 완료 상태?
setting.user?                # comment_template_config == 'user'?
workspace.user?(user)        # 이 워크스페이스의 멤버?

# 오늘 우리가 만든 것
privacy_blocked?(setting)    # 비공개 차단 조건?
template_required?(setting)  # 템플릿 필수 조건?

?는 실제 동작을 바꾸지 않는다 — 순전히 읽는 사람을 위한 약속이다. 이 메서드는 boolean을 리턴하니 if에 바로 쓸 수 있다는 의미다.

2. ! — "조심해, 위험해" 메서드

두 가지 중 하나를 의미한다:

# (1) 실패 시 예외를 던진다
record.save      # 실패하면 false 반환
record.save!     # 실패하면 ActiveRecord::RecordInvalid 예외 발생

# (2) 원본 객체를 직접 변경한다
arr = [3, 1, 2]
arr.sort         # 새 배열 반환, arr는 그대로
arr.sort!        # arr 자체가 [1, 2, 3]으로 변함
JavaScript의 Array.prototype.sort()가 원본을 바꾸는 것처럼 !가 붙은 건 "사이드 이펙트 있음"이라는 경고다. 이 레포에서 comment.discard!, workspace.save!가 모두 이 패턴이다.

3. 단락 평가 (Short-circuit Evaluation)

&&는 왼쪽이 falsy면 오른쪽을 아예 실행하지 않는다. JavaScript와 완전히 동일하다. 

# JavaScript
null && someFunction()   // someFunction() 실행 안 됨

# Ruby — 동일한 원리
nil && setting.user?     # setting.user? 실행 안 됨
false && expensive_call  # expensive_call 실행 안 됨

오늘 실무에서 이게 핵심이었다:

setting = workspace.one_on_one_setting  # nil일 수도 있음

# 이 줄은 안전하다
if setting&.user? && setting.use_required_template && ...
#     ↑ nil이면 nil 반환 → &&가 여기서 멈춤 → setting.use_required_template 실행 안 됨

단락 평가 덕분에 setting이 nil이어도 두 번째 조건에서 NoMethodError가 발생하지 않는다. 앞이 거짓이면 뒤는 건드리지 않는다.

4. &. — nil-safe 메서드 호출

Ruby 2.3에서 추가된 Safe Navigation Operator다. object&.method는 "object가 nil이면 nil을 반환하고, 아니면 method를 호출"한다. 

setting = nil

setting.user?    # NoMethodError: undefined method 'user?' for nil
setting&.user?   # => nil (예외 없이 조용히 nil 반환)
JavaScript의 Optional Chaining과 완전히 같다:
setting?.user() → setting&.user?

5. 오늘 실제로 겪은 버그

오늘 setting&.user?의 &.을 제거했더니 기존 테스트 12개가 한꺼번에 터졌다: 

NoMethodError (undefined method 'user?' for nil):
  app/controllers/v2/comments.rb:145

원인: 기존 테스트들은 API를 통하지 않고 팩토리로 OneOnOne을 직접 생성해서 one_on_one_setting이 nil이었다. 프로덕션에서는 GET /one_on_one_settings 호출 시 find_or_create_by!로 항상 생성되지만, 테스트는 그 경로를 거치지 않았다.

해결: 테스트의 before 블록에 GET 요청을 추가해서 실제 플로우를 재현했다:

before do
  stub_authenticate(assigne_user)
  get "/api/v1/workspaces/#{workspace.hash_id}/one_on_one_settings"
  # 이 호출이 find_or_create_by!를 트리거 → setting이 nil이 아님
end

이후 &. 없이도 안전하게 동작한다.

6. Grape의 mount와 helpers 전파

오늘 check_one_on_one_setting 안에서 workspace를 바로 쓸 수 있는지 의문이 생겼다. V2::Comments를 보면 이렇게 돼 있다: 

class V2::Comments < Grape::API   # Grape::API를 직접 상속
  helpers do
    def check_one_on_one_setting
      setting = workspace.one_on_one_setting  # workspace는 어디서?
    end
  end
end

V2::Comments는 GrapeBase를 include하지 않는다. workspace를 직접 선언한 곳도 없다. 그런데 왜 쓸 수 있을까?

마운트 구조

# app/controllers/v2/base.rb
class V2::Base < Grape::API
  include ::GrapeBase        # workspace, current_user 등이 여기서 정의됨

  resource 'workspaces/:workspace_id' do
    mount V2::Comments       # V2::Comments를 여기에 붙임
  end
end

mount는 Ruby의 클래스 상속(<)이 아니다. Grape 자체의 메커니즘으로, 마운트된 자식 API는 부모의 helpers 컨텍스트 안에서 실행된다. V2::Base가 GrapeBase를 include해서 가진 workspace가 요청 처리 시 V2::Comments 엔드포인트까지 전달되는 것이다.

React Context와 비슷하다. 부모 컴포넌트에서 Provider로 값을 제공하면, 어디에 중첩된 자식 컴포넌트든 useContext로 그 값을 꺼내 쓸 수 있다. V2::Base가 Provider, workspace가 context 값, V2::Comments가 자식 컴포넌트다.

핵심 구분

Ruby 클래스 상속 < Grape mount
관계 클래스 계층 (is-a) 라우트 조합 (has-routes)
helpers 공유 자동 상속 요청 컨텍스트로 전파
코드에서 class A < B mount A

정리

기호 의미 JS 대응
? boolean 반환 약속 is*() / has*() prefix
! 예외 던지거나 원본 변경 (명시적 관습 없음)
&& 단락평가 왼쪽 falsy → 오른쪽 실행 안 함 && 동일
&. nil이면 nil 반환, 아니면 메서드 호출 ?. Optional Chaining
