프록시 캐시

🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 캐시 — 브라우저와 Origin 서버 사이에 중간 캐시 서버를 두는 이유와 동작을 이해한다

핵심 한 줄

프록시 캐시(Proxy Cache) 서버는 Origin(원본) 서버와 클라이언트 사이에 위치해서 여러 클라이언트가 공유할 수 있는 캐시를 제공해 응답 시간을 단축하는 중간 서버다.

왜 프록시 캐시가 필요한가

한국 사용자가 미국에 있는 서버에서 같은 이미지를 수천 명이 각자 요청한다고 하자. 브라우저 캐시는 각자의 PC에만 저장되기 때문에, 다른 사람이 이미 받은 파일이라도 내 PC에는 없으면 나는 다시 미국 서버까지 요청해야 한다.

프록시 캐시 서버를 한국에 두면 다음과 같이 달라진다.

클라이언트 → 한국 프록시 캐시 서버: GET /banner.png
↓
[최초 — 프록시 캐시에 없음]
프록시 → 미국 Origin 서버: GET /banner.png
↓
Origin → 프록시: 200 OK + 이미지 (느림)
↓
프록시: 이미지를 프록시 캐시에 저장, 클라이언트에게 전달
↓
[두 번째 클라이언트]
클라이언트 → 프록시: GET /banner.png
↓
프록시: 캐시에 있음 → 미국 서버에 가지 않고 바로 응답 (빠름)

인기 있는 자원일수록 프록시 캐시 히트율이 높아 응답이 빨라진다. 유튜브·넷플릭스 같은 서비스가 전 세계에 CDN(Content Delivery Network, 사실상 프록시 캐시)을 두는 이유다.

private 캐시 vs public 캐시

캐시는 누가 접근할 수 있는지에 따라 두 종류로 나뉜다.

구분 저장 위치 접근 범위 예시
private 캐시 브라우저 로컬 해당 사용자만 브라우저 디스크·메모리 캐시
public 캐시 프록시·CDN 서버 여러 사용자 공유 CDN, 회사 내부 프록시

Cache-Control: public은 프록시·CDN이 이 응답을 캐시해도 된다는 허가다. Cache-Control: private은 브라우저 캐시에만 저장하고 프록시에는 저장하지 말라는 지시다. 개인 데이터(로그인 후 마이페이지, 장바구니 등)에는 반드시 private을 써야 한다.

Cache-Control: no-cache vs must-revalidate — 프록시 관점

이 두 지시어의 차이가 프록시 캐시 맥락에서 가장 뚜렷하게 드러난다(강의 38에서 이미 확인). 프록시 캐시가 Origin 서버와 통신이 끊긴 상황에서:

  • no-cache만 있을 때: 프록시가 임의로 오래된 캐시를 200 OK로 줄 수 있다.
  • must-revalidate가 있을 때: 프록시는 반드시 504 Gateway Timeout을 반환해야 한다.
프론트 개발자에게 CDN은 익숙한 단어다. Vercel, Cloudflare, AWS CloudFront 모두 프록시 캐시 서버다. next.config.js나 응답 헤더에 Cache-Control을 설정할 때 "이 헤더가 CDN(public 캐시)에도 영향을 주는구나"라고 이해하면 설정의 의미가 명확해진다. Rails에서 expires_in 1.hour, public: true를 쓰면 Cache-Control: public, max-age=3600이 붙어 CDN에도 캐시된다.

정리

  • 프록시 캐시는 Origin 서버와 클라이언트 사이의 중간 캐시 서버다
  • 여러 클라이언트가 같은 자원을 공유 → 인기 자원일수록 효과가 크다
  • Cache-Control: public: 프록시·CDN 저장 허용
  • Cache-Control: private: 브라우저 캐시에만 저장, 프록시에는 저장 금지
  • CDN은 전 세계에 분산된 대규모 프록시 캐시 네트워크다
