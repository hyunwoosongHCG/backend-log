DataWrapper — Swagger 응답 스키마는 누가 만드나

레슨 022 DataWrapper · grape-swagger model_parsers ~12분
🎯 ppback PR #5489 리뷰 중 multisource_feedbacks.rb의 success model: DataWrapper.new(Entities::MultisourceFeedbacks::MonitorEntity)를 보고 "이게 뭘 감싸는 거지?" 싶었던 것에서 시작 — MISSION.md

실제로 본 코드

Grape API 엔드포인트의 desc 블록 안에서 이런 코드를 봤다:

desc '360 피드백 평가자 리스트 수정' do
  success model: DataWrapper.new(Entities::MultisourceFeedbacks::MonitorEntity)
end
put ':id/assessors' do
  # ...
end

이름만 보면 "응답을 DataWrapper로 감싼다"는 뜻처럼 읽힌다. 그런데 실제로 이 엔드포인트의 응답을 만드는 코드에는 DataWrapper가 등장하지 않는다. 그렇다면 이건 대체 뭘 하는 클래스인가?

이름은 Wrapper인데, 진짜 응답을 감싸지는 않는다

lib/types/data_wrapper.rb를 열어보면 힌트가 있다 — 이 클래스는 로직이 없다. 그냥 값을 들고 있는 객체다: 

class DataWrapper
  attr_reader :model, :key, :is_array, :meta, :meta_key, :wrapper_name

  def initialize(model, wrapper_name: nil, key: :data, is_array: false, meta: false, meta_key: :meta)
    @model = model
    @key = key
    @is_array = is_array
    @meta = meta
    @meta_key = meta_key
  end
end

model(어떤 Entity인지), key(응답 키, 기본 :data), is_array(배열이냐 단일 객체냐), meta/meta_key(페이지네이션 같은 부가 정보) — 전부 설명(description)일 뿐 실행 로직이 없다. 즉 이 객체는 "이 API는 { "data": ... } 모양으로 응답한다"는 사실을 문서화하기 위한 선언이다.

Q1. DataWrapper가 실제로 하는 일은?

그럼 실제로 감싸는 건 누구인가 — present

실제로 응답을 { "data": ... } 모양으로 조립하는 코드는 완전히 별개의 위치, 엔드포인트 액션 안의 present 호출이다: 

get do
  examples = policy_scope(Example).page(params[:page]).per_page(params[:per_page])

  present :data, examples, with: Entities::Example   # ← 진짜 응답 조립은 여기
  present :meta, { total: examples.total_count }
end

DataWrapper.new(...)와 present :data, ...는 코드로 강제 연결되어 있지 않다. 둘 다 :data라는 키를 쓰기로 관례적으로 약속되어 있을 뿐이다. 그래서 둘 중 하나만 바꾸면(예: present :items, ...로 키를 바꿨는데 DataWrapper 선언은 그대로 두면) 실제 응답과 Swagger 문서가 서로 다른 모양을 가리키게 된다.

Q2. { "data": [...] } 모양의 실제 JSON 응답을 런타임에 만드는 코드는?

grape-swagger가 DataWrapper를 이해하는 방법 — model_parsers.register

grape-swagger는 기본적으로 Grape::Entity만 알아본다. DataWrapper 같은 커스텀 객체를 success model:에 넘기려면, "이 타입을 만나면 이 파서로 처리해라"라고 미리 등록해둬야 한다: 

# config/initializers/grape_swagger.rb
GrapeSwagger.model_parsers.register(DataWrapperParser, DataWrapper)

이 등록 덕분에 grape-swagger가 문서를 만들다가 DataWrapper 인스턴스를 만나면 DataWrapperParser#call을 호출해서 실제 JSON Schema 조각을 받아간다: 

# lib/types/data_wrapper_parser.rb
memo[wrapper.key] = if wrapper.is_array
                      { 'type' => 'array', 'items' => { '$ref' => "#/definitions/#{name}" } }
                    else
                      { 'type' => :object, '$ref' => "#/definitions/#{name}" }
                    end
memo[wrapper.meta_key] = wrapper.meta if wrapper.meta

is_array: true면 배열 스키마를, 아니면 단일 객체 스키마를 만들고, meta가 있으면 페이지네이션 같은 부가 필드도 스키마에 추가한다.

Q3. GrapeSwagger.model_parsers.register(DataWrapperParser, DataWrapper) 등록을 빼먹으면?

배열 + 메타(페이지네이션) 선언 예시

같은 PR 리뷰에서 본 다른 예시 — 배열 응답에 페이지네이션 메타를 얹은 경우:

success model: DataWrapper.new(Entities::Users::AdminObjectiveWeightEntity,
                                is_array: true,
                                meta_key: :total_pages,
                                meta: { type: 'Integer' })

이건 "응답은 AdminObjectiveWeightEntity 배열이고, total_pages라는 정수형 메타 필드가 같이 온다"고 문서에 선언하는 것이다. 실제로 그 필드를 채우는 건 역시 액션 안의 present :meta, { total_pages: ... } 몫이다.

💡 FE 비유: OpenAPI/Swagger 스펙을 YAML로 손으로 적어본 적이 있다면, DataWrapper.new(Entity, is_array: true, ...)는 그 responses.200.schema 부분을 YAML 대신 Ruby 객체로 선언하는 것과 같다.

다만 OpenAPI YAML은 스펙과 서버 코드가 물리적으로 분리되어 있어서 어긋나기 쉽다는 게 흔한 문제인데, 여기서도 DataWrapper 선언과 present 실행이 서로 다른 코드라서 똑같은 종류의 "선언 vs 실제 구현" 어긋남 위험이 있다는 점은 동일하다.
