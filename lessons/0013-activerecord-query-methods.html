ActiveRecord 쿼리 메서드

레슨 013 find · find_by · where · order · limit ~25분
🎯 DB에서 데이터를 꺼내는 모든 코드의 뼈대 — 실무에서 매일 마주친다 — MISSION.md

출발점: "테이블에서 데이터를 어떻게 꺼내나?"

레슨 8에서 belongs_to, has_many로 관계를 정의하는 법을 배웠다. 그런데 실제로 데이터를 꺼낼 때는 다른 메서드를 쓴다. 레슨 9에서 잠깐 본 .includes도 그 중 하나다.

SQL로 치면 이런 것들이다:

-- 이런 SQL을 ActiveRecord 메서드로 표현한다
SELECT * FROM users WHERE id = 3;
SELECT * FROM users WHERE role = 'admin' ORDER BY created_at DESC LIMIT 10;
프론트에서 array.find(), array.filter()로 배열에서 원소를 꺼내듯이, ActiveRecord는 DB 테이블에서 레코드를 꺼내는 메서드 모음이다. 차이는 배열이 아니라 DB이기 때문에 — 결국 SQL 쿼리로 변환된다는 것.

1. find — "이 ID의 레코드를 줘"

가장 단순한 형태. PK(Primary Key)로 딱 한 건을 찾는다.

User.find(3)
# → SELECT * FROM users WHERE id = 3 LIMIT 1

두 가지 규칙을 외운다:

  • 찾으면 → User 객체 하나를 반환
  • 없으면 → ActiveRecord::RecordNotFound 예외 발생

레슨 12에서 봤던 rescue_from ActiveRecord::RecordNotFound, with: :not_found가 바로 이 예외를 잡는 것이다. 컨트롤러에서 find(params[:id])를 쓰고, 없는 id가 오면 404가 자동으로 나가는 이유가 여기 있다.

Q1. ReviewTemplate.find(9999)를 실행했는데 id=9999인 레코드가 없다. 어떻게 되나?

2. find_by — "이 조건에 맞는 첫 번째 레코드를 줘"

ID가 아닌 다른 컬럼으로 찾을 때 쓴다. 조건에 맞는 첫 번째 레코드 하나만 반환한다.

User.find_by(email: 'alice@example.com')
# → SELECT * FROM users WHERE email = 'alice@example.com' LIMIT 1

Member.find_by(workspace: workspace, user: current_user)
# → 레슨 12에서 pundit_user 메서드가 쓴 바로 그것

find와의 결정적 차이:

메서드 없을 때 여러 건일 때
find(id) 예외 발생 ID는 유일하므로 해당 없음
find_by(…) nil 반환 첫 번째 건만 반환

그래서 없을 수도 있는 상황에서 예외 없이 처리하고 싶을 때 find_by를 쓰고, 결과에 &. (safe navigation)으로 접근한다: 

member = Member.find_by(workspace: workspace, user: current_user)
member&.admin?   # member가 nil이면 nil, 아니면 admin?의 결과

3. where — "이 조건에 맞는 전체 레코드를 줘"

여러 건을 꺼낼 때 쓴다. SQL의 WHERE 절 그대로다. 

Review.where(status: :submitted)
# → SELECT * FROM reviews WHERE status = 'submitted'

Review.where(status: :submitted, appraisee_id: 5)
# → 조건 여러 개는 AND로 연결된다

핵심: where는 즉시 실행되지 않는다

where의 반환값은 배열이 아니다. ActiveRecord::Relation이라는 객체다. 실제 SQL은 이 Relation이 실제로 데이터를 필요로 하는 순간에 실행된다.

JS의 Promise처럼 생각하면 된다. where는 "이런 쿼리를 실행하겠다"는 계획을 만들 뿐이고, .each, .to_a, .count처럼 실제 데이터가 필요한 순간에 await처럼 실행된다. 이 덕분에 조건을 체이닝(chaining)으로 쌓을 수 있다. 
# 체이닝: 각 메서드가 Relation을 반환하므로 계속 이어 붙일 수 있다
Review.where(status: :submitted)
      .order(created_at: :desc)
      .limit(10)
# → SELECT * FROM reviews
#     WHERE status = 'submitted'
#     ORDER BY created_at DESC
#     LIMIT 10
# SQL은 단 한 번만 실행된다

4. 자주 쓰는 체이닝 메서드

메서드 역할 SQL 대응
.order(col: :desc) 정렬 ORDER BY col DESC
.limit(n) 최대 n건 LIMIT n
.count 건수 세기 SELECT COUNT(*)
.first / .last 첫/마지막 한 건 (없으면 nil) LIMIT 1
.exists? 존재하면 true SELECT 1 ... LIMIT 1
.pluck(:col) 특정 컬럼 값만 배열로 SELECT col FROM ...

레슨 12에서 봤던 record.reviews.kept.exists? — 이제 완전히 읽힌다. record.reviews는 연관관계(has_many), .kept는 다음 레슨에서 배울 scope, .exists?는 방금 배운 체이닝 메서드다.

5. 한눈에 비교

# 하나를 ID로 — 없으면 예외
User.find(3)

# 하나를 조건으로 — 없으면 nil
User.find_by(email: 'alice@example.com')

# 여러 건을 조건으로 — Relation 반환 (체이닝 가능)
User.where(role: :admin).order(created_at: :desc).limit(5)

# 개수만 필요할 때
User.where(role: :admin).count

# id 배열만 필요할 때 (객체를 만들지 않아서 빠름)
User.where(role: :admin).pluck(:id)   # → [1, 4, 7, ...]
Q3. Review.where(status: :draft)를 실행한 직후, SQL이 DB에 날아갔는가?

핵심 요약

메서드 반환값 없을 때
find(id) 객체 하나 예외 발생 → 404
find_by(…) 객체 하나 nil
where(…) Relation (체이닝 가능) 빈 Relation (예외 없음)
