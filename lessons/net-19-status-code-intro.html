🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 상태코드 — 서버가 요청을 어떻게 처리했는지 숫자 세 자리로 알려주는 "결과 코드" 전체 지도를 그린다

핵심 한 줄

HTTP 상태코드(Status Code)는 클라이언트가 보낸 요청을 서버가 얼마나 성공적으로 처리했는지 세 자리 숫자로 알려주는 약속이다 — 앞자리 한 글자만 봐도 결과의 종류(성공·실패·리다이렉션)를 즉시 알 수 있다.

왜 필요한가

프론트 코드에서 axios.get('/api/users')를 호출하면 서버는 응답 메시지를 돌려준다. 그런데 그 응답이 성공인지, 실패인지, 다른 주소로 가야 하는지 어떻게 알 수 있을까? 바디(body) 안에 "ok": false를 심어서? 그러면 모든 팀이 서로 다른 규칙을 쓰게 된다.

HTTP는 이 문제를 표준 상태코드로 해결한다. 브라우저, axios, Rails — 모두가 같은 숫자 약속을 쓰기 때문에 별도 약속 없이도 "성공이냐 실패냐"를 즉시 구분할 수 있다.

Promise가 fulfilled / rejected 두 상태로 비동기 결과를 알려주듯, HTTP 상태코드는 세 자리 숫자로 요청 결과를 분류한다. axios가 2xx이면 .then(), 4xx·5xx이면 .catch()로 보내는 이유가 바로 이 규칙 때문이다.

1xx~5xx 대분류

범위 이름 의미 실무 빈도
1xx Informational (정보) 요청을 수신했고 처리 중 거의 안 씀
2xx Successful (성공) 요청을 정상 처리함 매우 자주
3xx Redirection (리다이렉션) 완료하려면 추가 동작 필요 자주
4xx Client Error (클라이언트 오류) 클라이언트가 잘못된 요청을 보냄 자주
5xx Server Error (서버 오류) 서버가 정상 처리를 못 함 간헐적

앞자리 한 글자 규칙

3자리 코드의 백의 자리(첫 번째 숫자)가 분류를 결정한다. 예를 들어 201은 2xx이므로 "성공", 404는 4xx이므로 "클라이언트 오류"다.

덕분에 표준에 없는 새 코드를 서버가 내보내더라도 클라이언트는 첫 자리만 보고 "일단 어느 범주인지"를 알 수 있다. 예: 299라는 코드를 모른다면 → "2xx니까 성공 계열"로 처리.

1xx — 거의 안 쓴다

100 Continue 같은 코드가 있지만, 실무에서 직접 다루는 일은 드물다. HTTP/1.1 이전 시절 흔적에 가깝다고 이해하면 충분하다.

실무 연결 — axios 인터셉터

axios는 상태코드를 기준으로 Promise를 resolve/reject한다. 보통 프로젝트 초기에 전역 인터셉터를 이렇게 설정한다. 

// 프론트: axios 인터셉터 예시
axios.interceptors.response.use(
  (response) => response,          // 2xx → 그대로 반환
  (error) => {
    const status = error.response?.status;
    if (status === 401) router.push('/login');   // 인증 없음
    if (status === 403) toast.error('권한 없음'); // 인가 실패
    if (status === 404) toast.error('없는 리소스');
    if (status >= 500) toast.error('서버 오류');
    return Promise.reject(error);
  }
);

Rails/Grape 쪽에서는 rescue_from으로 예외를 상태코드와 연결한다. 이 연결 방식은 4xx·5xx 레슨에서 자세히 다룬다.

정리

  • 상태코드는 요청 결과를 세 자리 숫자로 표현하는 HTTP 표준 약속
  • 백의 자리 1개만으로 1xx~5xx 다섯 범주 즉시 구분 가능
  • axios가 2xx면 resolve, 그 외면 reject하는 이유가 바로 이 규칙
  • 1xx는 거의 안 씀, 2xx/3xx/4xx/5xx가 실무 핵심
