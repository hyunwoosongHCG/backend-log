자동 타임스탬프와 숨은 콜백

레슨 019 created_at · updated_at · ActiveRecord::Timestamp · 콜백 ~15분
🎯 ppback Objective의 updated_at이 "어디서" 갱신되는지 추적하다가 나온 질문을 완성한다 — MISSION.md

질문에서 시작한다

ppback의 app/models/objective.rb를 아무리 뒤져봐도 updated_at을 직접 세팅하는 코드는 한 줄도 없다. 

def accept_check_ins_and_create_history(current_user_id, values)
  accept_check_ins
  update!(progress: calculate_progress, stage: 'open')
  update_obj_progress_history
  ...
end

그런데도 이 메서드가 실행되고 나면 objective.updated_at은 방금 시각으로 바뀌어 있다. 코드에 없는 동작이 어떻게 일어나는가?

파트 1: t.timestamps — 컬럼만 있으면 프레임워크가 알아서 채운다

마이그레이션(레슨 2026-06-25에서 배운)에서 테이블을 만들 때 보통 이렇게 쓴다. 

create_table :objectives do |t|
  t.string :name
  t.timestamps   # ← created_at, updated_at 두 컬럼을 한 번에 생성
end

t.timestamps로 만든 두 컬럼이 테이블에 존재하기만 하면, ActiveRecord는 별도 코드 없이 다음을 자동으로 한다.

컬럼 채워지는 시점 횟수
created_at 레코드가 처음 INSERT될 때 딱 1번
updated_at save / update / update!로 UPDATE가 실제로 나갈 때 매번 
objective = Objective.create(name: "Q3 목표")
objective.updated_at        # => 방금 생성된 시각 (자동)

objective.update(stage: :open)
objective.updated_at        # => 방금 시각으로 다시 자동 갱신
프론트에서는 createdAt: new Date()처럼 값을 직접 만들어 넣어야 한다.

Rails는 컬럼 이름이 created_at / updated_at이기만 하면 이 값 채우기를 아예 대신 해준다. 내가 신경 쓸 일이 사라지는 것이다.
Q1. key_result.update(weight: 3)를 호출했다. key_result.created_at은 어떻게 되는가?

파트 2: 이게 왜 "콜백"인가

Rails 모델에 before_save, after_create 같은 콜백을 직접 선언하면, 저장 전후 특정 시점에 내가 지정한 메서드가 자동으로 실행된다. 

# 내가 직접 선언하는 콜백
class Objective < ApplicationRecord
  before_save :log_something

  def log_something
    Rails.logger.info "저장 직전!"
  end
end

created_at/updated_at 자동 관리는 원리가 똑같다. 다만 내가 선언한 게 아니라, ActiveRecord::Timestamp라는 모듈이 모든 모델에 이미 이 콜백을 걸어둔 것이다. 그래서 모델 코드에는 아무것도 안 보이지만, 저장 직전에 "타임스탬프 컬럼 채우기" 콜백이 항상 조용히 실행되고 있다.

① objective.update!(stage: 'open') 호출
↓
② ActiveRecord::Timestamp의 숨은 before_save 실행 → updated_at = 지금
↓
③ 내가 before_save를 선언했다면 그것도 같이 실행
↓
④ 실제 UPDATE SQL 실행
Q2. created_at/updated_at이 모델 코드 한 줄 없이 자동으로 채워지는 이유는?

실무 연결: 자동인 것과 자동이 아닌 것의 경계

오늘 조사하면서 진짜 중요했던 지점은 이거다 — 자동 갱신은 "저장되는 그 레코드 자신의 컬럼"에만 적용된다.

ppback에는 ObjProgressHistory라는, Objective의 진행률 변화를 기록하는 별도 테이블이 있다. Objective가 저장될 때 이 테이블에도 자동으로 행이 생기면 좋겠지만, 그런 일은 일어나지 않는다. 대신 코드에 이렇게 명시적으로 써 있다. 

# app/models/objective.rb
def update_obj_progress_history
  ObjProgressHistory.create!(
    objective_id: id,
    progress:,
    time_at: updated_at,   # 방금 배운 그 자동 updated_at을 여기서 스냅샷으로 활용
  )
end

즉 "같은 row를 저장하는 것"은 프레임워크가 대신해주지만, "그 김에 다른 테이블에 행을 하나 더 만드는 것"은 반드시 내가 명시적으로 호출해야 한다. 이 경계를 헷갈리면, 나중에 "왜 이 경로에서는 이력이 안 남지?" 같은 버그를 만들기 쉽다 (실제로 ppback 가중치 수정 워커에서 이 호출이 빠진 사례를 발견했다).

Q3. Objective를 저장하면서 동시에 별도 테이블(ObjProgressHistory)에 이력을 남기고 싶다. 어떻게 해야 하는가?
