출발점 — "allowProductionPhase2Deployment, 이거 임의로 지은 이름인가?"

PR 리뷰 중 이 코드를 보고 궁금해졌다. 

const allowProductionPhase2Deployment = app.node.tryGetContext('allowProductionPhase2Deployment')
if (isDeploymentExplicitlyAllowed(allowProductionPhase2Deployment)) {
  const perplProduction = new PerformancePlusStack(app, 'ProductionPerformancePlusStack', { ... })
}

"이 문자열, CDK가 정해놓은 예약어인가 아니면 아무렇게나 지어도 되는 건가?" — 답부터 말하면 완전히 임의의 이름이다. 이 레슨은 CDK Context가 뭐고, 왜 이런 "위험한 스택은 명시적으로 켜야만 배포되게" 만드는 패턴을 쓰는지 정리한다.

CDK Context — 두 종류를 구분해야 한다

app.node.tryGetContext(key)는 CDK 앱에 전달된 컨텍스트 값에서 그 문자열 키를 그대로 조회하는 것뿐이다. 여기엔 두 종류가 있다.

  • CDK가 정의한 예약된 feature flag — cdk.json의 context 블록에 있는 @aws-cdk/aws-lambda:recognizeLayerVersion 같은 값들. @aws-cdk/<모듈>:<플래그명> 네임스페이스를 CDK 프레임워크 자체가 정의해두고, 특정 버전 이후 동작 방식을 바꿀 때 쓴다. 이건 임의로 짓는 게 아니라 CDK 공식 문서에 등록된 값이다.
  • 사용자가 임의로 짓는 커스텀 키 — allowDemoDeployment, allowProductionPhase2Deployment 같은 것들. CDK는 이런 키에 대해 아무 스키마도 강제하지 않는다. CLI에서 -c allowProductionPhase2Deployment=true로 넘길 때와 코드에서 tryGetContext('allowProductionPhase2Deployment')로 읽을 때, 문자열만 정확히 일치하면 된다. 이름 자체는 팀이 정한 컨벤션일 뿐이다.

배포 가드 패턴 — 위험한 스택은 opt-in으로

이 커스텀 컨텍스트 키가 실제로 쓰이는 곳은 "이 스택은 기본적으로 배포 대상에서 빼고, 명시적으로 플래그를 줬을 때만 포함시킨다"는 안전장치다. 

// deployment-guard.ts
export function isDeploymentExplicitlyAllowed(value: unknown): boolean {
  return value === true || value === 'true'
}

// bin/performance-plus.ts
const allowDemoDeployment = app.node.tryGetContext('allowDemoDeployment')
if (isDeploymentExplicitlyAllowed(allowDemoDeployment)) {
  const perplDemo = new PerformancePlusStack(app, 'Demo-PerformancePlusStack', { ... })
}

이렇게 해두면 cdk deploy --all을 실행해도 demo나 production Phase 2처럼 "지금 잘못 배포하면 실서비스에 위험한" 스택은 아예 CDK 앱에 등록조차 되지 않는다. 반드시 cdk deploy -c allowDemoDeployment=true Demo-PerformancePlusStack처럼 사람이 의도적으로 플래그를 켠 경우에만 그 스택이 존재한다. "실수로 배포됨"이라는 사고 자체가 코드 레벨에서 원천 차단된다 — 리뷰나 문서로 "조심하세요"라고 적어두는 것보다 훨씬 강한 안전장치다.

이 레포에서는 이 패턴을 한 헬퍼 함수를 재사용해서 demo(#273)와 production Phase 2(#274) 두 곳에 똑같이 적용했다 — 같은 문제(위험한 스택의 실수 배포)에 같은 해법을 반복 검증 없이 재사용한 셈이다.

CodePipeline이 ECS에 배포하는 방식 — imageDef.json 아티팩트

파이프라인이 빌드한 이미지를 실제로 어느 ECS 서비스에 배포할지는 EcsDeployAction이 결정한다. 

const sidekiqDeployAction = new codepipeline_actions.EcsDeployAction({
  actionName: 'SidekiqDeploy',
  service: props.x.sidekiqService,
  imageFile: buildOutput.atPath("sidekiqImageDef.json")   // ← CodeBuild가 만든 아티팩트 안의 파일
});

sidekiqImageDef.json은 "이 서비스에 배포할 이미지의 ECR URI가 이거다"라고 적힌 JSON 파일이고, 이 파일 자체는 CodeBuild의 buildspec(aws/buildspec-deploy.yml)이 빌드 단계에서 만들어낸다. 중요한 건 그 buildspec은 인프라 레포(CDK 코드)가 아니라 애플리케이션 레포(theplus-back)에 있다는 것 — "왜 이 태스크가 파이프라인을 안 타지?"를 조사할 때, 인프라 레포만 봐서는 절반만 확인할 수 있다는 걸 이번에 배웠다.

triggerOnPush — GitHub push가 자동으로 파이프라인을 돌리는가

const sourceAction = new codepipeline_actions.CodeStarConnectionsSourceAction({
  actionName: 'GithubSourceActionV2',
  owner: 'hcgtheplus',
  repo: props.x.repo,
  connectionArn: "...",
  triggerOnPush: false,   // ← GitHub push가 파이프라인을 자동 실행하지 않는다
  output: sourceOutput,
});

triggerOnPush는 소스 액션이 GitHub push 이벤트에 대해 AWS가 자동으로 파이프라인 실행을 걸어줄지(true) 아니면 push가 일어나도 가만히 있고 수동 또는 다른 외부 트리거를 기다릴지(false)를 결정한다. 이 레포는 계속 false였다 — 즉 push한다고 바로 배포되지 않는다.

triggerOnPush는 GitHub Actions의 on: push(자동 트리거) vs on: workflow_dispatch(수동 트리거) 차이와 정확히 같은 개념이다. 그리고 "위험한 스택은 opt-in context로만 배포 가능"한 패턴은, 프론트에서 프로덕션 배포 전에 별도 승인 스텝(manual approval gate)을 두는 것과 같은 발상 — "실수로 자동 실행되면 안 되는 것"을 코드/설정 레벨에서 강제로 막아두는 것이다.
