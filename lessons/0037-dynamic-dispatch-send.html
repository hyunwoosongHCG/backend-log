동적 메서드 디스패치 — send로 메서드 이름을 조립한다

레슨 037 send · 메타프로그래밍 · 화이트리스트 ~12분
🎯 코드를 보고 "이 요청이 어떤 경로로 흘러가는지" 머릿속에 그릴 수 있다 — MISSION.md

실제로 본 코드

레슨 36에서 본 History#target은 Review, Objective, Task 등 여러 모델을 가리킬 수 있었다. 문제는 "가리키는 대상에 따라 다른 알림 로직을 실행해야 한다"는 것. 코드가 이렇게 생겼다: 

# app/models/history.rb
def notify
  send("notify_#{target_type.underscore}")
  #    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ target_type이 "Review"면 notify_review 호출
end

def notify_review
  ...
end

def notify_objective
  ...
end

target_type이 "Review"면 "notify_review"라는 문자열이 만들어지고, send가 그 이름의 메서드를 실행 시점에 호출한다. 소스 코드 어디에도 notify_review를 직접 호출하는 줄이 없다 — 이름이 런타임에 문자열 조합으로 결정된다.

send란 무엇인가

Ruby에서 보통 메서드는 객체.메서드명처럼 이름을 코드에 직접 적어서 호출한다. send는 그 이름을 문자열이나 심볼로 대신 넘겨서 호출하게 해준다. 

history.notify_review        # 보통 호출 — 이름이 소스에 그대로 적혀있음
history.send("notify_review") # send 호출 — 이름이 문자열로 전달됨, 같은 결과

두 줄은 결과가 완전히 같다. 다른 점은 메서드 이름을 변수로 다룰 수 있다는 것 — 그래서 "notify_#{target_type.underscore}"처럼 런타임에 조립한 이름도 호출할 수 있다. 이런 걸 메타프로그래밍(metaprogramming)이라고 부른다 — "코드가 코드를 다루는 코드".

JS에서 객체의 속성에 문자열로 접근하는 것과 같은 아이디어다. 
// JS: 대괄호 표기법으로 동적 접근
const key = 'notifyReview';
obj[key]();               // obj.notifyReview()와 동일

// JS: 문자열 조합으로 핸들러 이름 결정
const handler = `handle${eventType}`;
this[handler](event);     // Ruby의 send와 정확히 같은 패턴
obj[key]()가 Ruby의 obj.send(key)와 1:1로 대응한다. 프론트에서 이벤트 타입별로 핸들러 이름을 조합해 호출해본 경험이 있다면 이미 써본 패턴이다.

왜 이렇게 짜는가 — case/when 대신 send

같은 걸 case로 짤 수도 있었다: 

def notify
  case target_type
  when 'Review' then notify_review
  when 'Objective' then notify_objective
  when 'Task' then notify_task
  # ABLE_TYPE에 새 모델이 추가될 때마다 여기도 한 줄씩 늘어난다
  end
end

send 버전은 ABLE_TYPE에 새 모델이 추가돼도 notify_새모델 메서드만 정의하면 끝난다 — notify 메서드 자체는 안 바뀐다. 분기문이 늘어나는 걸 막아주는 대신, "이 메서드가 실제로 어디서 호출되는지"가 소스 코드만 봐서는 안 보이게 된다는 비용을 지불한다.

메타프로그래밍의 위험 — 왜 화이트리스트가 필요한가

send는 private 메서드까지도 호출할 수 있을 만큼 강력하다. 만약 target_type이 사용자 입력에서 그대로 온 값이었다면, send는 개발자가 의도하지 않은 임의의 메서드(예: delete_all, destroy 등)를 실행하는 통로가 될 수 있다. 레슨 36에서 본 ABLE_TYPE 화이트리스트가 target_type을 미리 정해진 값으로만 제한하는 것이 바로 이 위험을 막는 장치다.

기억할 규칙 send에 넘기는 문자열이 외부 입력에서 왔다면, 반드시 화이트리스트로 검증한 뒤에 호출한다. 검증 없이 그대로 넘기면 eval과 비슷한 위험을 갖게 된다.
