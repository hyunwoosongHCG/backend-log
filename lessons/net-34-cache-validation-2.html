🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 캐시 — 날짜가 아닌 버전 태그(ETag)로 캐시를 더 정확하게 검증하는 방법을 배운다

핵심 한 줄

ETag(Entity Tag)는 서버가 자원에 붙이는 버전 식별자다. 브라우저는 만료 후 재요청 시 If-None-Match에 ETag를 담아 보내고, 서버가 같으면 304, 다르면 200 + 새 자원을 반환한다.

왜 ETag인가 — Last-Modified의 한계

이전 강의(검증 헤더와 조건부 요청1)에서 본 Last-Modified / If-Modified-Since는 날짜 비교 방식이라 두 가지 한계가 있었다.

  • 1초 이하 변경 미감지: 1초 안에 파일이 수정되면 날짜가 같아 변경을 놓친다.
  • 수정 후 원복 미감지: 내용을 바꿨다가 되돌려도 날짜가 달라 불필요하게 재전송한다.

ETag는 날짜 대신 서버가 자원의 내용 기반으로 생성한 임의 문자열이다. 내용이 같으면 ETag가 같고, 내용이 다르면 ETag가 다르다. 날짜 정밀도 문제가 없고, 수정-원복도 정확히 감지한다.

ETag / If-None-Match 흐름

[최초 요청] 브라우저 → 서버: GET /logo.png
↓
서버 → 브라우저: 200 OK
Cache-Control: max-age=60
ETag: "v1.3-abc123"
+ 바디(이미지 전체)
↓
브라우저: 이미지 + ETag 값을 캐시에 저장
↓
[60초 이내] 캐시 유효 → 서버에 묻지 않고 캐시 사용
↓
[60초 경과 — 캐시 만료] 브라우저 → 서버:
GET /logo.png
If-None-Match: "v1.3-abc123"
↓
서버: 현재 자원의 ETag와 If-None-Match 값을 비교
↓ (동일 — 변경 없음)
서버 → 브라우저: 304 Not Modified (바디 없음)
Cache-Control: max-age=60
ETag: "v1.3-abc123"
↓
브라우저: 304 수신 → 캐시 자원 재사용, 유효기간 갱신

ETag가 다를 때 (자원이 변경된 경우)

서버: ETag가 다르다 → 200 OK
ETag: "v1.4-def456" (새 버전)
+ 바디(새 이미지)
↓
브라우저: 새 자원 + 새 ETag로 캐시 갱신

ETag는 블랙박스

ETag 값의 생성 방식은 서버가 알아서 결정한다. 파일 내용의 해시값, 버전 번호, 타임스탬프 + 크기 조합 등 어떤 방식이든 상관없다. 클라이언트(브라우저)는 ETag 값이 무슨 뜻인지 해석하지 않는다 — 그냥 받아서 저장하고, 재요청 시 그대로 돌려줄 뿐이다.

이 때문에 캐시 로직의 복잡성이 서버 안에 숨겨진다(블랙박스). 클라이언트는 메커니즘을 몰라도 올바르게 동작한다.

Last-Modified vs ETag 비교

항목 Last-Modified / If-Modified-Since ETag / If-None-Match
비교 기준 날짜(시각) 서버가 부여한 문자열 태그
1초 이하 변경 감지 못할 수 있음 정확히 감지
수정 후 원복 날짜 바뀌어 불필요 재전송 내용 같으면 ETag 같아 304
서버 제어 수정 날짜 기반 — 파일 시스템 의존 ETag 생성 로직을 서버가 완전히 제어
우선순위 ETag가 있으면 ETag를 우선 사용 사실상 현대 표준
React에서 컴포넌트 props를 비교할 때 날짜가 아니라 값 자체를 비교한다(memo, useMemo). ETag도 그와 같다 — 언제 수정했냐가 아니라 "지금 내용이 같냐"를 비교한다. 빌드 시 파일명에 붙는 해시(main.a3f9c2.js)가 ETag와 유사한 역할을 한다 — 내용이 달라지면 해시가 달라지고, 브라우저는 새 파일로 인식한다.

정리

  • ETag는 서버가 자원에 부여하는 버전 식별 문자열이다
  • 만료 후 재요청 시 브라우저는 If-None-Match에 ETag를 담아 보낸다
  • 서버가 비교해 같으면 304(바디 없음), 다르면 200 + 새 자원을 반환한다
  • ETag 생성 방식은 서버가 결정 — 클라이언트는 값을 해석하지 않는다(블랙박스)
  • 날짜 기반 Last-Modified보다 정밀하며 현대 캐시의 표준 방식이다
