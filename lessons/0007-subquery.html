서브쿼리란?

쿼리 안에 또 다른 쿼리를 넣는 것이다. 바깥 쿼리가 안쪽 쿼리의 결과를 입력으로 받는 구조다. 

-- 기본 구조
SELECT * FROM 바깥테이블
WHERE id IN (
  SELECT id FROM 안쪽테이블 WHERE 조건  ← 서브쿼리
)
JavaScript에서 배열 필터를 중첩하는 것과 비슷하다:

const ids = users.filter(u => u.active).map(u => u.id)
const result = orders.filter(o => ids.includes(o.userId))

서브쿼리는 이걸 SQL 한 문장으로 쓰는 것이다.

ppback 컨트롤러에서 서브쿼리가 생기는 위치

버그를 수정했던 컨트롤러를 다시 보자:

# app/controllers/v2/admins/appraisal_open_results.rb

# Step 1: 기본 필터
appraisees = AppraisalAppraisee
  .joins(appraisal_group: %i[appraisal appraisal_group_setting])
  .where(appraisal: { workspace_id: workspace.id })
  ...

# Step 2: AppraiseesQuery로 파라미터 기반 추가 필터
appraisees = AppraiseesQuery.new(relation: appraisees).filtered(params)
# 이 시점에서 appraisees는 joins + where가 복잡하게 쌓인 상태

# Step 3: ← 서브쿼리 생성
appraisees = AppraisalAppraisee
  .where(id: appraisees.select(:id))   # ← 여기서 서브쿼리 발생
  .joins(appraisal_group: ...)
  .order(...)
  .page(params[:page])

appraisees.select(:id)가 만들어내는 SQL:

SELECT * FROM appraisal_appraisees
WHERE id IN (
  SELECT id FROM appraisal_appraisees    ← Step 2의 쿼리 전체가 서브쿼리로 들어감
  INNER JOIN appraisal_groups ON ...
  INNER JOIN appraisals ON ...
  WHERE appraisal_result_formal_objection.status = 'review'  ← 버그 지점
)

왜 서브쿼리로 감싸나?

Step 2에서 AppraiseesQuery가 여러 JOIN과 조건을 쌓은 상태로 정렬·페이지네이션을 추가하면 JOIN이 중복되거나 ORDER BY가 잘못된 컨텍스트에서 적용되는 문제가 생긴다. 서브쿼리로 감싸면 ID 목록만 뽑아내고, 바깥 쿼리에서 깔끔하게 새로 시작할 수 있다.

# 서브쿼리 없이 하려면?
appraisees
  .joins(...)        # Step 2에서 이미 joins가 쌓여 있음
  .joins(...)        # 여기서 같은 joins를 또 추가 → 중복 JOIN → 예기치 않은 결과
  .order(...)
  .page(...)

# 서브쿼리로 감싸면?
AppraisalAppraisee
  .where(id: appraisees.select(:id))  # Step 2 결과는 "ID 목록"으로만 쓰임
  .joins(...)        # 바깥에서 정렬/페이지용 JOIN을 깔끔하게 다시 선언
  .order(...)
  .page(...)

select(:id)는 언제 실행되나?

즉시 실행되지 않는다. ActiveRecord는 쿼리를 lazy하게 빌드한다 — 실제로 결과가 필요한 순간에만 SQL을 DB에 보낸다. 

appraisees.select(:id)
# Ruby 객체 상태: "SELECT id FROM appraisal_appraisees WHERE ..." 라는 계획만 있음
# DB에 아직 아무것도 보내지 않음

AppraisalAppraisee.where(id: appraisees.select(:id))
# 이 시점에서도 아직 실행 안 됨

.page(params[:page]).per(params[:size])
# 실제로 결과가 필요한 순간 → 서브쿼리 포함한 전체 SQL이 한 번에 DB로 날아감
JavaScript의 Promise chain과 비슷하다 — .then()을 아무리 연결해도 실제 실행은 런타임이 결정한다. ActiveRecord는 .each, .to_a, .count 등 "결과가 필요한 메서드"를 만날 때 쿼리를 실행한다.

정리

개념 설명
WHERE id IN (...) 안쪽 쿼리 결과 ID 목록에 속하는 행만 선택
relation.select(:id) SQL 서브쿼리 조각을 만드는 것 (즉시 실행 아님)
서브쿼리로 감싸는 이유 복잡한 중간 쿼리를 ID 목록으로 격리하고 바깥에서 깔끔하게 재시작
