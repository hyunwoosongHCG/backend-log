ActiveRecord 연관관계

레슨 008 ActiveRecord · belongs_to · has_one · has_many · joins · preload · N+1 ~30분
🎯 미션 연결: 버그 수정에서 has_one :appraisal_result_formal_objection이 있어야 joins(:appraisal_result_formal_objection)가 가능했다. 왜 그런지 이해한다. — 워크로그

연관관계 선언이 하는 일

Rails 모델에서 연관관계를 선언하면 두 가지를 얻는다:

  • 편의 메서드: appraisee.appraisal_result_formal_objection 같은 메서드
  • JOIN 단서: ActiveRecord가 JOIN할 때 어떤 컬럼을 연결할지 알게 됨

선언하지 않으면 joins(:appraisal_result_formal_objection)을 써도 Rails가 어떤 FK로 연결할지 몰라 에러가 난다.

belongs_to — FK가 내 테이블에 있다

appraisal_result_formal_objections 테이블
┌────┬──────────────────────┬────────┐
│ id │ appraisal_appraisee_id│ status │  ← appraisal_appraisee_id가 FK
└────┴──────────────────────┴────────┘
# app/models/appraisal_result_formal_objection.rb
class AppraisalResultFormalObjection < ApplicationRecord
  belongs_to :appraisal_appraisee  # FK(appraisal_appraisee_id)가 이 테이블에 있음
end

belongs_to는 "나한테 FK가 있다"는 선언이다. "내가 상대방에게 속한다"는 의미다.

React에서 자식 컴포넌트가 parentId prop을 받는 것처럼, belongs_to 모델은 상대방의 ID를 자기 테이블에 저장한다.

has_one — FK가 상대 테이블에 있다 (1:1)

# app/models/appraisal_appraisee.rb
class AppraisalAppraisee < ApplicationRecord
  has_one :appraisal_result_formal_objection  # FK가 상대 테이블에 있음
end

has_one은 "상대방 테이블에 내 ID를 참조하는 FK가 있다"는 선언이다. 1:1 관계이므로 한 appraisee에 이의신청이 최대 하나다. 

appraisee = AppraisalAppraisee.find(1)
appraisee.appraisal_result_formal_objection
# SELECT * FROM appraisal_result_formal_objections
# WHERE appraisal_appraisee_id = 1
# LIMIT 1

has_many — FK가 상대 테이블에 있다 (1:N)

members 테이블
┌────┬─────────┬──────────────┐
│ id │ user_id │ workspace_id │  ← user_id가 FK
└────┴─────────┴──────────────┘
# app/models/user.rb
class User < ApplicationRecord
  has_many :members  # 한 유저가 여러 워크스페이스에 멤버로 속할 수 있음
end

user.members  # SELECT * FROM members WHERE user_id = ?

has_many는 has_one과 구조가 같다 — FK가 상대 테이블에 있다. 차이는 결과가 배열이라는 것이다.

세 가지 한눈에 보기

선언 FK 위치 결과 예시
belongs_to 내 테이블 단일 객체 이의신청 → 평가대상자
has_one 상대 테이블 단일 객체 평가대상자 → 이의신청
has_many 상대 테이블 배열 유저 → 멤버십들

joins vs preload — 목적이 다르다

둘 다 연관 데이터를 가져오는 방법이지만 쓰임새가 완전히 다르다.

joins — 필터링·정렬용

SQL JOIN을 추가한다. WHERE 조건이나 ORDER BY에 상대 테이블 컬럼을 쓸 때 사용한다.

AppraisalAppraisee
  .joins(:appraisal_result_formal_objection)
  .where(appraisal_result_formal_objection: { status: :review })

-- 생성되는 SQL
SELECT appraisal_appraisees.*
FROM appraisal_appraisees
INNER JOIN appraisal_result_formal_objections
  ON appraisal_result_formal_objections.appraisal_appraisee_id = appraisal_appraisees.id
WHERE appraisal_result_formal_objections.status = 1

주의: joins로 가져온 appraisee에서 appraisee.appraisal_result_formal_objection을 Ruby에서 접근하면 추가 쿼리가 발생한다. joins는 JOIN을 SQL에 추가할 뿐, 연관 데이터를 Ruby 메모리에 올리지 않는다.

preload — N+1 방지용

별도 쿼리를 한 번 더 날려서 연관 데이터를 Ruby 메모리에 미리 올린다. 연관 데이터를 Ruby에서 접근할 때 사용한다.

appraisees = AppraisalAppraisee
  .preload(:appraisal_result_formal_objection)

-- 생성되는 SQL (두 번)
SELECT * FROM appraisal_appraisees ...
SELECT * FROM appraisal_result_formal_objections
  WHERE appraisal_appraisee_id IN (1, 2, 3, ...)

이후 appraisee.appraisal_result_formal_objection을 아무리 호출해도 추가 쿼리 없음. 이미 메모리에 있기 때문이다.

비유:
joins = 도서관에서 책을 찾는 인덱스 (위치만 알려줌)
preload = 책을 미리 내 책상 위에 꺼내놓기 (바로 펼쳐볼 수 있음)

ppback 컨트롤러에서 두 가지를 같이 쓰는 이유

appraisees = AppraisalAppraisee
  .where(id: appraisees.select(:id))
  .joins(appraisal_group: %i[appraisal appraisal_group_setting])  # 정렬용
  .order(...)
  .preload(:appraisal_result_formal_objection, ...)               # 렌더링용
  .page(params[:page])
  • joins: ORDER BY에 appraisal 컬럼을 쓰기 위해 JOIN 필요
  • preload: 결과를 JSON으로 렌더링할 때 각 appraisee의 이의신청 데이터가 필요 → N+1 방지
