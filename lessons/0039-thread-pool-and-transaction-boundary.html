스레드 풀과 트랜잭션 경계 — 비동기 부수효과가 커밋을 앞지르는 문제

🎯 코드를 보고 "이 요청이 어떤 경로로 흘러가는지" 머릿속에 그릴 수 있다 — MISSION.md

출발점 — "이건 굳이 동기로 안 해도 되지 않나?"

"판매자에게 판매 건수를 채웠다고 푸시 알림 보내기", "로그인 시 포인트 적립"처럼, 요청을 처리하는 핵심 흐름(로그인 성공, 판매 건수 갱신)과 결과가 달라도 되는 부수효과가 있다. 이런 걸 별도 스레드로 보내면 메인 흐름은 그 부수효과를 기다리지 않고 바로 응답할 수 있다 — 디버깅은 어려워지지만 인터랙션은 빨라진다는 직관은 맞다.

그런데 여기서 "특정 로직을 별도 스레드로 보내면, 그 안에서 난 예외는 원래 스레드의 rescue로 안 잡힌다"는 걸 알게 됐다면 — 정확히 봤다. 다만 이유가 하나 더 있다. 이번 레슨에서는 스레드 풀이 정확히 무엇이고, 왜 트랜잭션과 스레드를 함께 생각해야 하는지를 정리한다.

스레드 풀(Thread Pool)이란 무엇인가

OS 스레드를 만들고 없애는 건 공짜가 아니다 — 커널이 컨텍스트 전환, 스택 메모리 할당을 해야 한다. 요청이 들어올 때마다 Thread.new로 새 스레드를 만들면, 트래픽이 몰릴 때 스레드가 무한정 늘어나서 메모리와 CPU가 고갈될 수 있다.

스레드 풀은 미리 정해진 개수(N개)의 워커 스레드를 만들어두고, 작업을 큐에 쌓아뒀다가 노는 스레드가 꺼내서 처리 → 끝나면 스레드를 죽이지 않고 풀에 반납해서 재사용하는 자원 관리 장치다. "비동기로 실행돼서 빠르다"는 그 결과지, 스레드 풀이 존재하는 목적 자체는 아니다 — 목적은 스레드 생성 비용과 자원 고갈을 막는 것이다.

실무 스택에서 이 스레드 풀이 실체화된 것이 Sidekiq이다. Sidekiq 프로세스는 내부에 워커 스레드 풀(기본 concurrency 5~25개)을 두고, Redis 큐에서 job을 꺼내 각 스레드가 처리한다. 웹 요청을 처리하는 프로세스와는 완전히 분리된 프로세스이자 스레드 집합이다.

트랜잭션은 "스레드" 단위로 스코프된다

레슨 24에서 트랜잭션을 "여러 DB 작업을 하나의 단위로 묶어서 전부 성공하거나 전부 실패하게 만드는 것"이라고 배웠다. 이 트랜잭션이 실제로 걸리는 대상은 DB 커넥션이다. Rails는 커넥션 풀에서 스레드마다 자기만의 커넥션을 체크아웃한다 (스레드 로컬). 그래서 다음 코드는 겉보기와 다르게 동작한다. 

ActiveRecord::Base.transaction do
  sale.update!(count: sale.count + 1)     # ① 커넥션 A, 아직 커밋 전

  Thread.new do
    PushNotifier.notify(seller)           # ② 별도 스레드 → 별도 커넥션(B) 또는 DB 커넥션 없이 실행
  end
end
# ③ 여기서 커밋 (또는 롤백)

②는 ①과 같은 트랜잭션에 속하지 않는다 — 아예 다른 커넥션이거나, DB 접근이 없다면 트랜잭션 개념 자체가 적용되지 않는다. 게다가 스레드 생성은 매우 빠르기 때문에, 보통 부모 스레드가 ③의 커밋을 먼저 끝내버리는 순서로 실행된다(레이스). 문제는 ③이 실패해서 롤백되는 경우다 — ①은 취소되지만, 이미 나가버린 ②의 푸시 알림은 취소할 수 없다. "판매 달성" 알림은 사용자에게 도착했는데 실제 DB에는 반영이 안 된 상태가 남는다. 이건 원자성(Atomicity)이 깨지는 문제이지, 아직 예외 캐치 이야기는 시작도 안 했다.

거기에 더해 — 예외는 스레드 경계를 못 넘는다

Ruby에서 각 스레드는 자기만의 call stack을 가진다. 자식 스레드에서 발생한 예외는 부모 스레드의 begin/rescue로 전파되지 않는다 — 자식 스레드 내부에서 직접 잡아야 한다. 안 잡으면 스레드가 조용히 죽고, 로그도 안 남아서 디버깅이 매우 어려워진다. 

begin
  Thread.new do
    raise "푸시 발송 실패"   # 부모의 rescue로 전파되지 않는다
  end
rescue => e
  # 여기 안 걸림
end

Thread.new do
  begin
    PushNotifier.notify(seller)
  rescue => e
    Rails.logger.error("push notify failed: #{e.message}")  # 반드시 스레드 내부에서 처리
  end
end
JavaScript의 setTimeout과 정확히 같은 모양이다. setTimeout 콜백은 별도의 매크로태스크로 이벤트 루프에 다시 들어가기 때문에, 그걸 감싼 try/catch는 콜백 안에서 던진 에러를 못 잡는다. 
try {
  setTimeout(() => { throw new Error('fail'); }, 0);
} catch (e) {
  // 여기 안 잡힘 — setTimeout 콜백은 분리된 실행 컨텍스트에서 돈다
}

setTimeout(() => {
  try {
    riskyPush();
  } catch (e) {
    console.error('push failed', e);   // 콜백 안에서 직접 잡아야 함
  }
}, 0);
"밖에서 감싼 try/catch가 안 먹힌다"는 감각을 이미 프론트에서 겪어봤다면, 스레드 경계도 똑같이 생각하면 된다 — 실행 컨텍스트가 분리되면 예외 전파 경로도 끊긴다.

그래서 실무에서는 어떻게 하나 — after_commit

레슨 38에서 배운 after_commit이 정확히 이 문제를 위한 관용구다. "부모 트랜잭션이 실제로 COMMIT된 뒤에만" 비동기 작업(Sidekiq job enqueue)을 실행하도록 걸면, 롤백될 데이터에 대해 알림이 먼저 나가는 레이스 자체가 성립하지 않는다. 

class Sale < ApplicationRecord
  after_commit :notify_seller, on: :create   # 커밋 이후에만 실행 — 롤백되면 아예 호출 안 됨

  def notify_seller
    PushNotificationJob.perform_async(id)    # Sidekiq 스레드 풀에 위임, 실패는 Sidekiq 재시도가 처리
  end
end

정리하면 역할 분담은 이렇다 — "언제 실행할지"는 after_commit이 보장하고(부모 트랜잭션 커밋 확정 후에만), "실패하면 어떻게 할지"는 실행하는 쪽(Sidekiq job)이 자기 안에서 직접 처리한다(재시도, 로깅). 원 요청의 트랜잭션이 그 실패를 대신 잡아줄 방법은 없다 — 이미 분리된 실행 컨텍스트이기 때문이다.
