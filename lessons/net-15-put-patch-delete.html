PUT, PATCH, DELETE

네트워크 · HTTP 메서드 강의 18 06:11
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 메서드 — PUT이 "전체 대체"인 이유, 그리고 PATCH가 따로 존재하는 이유를 이해한다

핵심 한 줄

PUT은 리소스를 완전히 덮어쓰고, PATCH는 필요한 필드만 바꾸며, DELETE는 리소스를 제거한다 — 세 메서드 모두 클라이언트가 URI를 직접 지정한다.

PUT — 리소스를 완전히 대체한다

PUT은 클라이언트가 리소스 URI를 알고 있을 때 쓴다. 해당 URI에 리소스가 있으면 대체(Replace)하고, 없으면 생성(Create)한다. 

# 회원 1번의 전체 정보를 PUT으로 대체
PUT /members/1 HTTP/1.1
Content-Type: application/json

{
  "name": "김철수",
  "email": "kim@example.com",
  "age": 30
}

PUT의 핵심 위험: 필드 누락 = 삭제

PUT은 요청 바디의 내용으로 리소스를 통째로 덮어쓴다. 일부 필드만 보내면 나머지는 사라진다. 

# 현재 서버의 회원 1번 상태
{ "name": "김철수", "email": "kim@example.com", "age": 30 }

# age만 수정하려고 PUT 전송 — email을 빼먹었다
PUT /members/1
{ "name": "김철수", "age": 31 }

# 결과: email이 사라진다!
{ "name": "김철수", "age": 31 }   ← email 필드 소멸

이것이 PUT과 PATCH의 결정적 차이다. PUT은 "부분 수정"이 아니라 "완전 교체"다.

POST와의 차이: 클라이언트가 URI를 지정

POST로 리소스를 만들 때는 서버가 ID를 결정한다 (POST /members). PUT은 클라이언트가 정확한 URI를 알고 지정한다 (PUT /members/1).

PATCH — 일부 필드만 변경한다

PATCH는 리소스를 부분적으로 수정한다. 변경하고 싶은 필드만 보내면 나머지는 그대로 유지된다. 

# age만 수정 — email은 건드리지 않는다
PATCH /members/1 HTTP/1.1
Content-Type: application/json

{ "age": 31 }

# 결과: age만 바뀌고 나머지는 유지
{ "name": "김철수", "email": "kim@example.com", "age": 31 }

실무에서 수정 API는 대부분 PATCH를 사용한다. 전체 객체를 다시 보내는 것보다 변경된 필드만 보내는 것이 효율적이고 안전하기 때문이다.

DELETE — 리소스를 삭제한다

DELETE는 가장 단순하다. URI로 리소스를 지정하고, 서버가 해당 리소스를 제거한다. 요청 바디는 거의 사용하지 않는다. 

# 회원 1번 삭제
DELETE /members/1 HTTP/1.1
Host: api.example.com

# 응답: 204 No Content (삭제 성공, 반환할 바디 없음)
HTTP/1.1 204 No Content

세 메서드 비교

메서드 역할 URI 지정 리소스 없을 때 멱등
PUT 전체 대체(Replace) 클라이언트 지정 새로 생성 멱등
PATCH 부분 변경(Partial Update) 클라이언트 지정 구현마다 다름 구현마다 다름
DELETE 삭제(Remove) 클라이언트 지정 404 또는 무시 멱등
React 폼에서 "수정" 버튼을 눌렀을 때 변경된 필드만 서버에 보내는 패턴 — 그게 PATCH다. PUT은 폼의 모든 필드를 항상 포함해서 보내는 것이다. 실무에서 Rails의 update 액션은 PATCH와 PUT을 둘 다 받지만, form_with가 기본으로 PATCH를 사용하는 이유가 바로 "부분 수정이 더 안전하기 때문"이다.

정리

  • PUT = 완전 대체. 필드 누락 = 해당 필드 삭제. 클라이언트가 URI 지정
  • PATCH = 부분 수정. 보낸 필드만 바뀜. 실무에서 수정 API의 기본
  • DELETE = 리소스 제거. 바디 없이 URI만으로 식별해 삭제
  • PUT과 DELETE는 멱등(Idempotent) — 여러 번 호출해도 결과 동일
