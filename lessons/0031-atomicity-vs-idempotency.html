원자성(Atomicity) vs 멱등성(Idempotency)

같은 "실패"인데 원인이 다르다

Performance Plus의 카프카 연동 이슈를 팀 시니어(juhoLee)와 이야기하다가 나온 구분인데, 정리하고 나니 앞으로 장애를 볼 때마다 계속 쓸 것 같은 프레임이라 레슨으로 남긴다. 실제 대화는 이랬다:

"가끔 연동 뻑나거나(user_id null로 연동), 일부만 되거나" → 원자성 문제
"반만 돌았는데 다시 연동요청 하거나" → 멱등성 문제

— 겉보기엔 둘 다 "연동이 이상하게 됐다"는 같은 증상이지만, 원인과 고치는 방법이 완전히 다르다.

원자성(Atomicity) — "하다 만 상태"가 생기는 문제

원자성은 하나의 작업 단위가 전부 성공하거나 전부 실패해야 한다는 원칙이다. user_id가 null인 채로 연동됐다는 건, 레코드 저장 로직 중간 어딘가에서 실패했는데 그 실패가 앞서 저장된 부분까지 되돌리지(rollback) 못하고 "일부만 반영된 상태"로 끝났다는 뜻이다. 이건 레슨 24에서 다룬 transaction do ... end + save!/update!로 고치는 문제다 — "쓰기 하나 안에서" 일부만 저장되는 걸 막는다.

멱등성(Idempotency) — "같은 요청을 또 받아도" 안전한가

멱등성은 다른 축이다. 같은 요청(또는 같은 카프카 메시지)이 한 번 더 도착해도, 결과가 처음 한 번 처리했을 때와 똑같아야 한다는 원칙이다. "반만 돌았는데 다시 연동요청" 상황을 떠올려보자 — 처리 중간에 타임아웃이 나서 클라이언트(또는 카프카 Consumer)가 재시도를 했는데, 만약 처리 로직이 매번 새 레코드를 생성하는 방식이라면 같은 멤버가 두 번 생성되는 중복 데이터가 만들어진다.

프론트에서 "저장" 버튼을 급하게 두 번 눌렀을 때를 생각하면 된다. POST /orders가 멱등하지 않으면 주문이 두 개 생긴다. 반면 PUT /orders/42(같은 ID로 덮어쓰기)는 몇 번을 다시 보내도 결과가 같다 — 이게 멱등성이다. 카프카에서는 이 "재전송"이 사용자의 실수가 아니라 Kafka Consumer의 재시도, 재배포로 인한 재시작, at-least-once 전달 보장 때문에 자연스럽게 일어난다는 게 다를 뿐이다.
Q1. "결제 승인 메시지를 처리하다가 DB write 하나가 실패했는데, 이미 처리한 다른 write는 롤백되지 않고 남아있다." 이건 어떤 문제인가?
Q2. "Consumer가 재시작되면서 이미 처리했던 메시지를 오프셋 문제로 한 번 더 받았는데, 그 결과로 똑같은 멤버 레코드가 두 개 생겼다." 이건 어떤 문제인가?

이미 멱등하게 짜여 있던 실제 코드

theplus-back의 OrganizationMemberConsume를 다시 보면, 이미 멱등성을 신경 쓴 흔적이 있다: 

# theplus-back/app/services/consume/organization_member_consume.rb (요약)
org_member = member.organization_members.kept.find_or_initialize_by(organization_id: organization.id)
# ... 값 설정
org_member.save!

find_or_initialize_by는 "있으면 찾아서 쓰고, 없으면 새로 만든다"는 upsert 방식이다. 같은 Appointment 이벤트가 재시도로 두 번 들어와도, 두 번째는 이미 존재하는 org_member를 찾아서 값만 다시 덮어쓸 뿐 중복 레코드를 만들지 않는다. 이게 멱등성을 만족시키는 가장 흔한 패턴이다 — "생성"이 아니라 "있으면 갱신, 없으면 생성"으로 짜는 것.

반대로 만약 어딘가에 Organization.create!(...)처럼 무조건 새로 만드는 코드가 있다면, 그 경로는 멱등하지 않다 — 재시도가 곧 중복 생성이 된다. 실제 코드를 볼 때 "이 메시지가 두 번 들어오면 어떻게 되지?"라고 스스로 물어보는 습관이 여기서 나온다.
