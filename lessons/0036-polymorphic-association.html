폴리모픽 연관관계 — History 하나로 여러 모델 가리키기

🎯 코드를 보고 "이 요청이 어떤 경로로 흘러가는지" 머릿속에 그릴 수 있다 — MISSION.md

출발점: 알림은 다 다른데, 테이블은 하나

Performance Plus에는 Review 리마인드, Objective 변경, Task 배정 등 "알림을 보내야 하는 이벤트"가 수십 종류 있다. 그런데 이 모든 이벤트를 기록하는 테이블은 histories 딱 하나다. Review 관련 알림 이력도, Objective 관련 알림 이력도 전부 같은 테이블에 쌓인다.

레슨 8에서 배운 belongs_to는 항상 고정된 하나의 테이블을 가리켰다:

class Comment < ApplicationRecord
  belongs_to :task  # comments.task_id → tasks.id 로 고정
end

그런데 History는 어떤 레코드는 Review를, 어떤 레코드는 Objective를, 어떤 레코드는 Task를 가리켜야 한다. review_id, objective_id, task_id를 전부 컬럼으로 만들면 한 레코드에 하나만 값이 있고 나머지는 항상 NULL인 낭비 구조가 된다. 이럴 때 쓰는 것이 폴리모픽 연관관계(polymorphic association)다.

실제로 본 코드

app/models/history.rb에서 이런 선언을 봤다:

# app/models/history.rb
class History < ApplicationRecord
  belongs_to :target, polymorphic: true
  #                    ^^^^^^^^^^^^^^^^^ 고정된 테이블이 없다는 선언

  ABLE_TYPE = %w[Review Objective Task Feedback ...] # target_type 화이트리스트
end

polymorphic: true가 붙으면 Rails는 FK 컬럼 하나(target_id) 대신 두 개의 컬럼을 쓴다:

  • target_id — 가리키는 레코드의 PK (예: review.id)
  • target_type — 가리키는 모델 클래스 이름을 문자열로 저장 (예: "Review")

즉 histories 테이블의 한 로우는 이렇게 생겼다:

| id | target_type | target_id | event_type      |
|----|-------------|-----------|------------------|
| 1  | "Review"    | 42        | "ReviewReminded" |
| 2  | "Objective"  | 7         | "ObjectiveCreated" |
| 3  | "Task"      | 15        | "TaskAssigned"   |

같은 테이블, 같은 컬럼 두 개로 Review 42번도, Objective 7번도, Task 15번도 전부 가리킬 수 있다. history.target을 호출하면 Rails가 target_type 값을 보고 Review.find(42) 같은 걸 자동으로 실행해준다.

TypeScript의 discriminated union과 거의 같은 아이디어다. 
// TS: type 필드로 어떤 모양인지 구분
type Target =
  | { type: 'Review'; id: number }
  | { type: 'Objective'; id: number }
  | { type: 'Task'; id: number };

function loadTarget(t: Target) {
  switch (t.type) {          // target_type과 동일한 역할
    case 'Review': return fetchReview(t.id);
    case 'Objective': return fetchObjective(t.id);
  }
}
target_type이 TS의 type 판별 필드, target_id가 각 분기의 id와 같은 역할을 한다. 다만 Rails는 이 switch 분기를 직접 안 써도 target_type 문자열 그대로 constantize해서 클래스를 찾아준다는 점이 다르다.

왜 이렇게까지 하나 — 트레이드오프

방식 장점 단점
모델별 컬럼
review_id, objective_id, ...		 DB 레벨 FK 제약(참조 무결성) 가능 모델 늘어날 때마다 마이그레이션, 항상 대부분 컬럼이 NULL
폴리모픽
target_type + target_id		 컬럼 2개로 무한히 확장 가능 DB가 target_type이 실제 존재하는 클래스인지 보장 못 함 → 애플리케이션 레벨 검증 필요

그 "애플리케이션 레벨 검증"이 바로 ABLE_TYPE 상수다. target_type에 아무 문자열이나 들어갈 수 없도록 화이트리스트로 막아둔 것 — DB가 못 해주는 무결성 검사를 코드가 대신 하고 있는 것이다.
