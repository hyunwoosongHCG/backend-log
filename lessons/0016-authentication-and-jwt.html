🎯 레슨 12는 인가(Authorization)를 완성했다. 이번엔 그 앞 단계인 인증(Authentication)을 채운다 — MISSION.md

인증과 인가 — 먼저 정확히 구분한다

두 개념이 자주 혼용되지만 완전히 다른 질문에 답한다:

개념 질문 실패하면 담당
인증 (Authentication) "당신이 누구인가?" 401 Unauthorized authenticate_user!
인가 (Authorization) "당신이 이걸 할 수 있는가?" 403 Forbidden Pundit (레슨 12)

순서도 중요하다: 인증 먼저, 인가 나중. "누구인지 모르면 권한 확인 자체가 의미 없다."

사무실 출입을 예로 들면:

인증 = 사원증으로 신원 확인 → "이 사람이 우리 회사 직원인가?" (401: 사원증 없음)
인가 = 층별 권한 확인 → "이 직원이 서버실에 들어갈 수 있는가?" (403: 권한 없음)

토큰 기반 인증 흐름

세션/쿠키 방식 대신 이 프로젝트는 토큰을 쓴다. 흐름은 두 단계다.

① 로그인: 클라이언트가 이메일+비밀번호로 POST /auth/sign_in
↓
② 서버가 신원 확인 후 JWT 토큰을 응답으로 발급
↓ 클라이언트가 토큰을 로컬에 저장
③ 이후 요청마다 HTTP 헤더에 토큰을 실어 보냄
↓
④ 서버의 authenticate_user!가 토큰을 검증 → current_user 설정
↓ 토큰 유효하면 계속, 유효하지 않으면 401
⑤ 컨트롤러 로직 실행

헤더에는 이렇게 실린다:

Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjozfQ.abc123

Bearer는 "이 토큰을 가진 사람에게 권한을 줘라"는 의미의 접두사 관용어다. 프론트에서 API 요청 시 헤더에 붙이는 그것이다.

JWT 구조 — 세 파트

JWT(JSON Web Token)는 점(.)으로 구분된 세 부분으로 이루어진다. 위 토큰을 분해하면: 

eyJhbGciOiJIUzI1NiJ9          ← Header  (알고리즘 정보)
.
eyJ1c2VyX2lkIjozfQ            ← Payload (실제 데이터, Base64 인코딩)
.
abc123                         ← Signature (위변조 확인용)

Payload를 Base64 디코딩하면 JSON이 나온다: 

{ "user_id": 3, "exp": 1751234567 }

서버는 이 user_id로 DB에서 사용자를 찾아 current_user에 담는다. Signature 덕분에 클라이언트가 Payload를 임의로 수정하면 서버가 감지한다.

Q1. JWT에서 실제 사용자 데이터(user_id 등)가 담기는 파트는?

authenticate_user! 내부 — 이 프로젝트 기준

Grape에서 before 훅으로 모든 요청 앞에 실행된다: 

# app/controllers/grape_base.rb
before do
  authenticate_user!
end

def authenticate_user!
  token = headers['Authorization']&.split('Bearer ')&.last
  payload = JwtService.decode(token)          # 토큰 검증 + 디코딩
  @current_user = User.find(payload['user_id'])
rescue JWT::DecodeError, ActiveRecord::RecordNotFound
  error!('Unauthorized', 401)                 # 유효하지 않으면 즉시 401
end

before do ... end는 레슨 15에서 배운 블록이다. 모든 엔드포인트가 실행되기 전에 이 블록이 먼저 실행된다.

current_user vs pundit_user

def current_user
  @current_user        # JWT에서 꺼낸 User 객체
end

def pundit_user        # 레슨 12에서 봤던 것
  Member.find_by(workspace: workspace, user: current_user) || current_user
end

current_user는 인증 결과 (누구인가), pundit_user는 인가 기준 (이 워크스페이스에서 어떤 역할인가).

Q2. 토큰 없이 요청을 보내면 어떤 HTTP 상태코드가 반환되나?

전체 요청 흐름 — 인증 + 인가 합산

① 요청 도착
↓ before 블록
② authenticate_user! — JWT 검증 → 실패 시 401
↓ 성공: current_user 설정됨
③ 라우트 핸들러 진입
↓
④ authorize record, :action? — Pundit 권한 확인 → 실패 시 403
↓ 성공: 실제 로직 실행
⑤ 정상 응답 반환

핵심 요약

개념 한 줄 설명
JWT Header.Payload.Signature — 자기완결적 토큰
Bearer Token Authorization 헤더로 JWT를 전송하는 방식
authenticate_user! before 블록에서 JWT 검증 → current_user 설정
401 인증 실패 (누군지 모름)
403 인가 실패 (누군진 알지만 권한 없음)
