ActiveRecord::Base 가 주는 것

ActiveRecord::Base를 상속하는 순간 아무 코드 없이도 다음이 전부 생긴다:

능력 예시
테이블 자동 연결 OneOnOneSchedule → one_on_one_schedules
컬럼 getter/setter 자동 생성 object.user_id, object.place
쿼리 메서드 .where(), .find(), .includes()
저장/삭제 .save!, .destroy
연관관계 선언 지원 belongs_to, has_many 등

테이블 자동 매핑 규칙

Rails는 클래스 이름을 snake_case + 복수형으로 바꿔 테이블 이름으로 쓴다. 별도 설정 없이 이 규칙만으로 DB와 연결된다. 

모델 클래스명              → 테이블명
────────────────────────────────────────
User                      → users
OneOnOne                  → one_on_ones
OneOnOneSchedule          → one_on_one_schedules
AppraisalAppraisee        → appraisal_appraisees
WorkspaceInvitation       → workspace_invitations

변환 단계:

OneOnOneSchedule
  1단계: CamelCase → snake_case  →  one_on_one_schedule
  2단계: 단수 → 복수              →  one_on_one_schedules
  3단계: 이 이름으로 DB 테이블 탐색

영어 불규칙 복수형도 Rails가 안다: 

Person  → people
Mouse   → mice
Tooth   → teeth
JS에서 파일 이름으로 컴포넌트를 자동 라우팅하는 것과 비슷한 원리다. Next.js에서 pages/about.js 가 /about 경로가 되듯, Rails에서 OneOnOneSchedule 모델이 one_on_one_schedules 테이블에 연결된다.

컬럼 getter는 왜 선언 안 해도 되나

schema.rb에 컬럼이 있으면 Rails가 앱 시작 시 DB 메타데이터를 읽어서 각 컬럼에 대한 getter/setter를 자동으로 생성한다. 

# schema.rb에 이 컬럼들이 있으면
create_table "one_on_one_schedules" do |t|
  t.integer "user_id"
  t.string  "place"
  t.datetime "time"
end

# 아무 선언 없이도 모델에서 바로 쓸 수 있다
schedule = OneOnOneSchedule.find(1)
schedule.user_id   # → 5
schedule.place     # → "회의실 A"
schedule.time      # → 2026-06-30 14:00:00

반면 belongs_to :user처럼 연관관계는 자동으로 생기지 않는다. 컬럼(user_id)은 자동이지만, 그 숫자로 users 테이블을 조회해 객체를 돌려주는 .user 메서드는 명시적으로 선언해야 Rails가 만들어준다. 

schedule.user_id   # 자동 — 컬럼이 있으면 됨
schedule.user      # belongs_to :user 없으면 → NoMethodError

ApplicationRecord — ppback 공통 레이어

Rails 5부터 도입된 패턴이다. ActiveRecord::Base를 직접 상속하는 대신 중간에 ApplicationRecord를 끼워, 프로젝트 전체 모델에 공통 설정을 한 곳에서 관리한다. 

# app/models/application_record.rb
class ApplicationRecord < ActiveRecord::Base
  self.abstract_class = true   # ← 이 클래스 자체는 테이블이 없다는 선언

  # production/staging 환경에서는 읽기 전용 replica DB 사용
  if Rails.env.in?(%w[production staging dump demo])
    connects_to database: { writing: :primary, reading: :replica }
  end
end

self.abstract_class = true 가 중요하다. 이걸 선언하지 않으면 Rails가 application_records라는 테이블을 찾으려고 한다. "나는 테이블 없는 추상 기반 클래스야"라는 표시다.

ppback의 ApplicationRecord는 여기에 추가로 운영 환경에서 읽기 쿼리는 replica DB로 보내는 설정을 한다. 쓰기(INSERT/UPDATE/DELETE)는 primary, 읽기(SELECT)는 replica — 부하 분산이다.

전체 구조 요약

DB (MySQL)
  one_on_one_schedules 테이블
    id, user_id, one_on_one_id, place, time, ...

        ↕  ActiveRecord::Base가 자동 연결

app/models/one_on_one_schedule.rb
  class OneOnOneSchedule < ApplicationRecord
    belongs_to :user        ← user_id FK로 User 객체 로드 메서드 생성
    belongs_to :one_on_one  ← one_on_one_id FK로 OneOnOne 객체 로드
    belongs_to :workspace

    def send_calendar_event ← 비즈니스 로직
      ...
    end
  end

        ↕  컨트롤러가 쿼리해서 인스턴스 생성

Entity (렌더링)
  object = OneOnOneSchedule 인스턴스
  object.user_id  → 컬럼 직접 읽기 (쿼리 없음)
  object.user     → belongs_to로 생성된 메서드 → DB SELECT
