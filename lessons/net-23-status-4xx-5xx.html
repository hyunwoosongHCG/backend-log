4xx 클라이언트 오류, 5xx 서버 오류

네트워크 · HTTP 상태코드 강의 26 09:50
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 상태코드 — 잘못한 쪽이 클라이언트(4xx)인지 서버(5xx)인지 정확히 구분하고, Rails의 rescue_from과 axios 인터셉터로 연결하는 법을 배운다

핵심 한 줄

4xx는 클라이언트가 고쳐야 해결되는 오류(잘못된 요청·인증 없음·권한 없음·없는 리소스), 5xx는 서버가 고쳐야 해결되는 오류 — 같은 요청을 다시 보내면 4xx는 또 실패하지만 5xx는 서버 수리 후 성공할 수 있다.

4xx — 클라이언트 오류 (Client Error)

클라이언트가 잘못된 형식·데이터·인증을 보낸 경우다. 서버는 문제 없이 작동 중이고, 클라이언트가 요청을 고쳐야 해결된다. 재시도해도 같은 실패가 반복된다.

코드 이름 의미 실무 예시
400 Bad Request 요청 형식·데이터가 잘못됨 필수 파라미터 누락, 타입 오류
401 Unauthorized 인증(Authentication)이 안 됨 토큰 없음·만료, 로그인 필요
403 Forbidden 인증은 됐으나 권한(Authorization)이 없음 일반 회원이 관리자 페이지 접근
404 Not Found 리소스를 찾을 수 없음 없는 ID 조회, 잘못된 URL
422 Unprocessable Entity 형식은 맞지만 유효성 검증 실패 이메일 형식 오류, 중복된 값

401 vs 403 — 가장 헷갈리는 구분

이름이 "Unauthorized(권한 없음)"이지만 401은 사실 인증(Authentication) 문제다. "누구인지 모르겠다 → 로그인해라"는 의미다. 403은 "누구인지는 알지만, 이 리소스에는 접근 권한이 없다"는 인가(Authorization) 문제다.

401 Unauthorized 403 Forbidden
문제 인증(Authentication) 미완료 인가(Authorization) 거부
상황 로그인 안 됨 · 토큰 만료 로그인은 됐으나 접근 권한 없음
해결 로그인 페이지로 이동 "권한 없음" 메시지 표시
React Router의 PrivateRoute에서 로그인 여부를 확인해 /login으로 보내는 것이 401 처리, 로그인은 됐지만 admin 여부를 확인해 "접근 거부" 페이지를 보여주는 것이 403 처리다. Pundit이 권한 거부를 던지면 403, JWT가 만료되면 401.

400 vs 422 — 데이터 오류의 세분화

400은 요청 자체가 서버가 이해할 수 없는 형식일 때, 422는 형식은 올바른 JSON/XML이지만 비즈니스 유효성 검사(validation)를 통과 못 했을 때다. Rails에서는 ActiveRecord 유효성 검사 실패를 422로 응답하는 패턴이 일반적이다. 

# Rails: 422 Unprocessable Entity (유효성 검사 실패)
def create
  user = User.new(user_params)
  if user.save
    render json: user, status: :created  # 201
  else
    render json: { errors: user.errors }, status: :unprocessable_entity  # 422
  end
end

5xx — 서버 오류 (Server Error)

서버 내부에 문제가 생겨 정상 처리를 못 한 경우다. 클라이언트 요청이 올바르더라도 발생한다. 서버를 수리하면 같은 요청이 성공할 수 있다.

코드 이름 의미 실무 예시
500 Internal Server Error 서버 내부의 예상치 못한 오류 처리되지 않은 예외, nil 역참조
503 Service Unavailable 서버가 일시적으로 요청 처리 불가 점검 중, 트래픽 과부하

500 — 잘못 쓰기 쉬운 코드

500은 정말 서버 로직 오류일 때만 써야 한다. 비즈니스 규칙상 "처리 불가" 상황(잔액 부족, 중복 신청 등)에 500을 쓰면 안 된다. 그런 경우는 4xx(주로 422 또는 409 Conflict)가 맞다. 500을 모든 오류에 남발하면 모니터링에서 진짜 서버 장애를 발견하기 어려워진다. 

# 잘못된 예: 비즈니스 오류에 500 사용
def transfer
  if balance < amount
    render json: { error: '잔액 부족' }, status: 500  # ❌ 잘못됨
  end
end

# 올바른 예: 비즈니스 오류는 4xx
def transfer
  if balance < amount
    render json: { error: '잔액 부족' }, status: :unprocessable_entity  # 422 ✓
  end
end

Rails rescue_from — 상태코드 자동 매핑

Rails와 Grape에서는 rescue_from으로 예외 클래스를 상태코드와 연결한다. 이렇게 하면 컨트롤러마다 begin/rescue를 쓰지 않아도 된다. 

# Rails: ApplicationController에 전역 예외 처리 등록
class ApplicationController < ActionController::API
  rescue_from ActiveRecord::RecordNotFound do |exception|
    render json: { error: exception.message }, status: :not_found  # 404
  end

  rescue_from Pundit::NotAuthorizedError do |exception|
    render json: { error: '접근 권한이 없습니다' }, status: :forbidden  # 403
  end

  rescue_from ActiveRecord::RecordInvalid do |exception|
    render json: { errors: exception.record.errors }, status: :unprocessable_entity  # 422
  end
end

프론트 axios 인터셉터 — 상태코드 분기

// 프론트: axios 인터셉터로 상태코드별 처리
axios.interceptors.response.use(
  (response) => response,  // 2xx → 통과
  (error) => {
    const status = error.response?.status;

    if (status === 401) {
      // 인증 없음 → 로그인 페이지로
      store.dispatch('logout');
      router.push('/login');
    } else if (status === 403) {
      // 권한 없음 → 토스트 메시지
      toast.error('접근 권한이 없습니다.');
    } else if (status === 404) {
      toast.error('요청한 리소스를 찾을 수 없습니다.');
    } else if (status === 422) {
      // 유효성 검사 실패 → 에러 메시지 표시
      const errors = error.response.data.errors;
      displayValidationErrors(errors);
    } else if (status >= 500) {
      toast.error('서버에 문제가 발생했습니다. 잠시 후 다시 시도해주세요.');
    }

    return Promise.reject(error);
  }
);

정리

  • 400: 요청 형식·데이터 오류 (클라이언트가 고쳐야)
  • 401: 인증(Authentication) 없음 → 로그인 유도
  • 403: 인가(Authorization) 거부 → 권한 없음 메시지
  • 404: 리소스 없음 — Rails RecordNotFound와 자연스럽게 매핑
  • 422: 유효성 검사 실패 — ActiveRecord RecordInvalid에 적합
  • 500: 진짜 서버 로직 오류만 — 비즈니스 오류에 남발 금지
  • 503: 서버 일시 불가 — 점검·과부하
  • Rails rescue_from + 프론트 axios 인터셉터로 계층별 처리
