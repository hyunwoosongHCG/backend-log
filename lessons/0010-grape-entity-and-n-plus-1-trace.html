백엔드에서 "렌더링"이란?

프론트엔드에서 렌더링은 화면에 그리는 것이다.
백엔드에서 렌더링은 Ruby 객체를 JSON으로 변환해서 HTTP 응답으로 내보내는 것이다. 화면과 전혀 관계없다. 

프론트 요청 (GET /api/v2/workspaces/.../one_on_ones/histories)
    ↓
컨트롤러 — DB에서 데이터 조회 → Ruby 객체
    ↓
"렌더링" — Ruby 객체를 JSON 문자열로 변환
    ↓
HTTP 응답으로 전송 → 프론트가 받아서 화면에 그림

Entity가 하는 일이 바로 이 변환이다:

# Ruby 객체
#<OneOnOneSchedule id=1, user_id=5, one_on_one_id=3 ...>

# Entity가 변환한 JSON
{
  "id": 1,
  "creator_id": 5,       ← expose :user_id, as: :creator_id
  "assignee_id": 3,      ← delegate :assignee_id, to: :one_on_one
  "creator_info": { ... } ← expose :creator_info, using: Entities::UserEntity
}
React에서 JSON.stringify(state)로 상태를 문자열로 바꾸는 것과 비슷하다. Entity는 모델 데이터를 "어떤 키로, 어떤 값으로, 어떤 형식으로" JSON에 담을지 정의하는 변환 설명서다.

오늘 배운 것 한눈에 정리

개념 한 줄 요약 JS 대응
object Entity 안에서 렌더링 대상 모델 인스턴스 props
delegate :x, to: :object object.x 를 x로 단축 구조 분해 const { x } = props
Entities:: Ruby 모듈 네임스페이스 — 파일 경로와 1:1 대응 폴더 import 경로
< Grape::Entity expose·present·object 기능을 상속받음 컴포넌트 extend

expose — Ruby DSL (어노테이션처럼 생긴 클래스 메서드)

expose는 어노테이션이 아니다. 클래스 본문에서 호출하는 Ruby 메서드다. Grape::Entity가 제공하는 DSL(Domain Specific Language)로, 클래스 정의 시점에 실행된다. 

# 이렇게 생겼지만
expose :user_id, as: :creator_id, documentation: { type: 'Integer', example: 1 }

# 사실 이걸 호출하는 것과 같다
self.expose(:user_id, as: :creator_id, documentation: { type: 'Integer', example: 1 })
React에서 static propTypes = { ... } 또는 TypeScript 데코레이터 @Column()처럼 "클래스를 정의하는 시점에 메타데이터를 등록하는 구문"이다. 문법이 어노테이션처럼 보이지만 실제로는 메서드 호출이다.

expose 옵션들

# 기본 — 모델의 user_id 컬럼을 그대로 노출
expose :user_id

# as: — JSON 키 이름을 바꿔서 노출
expose :user_id, as: :creator_id
# → { "creator_id": 1 }  (user_id가 아니라 creator_id로 나감)

# using: — 다른 Entity로 중첩 렌더링
expose :creator_info, using: Entities::UserEntity
# → creator_info 키 아래 UserEntity 형식으로 렌더링

# documentation: — Swagger 문서 자동 생성용 메타데이터 (런타임 동작과 무관)
expose :id, documentation: { type: 'Integer', example: 1 }

# if: — 조건부 노출
expose :place, if: ->(_instance) { next_schedule.place != place }

expose와 def의 관계

expose :creator_info, using: Entities::UserEntity
# ↑ "creator_info라는 키로 UserEntity 형식을 노출하겠다" 선언
# Rails가 렌더링할 때 아래 메서드를 자동으로 호출함

def creator_info
  object.user  # 실제 데이터를 어디서 가져올지 정의
end

expose :x만 쓰면 Rails가 object.x를 자동으로 호출한다. 커스텀 로직이 필요할 때만 def x를 추가한다.

컨트롤러 → Entity 흐름 추적하기

N+1이 어디서 생기는지 파악하려면 이 흐름을 따라가야 한다:

컨트롤러
  └─ present :data, one_on_one_schedules, with: Entity
       └─ 스케줄 100개 × Entity 렌더링
            └─ expose :creator_info  → def creator_info → object.user  ← 쿼리 100번?
            └─ expose :assignee_info → def assignee_info → one_on_one.assignee ← 쿼리 100번?

실제 컨트롤러 코드 (v2/admins/one_on_ones.rb)

one_on_one_schedules =
  OneOnOneSchedule
    .where(one_on_one_id: one_on_ones.select(:id))
    .page(params[:page]).per(params[:size])
    .includes(
      template: :elements,           # schedule.template, template.elements
      user: :members,                # schedule.user (= creator), user.members
      one_on_one: [
        assignee: :members,          # one_on_one.assignee, assignee.members
        manager:  :members,          # one_on_one.manager, manager.members
      ],
    )

present :data,
        one_on_one_schedules,
        workspace:,
        with: Entities::OneOnOneSchedules::HistoryScheduleEntity

includes가 없었다면 Entity가 100개 스케줄을 렌더링하면서 object.user, one_on_one.assignee, one_on_one.manager를 각각 100번씩 조회했을 것이다.

이번 PR이 해결한 것 — id → ids

includes로 N+1을 막았어도, 프론트엔드가 실제로 필요한 것이 user 객체 전체가 아니라 id뿐이라면 User 테이블을 join하고 members까지 preload하는 것 자체가 낭비다. 

# 변경 전 — User 전체를 로드해서 id를 뽑음
def creator_id
  object.user_id  # 사실 이건 이미 schedule 테이블에 있는 컬럼
end

# 변경 후 — schedule 테이블의 컬럼을 바로 노출 (User 조회 불필요)
expose :user_id, as: :creator_id
# 변경 전 — assignee_id를 얻으려고 one_on_one 전체를 로드
delegate :assignee_id, to: :one_on_one  # one_on_one 쿼리 필요

# 변경 후 — one_on_one 테이블의 assignee_id 컬럼을 직접 노출
delegate :assignee_id, to: :one_on_one
# (one_on_one은 이미 includes로 로드됨 — 추가 쿼리 없음)

코드를 타고 들어가는 법

Entity 코드를 볼 때 "이게 쿼리를 날리나?"를 확인하는 체크리스트:

코드 패턴 쿼리 발생 여부 해결
object.컬럼명 없음 — 이미 로드된 값 -
object.연관관계 includes 없으면 N+1 컨트롤러에서 includes 추가
연관.연관.컬럼 체인 깊이만큼 위험 includes(a: :b) 중첩
@memoize ||= ... 첫 호출만 쿼리, 이후 캐시 같은 Entity 안에서 반복 접근 시 사용
