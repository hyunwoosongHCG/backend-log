핵심 한 줄

HTTP 메시지는 start-line / header / 빈 줄 / body의 4-파트 구조다. 프론트에서 매일 보던 DevTools Network 탭의 Headers·Payload·Response 탭이 바로 이 구조를 눈에 보이게 펼쳐놓은 것이다.

HTTP 메시지의 전체 구조

HTTP 메시지에는 두 종류가 있다: 요청(Request)과 응답(Response).

// 공통 구조
start-line      ← 요청: 메서드·경로·버전  /  응답: 버전·상태코드·설명
header-field    ← 부가 정보 (여러 줄 가능)
                ← 빈 줄 (CRLF) — 헤더와 바디를 구분하는 경계선
message-body    ← 실제 데이터 (없을 수도 있음)

1. Start-Line (시작 줄)

요청 메시지의 시작 줄 — Request-Line

형식: [메서드] [요청 대상] [HTTP 버전] CRLF

GET /search?q=hello&hl=ko HTTP/1.1
POST /api/users HTTP/1.1
DELETE /api/posts/42 HTTP/1.1
  • 메서드(Method): GET·POST·PUT·PATCH·DELETE 등 서버에 할 행동
  • 요청 대상(Request-Target): 절대 경로 + 쿼리스트링. 예: /search?q=hello&hl=ko
  • HTTP 버전(HTTP-Version): HTTP/1.1, HTTP/2

응답 메시지의 시작 줄 — Status-Line

형식: [HTTP 버전] [상태 코드] [이유 문구] CRLF

HTTP/1.1 200 OK
HTTP/1.1 404 Not Found
HTTP/1.1 500 Internal Server Error
  • HTTP 버전: 서버가 지원하는 버전
  • 상태 코드(Status-Code): 3자리 숫자. 요청 성공/실패/리다이렉트 등
  • 이유 문구(Reason-Phrase): 상태 코드를 사람이 읽기 쉽게 설명한 문자열
DevTools Network 탭에서 요청 하나를 클릭하면 보이는 것들이 바로 이 구조다. "General" 섹션의 Request URL·Request Method·Status Code가 start-line에 해당하고, "Request Headers" / "Response Headers"가 header에 해당하고, "Request Payload" / "Response" 탭이 body에 해당한다. 이미 매일 보고 있었다 — 이제 그 이름을 알게 된 것이다.

2. HTTP 헤더 (Header)

형식: [필드 이름]: [필드 값]

// 요청 헤더 예시
Host: www.google.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Accept: application/json, text/plain

// 응답 헤더 예시
Content-Type: application/json; charset=utf-8
Content-Length: 3423
Cache-Control: max-age=3600

헤더에는 전송에 필요한 모든 부가 정보가 담긴다. 바디 내용의 종류(Content-Type), 크기(Content-Length), 압축 방식(Content-Encoding), 인증 정보(Authorization), 캐시 정책(Cache-Control), 브라우저 정보(User-Agent) 등.

HTTP 표준이 정의한 수십 가지 헤더 외에도, 임의의 헤더를 추가할 수 있다. 

// 커스텀 헤더도 가능
X-Custom-Header: my-value
X-Request-Id: abc-123

3. 빈 줄 (Empty Line / CRLF)

헤더와 바디 사이에 반드시 빈 줄(CRLF, Carriage Return + Line Feed)이 있어야 한다. 이 빈 줄이 "헤더 끝, 바디 시작"을 알리는 경계선이다. 빈 줄이 없으면 서버(또는 클라이언트)가 어디부터 바디인지 알 수 없다.

4. HTTP Message Body (메시지 바디)

실제 전송하려는 데이터가 들어간다. byte로 표현할 수 있는 모든 것이 가능하다: HTML·JSON·이미지·영상·파일. 

// JSON 응답 바디 예시
{
  "id": 1,
  "name": "홍길동",
  "email": "hong@example.com"
}

// HTML 응답 바디 예시
<!DOCTYPE html>
<html>
  <head><title>홈</title></head>
  <body>안녕하세요</body>
</html>

GET 요청처럼 데이터를 보내지 않는 경우 바디가 비어 있어도 된다. 반면 POST·PUT·PATCH 요청은 보통 바디에 데이터를 담는다.

요청과 응답 메시지 전체 예시

// HTTP 요청 메시지
GET /search?q=hello HTTP/1.1         ← start-line (Request-Line)
Host: www.google.com                 ← header
Accept: text/html                    ← header
                                     ← 빈 줄 (CRLF)
                                     ← body 없음 (GET은 보통 없음)

// HTTP 응답 메시지
HTTP/1.1 200 OK                      ← start-line (Status-Line)
Content-Type: text/html; charset=utf-8  ← header
Content-Length: 3423                 ← header
                                     ← 빈 줄 (CRLF)
<!DOCTYPE html>...                   ← body (실제 HTML 내용)

정리

  • HTTP 메시지 구조: start-line → header → 빈 줄 → body
  • 요청 start-line: 메서드 경로 HTTP버전 (예: GET /users HTTP/1.1)
  • 응답 start-line: HTTP버전 상태코드 이유문구 (예: HTTP/1.1 200 OK)
  • 헤더: 전송에 필요한 부가 정보 전부 (Content-Type, Authorization, Cache-Control 등)
  • 빈 줄: 헤더와 바디의 경계선, 반드시 필요
  • 바디: byte로 표현 가능한 모든 실제 데이터
