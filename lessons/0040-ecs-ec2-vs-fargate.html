출발점 — "ECS는 뭐고, EC2랑 Fargate는 뭐가 다른데?"

aws-infra-architecture 레포의 performance-plus 스택을 리뷰하다가 브랜치 이름부터 막혔다 — feat/ecs-migrate-ec2-to-fargate. ECS가 뭔지, EC2가 뭔지, Fargate가 뭔지 하나도 몰랐다. 이 레슨은 그 세 단어를 코드에 나온 순서 그대로 정리한다.

ECS(Elastic Container Service) — 컨테이너 오케스트레이터

ECS는 Docker 컨테이너를 "몇 개 띄울지, 죽으면 다시 살릴지, 로드밸런서에 어떻게 연결할지"를 관리해주는 AWS 서비스다. 계층 구조는 이렇다.

  • 클러스터(Cluster) — 컨테이너들이 모여 사는 논리적 그룹. 실제 배포 대상의 최상위 단위.
  • 서비스(Service) — 클러스터 안에서 "이 태스크를 N개 계속 띄워놔라"를 관리하는 컨트롤러. 죽으면 재시작, ALB 연결, 오토스케일링이 여기 붙는다.
  • 태스크 정의(Task Definition) — "어떤 이미지를, CPU/메모리 얼마로, 어떤 커맨드로 띄울지" 적은 설계도. 컨테이너 자체가 아니라 blueprint.

실제 코드로 보면 이 셋이 그대로 대응된다. 

// Task Definition — 설계도 (people-sidekiq-task.ts)
this.sidekiqServerTask = new ecs.TaskDefinition(this, 'PESidekiqTaskDef', {
  family: `${env}-PeopleSidekiq-Task`,
  compatibility: ecs.Compatibility.FARGATE,
  cpu: '2048',
  memoryMiB: '4096',
})

// Service — 그 설계도로 N개를 계속 띄워놓는 관리자 (ecs-service.ts)
this.sidekiqService = new ecs.FargateService(this, 'SidekiqService', {
  cluster: this.cluster,
  taskDefinition: ecsInfo.sidekiqTask,   // ← 위 Task Definition
  ...
})

EC2 launch type vs Fargate launch type

ECS가 "무엇을 몇 개 띄울지"를 정한다면, launch type은 "그걸 어디서(어떤 컴퓨팅 위에서) 돌릴지"를 정한다. 둘 다 같은 ECS 서비스/태스크 개념 위에서 동작하지만, 밑단이 완전히 다르다.

EC2 launch type Fargate launch type
컨테이너가 도는 곳 내가 띄운 EC2 인스턴스 AWS가 관리하는 서버리스 인프라
인스턴스 타입/AMI 선택 직접 (LaunchTemplate) 필요 없음
패치·보안 업데이트 직접 AWS 책임
용량 스케일링 ASG + CapacityProvider 직접 구성 Fargate가 태스크 단위로 자동 확보
배치(Placement) 전략 여러 인스턴스 중 어디에 놓을지 지정 가능 지원 안 함 — AWS가 서브넷 간 자동 분산
SSH 접속 키페어로 인스턴스 SSH 불가 — ECS Exec(SSM)로 컨테이너 직접 접속

이번 마이그레이션(Phase 1: #273)에서 Ec2Service → FargateService로 바꾸고, Phase 2(#274)에서 EC2 launch type을 위해 존재하던 LaunchTemplate/ASG/ AsgCapacityProvider 코드(ecs.ts)를 통째로 삭제했다. Fargate는 그런 걸 아예 요구하지 않기 때문이다.

Fargate로 바꿔도 "여전히 우리가 정하는 것"

여기서 착각하기 쉬운 지점 — Fargate가 "서버"를 대신 관리해준다고 해서 모든 설정이 사라지는 건 아니다. 정확히는 "호스트 레벨" 결정은 Fargate가 가져가고, "태스크 레벨" 스펙은 여전히 우리가 지정한다.

  • CPU/메모리 크기 — cpu: '2048', memoryMiB: '4096' (Task Definition)
  • CPU 아키텍처(ARM64 vs x86) — runtimePlatform
  • 서비스가 몇 개 떠 있을지, 오토스케일링 임계치 — ServiceMinCapacity/cpuScaling 등 (Service)

ARM64/Graviton은 EC2 → Fargate 전환과 무관하게 유지된다

이 레포는 이미 ARM64(Graviton) 아키텍처로 이미지를 빌드해서 쓰고 있었다. Phase 2에서 삭제된 건 "M7G(Graviton3) EC2 인스턴스 타입을 고르는 LaunchTemplate"뿐이고, Task Definition의 아키텍처 지정은 그대로 남았다. 

// people-sidekiq-task.ts — Phase 2 이후에도 그대로 남아있는 부분
runtimePlatform: {
  cpuArchitecture: ecs.CpuArchitecture.ARM64,   // 이미지가 ARM64 빌드이므로 명시 필수
  operatingSystemFamily: ecs.OperatingSystemFamily.LINUX,
},

Fargate도 ARM64 runtime을 지원하고, x86보다 vCPU/메모리당 더 저렴하다 — EC2 시절 Graviton을 쓰던 이유(비용·성능)가 Fargate에서도 그대로 이어진다. 바뀐 건 "그 ARM64 워크로드를 누가 호스팅하느냐"일 뿐, "ARM으로 포팅해둔 작업 자체"는 여전히 이득을 보고 있다.

EC2 launch type은 "내가 서버를 사서 Node.js 앱을 직접 배포·운영"하는 것과 비슷하고, Fargate는 Vercel/Netlify에 배포하는 것과 비슷하다 — 인스턴스 타입도, OS 패치도, 스케일링 설정도 신경 안 쓰고 "이 빌드를 얼마의 메모리로 실행해줘"만 지정하면 나머지는 플랫폼이 알아서 한다. 다만 Vercel도 "함수 메모리/타임아웃"은 여전히 내가 지정하는 것처럼, Fargate도 "태스크 CPU/메모리"는 여전히 우리 몫이다.
