거의 모든 레포가 카프카를 쓰고 있었다

이 조직의 레포를 훑어보면서 놀랐던 점 — 급여(Pay) 모듈 정도를 빼면 거의 모든 서비스가 카프카를 쓰고 있었다. 근태·연차·일일집계, 외부 출입(Secom/NICE) 기록 처리까지도. 그런데 팀 시니어(juhoLee)에게 이 얘길 하니 돌아온 답은 예상 밖이었다:

"사용 안 해야 하는 곳도 쓰고있긴 하죠ㅋㅋㅋ 잡 큐 대신 카프카로 다 때려박아서 사용하는 느낌... 기술 오용이랄까요"

"그런 건 큐가 필요한 거죠, 워커를 돌릴. Redis 같은 걸로 충분합니다"

이 레슨은 "카프카를 배우자"가 아니라, "카프카가 필요한 순간과 필요 없는 순간을 구분하는 눈"을 기르기 위한 레슨이다. 도구를 쓸 줄 아는 것보다, 안 써야 할 때를 아는 게 실무에서 훨씬 어렵다.

카프카와 Sidekiq은 겉보기엔 비슷하다

레슨 17에서 배운 Sidekiq(잡 큐)와 카프카(이벤트 스트림)는 둘 다 "지금 당장 안 하고 나중에 처리한다"는 점에서 비슷해 보인다. 하지만 원래 풀도록 설계된 문제가 다르다:

Sidekiq(잡 큐) Kafka(이벤트 스트림)
누가 읽나 보통 같은 앱의 워커 하나 (1:1) 다른 서비스 여러 개가 각자 구독 가능 (1:N)
처리 후 메시지 사라짐 (재실행 불가) 보관 기간 동안 남음 → 재처리·재생(replay) 가능
주 용도 "이 작업을 비동기로 실행해줘" (태스크 실행) "이런 일이 있었다는 걸 여러 곳에 알려줘" (사실의 전파)
인프라 무게 Redis 하나 (가볍다) 브로커 클러스터(MSK), 파티션·컨슈머 그룹 운영 필요 (무겁다)
Q1. 근태 모듈에서 "출퇴근 기록이 생기면 일일 집계를 다시 계산해줘"라는 작업이 있다. 이 작업을 소비하는 곳은 근태 모듈 자기 자신뿐이다. 어떤 도구가 더 적합한가?

이 프로젝트에서 카프카가 "맞는" 이유가 있는 곳

반대로, 레슨 25~32에서 계속 봐온 optimiz-async, record-hr-organization-code 같은 HR/조직 동기화 이벤트는 카프카가 맞는 선택이다:

  • 여러 서비스가 같은 이벤트를 구독한다 — theplus-back뿐 아니라 optimiz-time도 record-hr-organization-code를 구독한다 (1:N)
  • 발행자와 구독자가 다른 팀·다른 배포 주기의 서비스다 — Java(HR/System)와 Ruby(Performance)가 서로의 배포를 기다리지 않고 독립적으로 동작해야 한다
  • 실패 이력을 남기고 재처리해야 한다 — 레슨 29에서 본 DLQ, KafkaFailedMessage 테이블이 이걸 활용한다

반면 근태 일일집계, 외부 출입 기록 저장처럼 "한 서비스 안에서, 한 워커가 처리하면 끝"인 작업은 이 세 조건 중 어느 것도 해당하지 않는다 — 카프카가 주는 이점(다중 구독, 재생, 순서 보장)을 전혀 쓰지 않으면서, 운영 부담(브로커 클러스터, 파티션 설계, Consumer Group 관리)만 떠안는 셈이다.

Q2. 다음 중 카프카를 쓸 이유가 가장 뚜렷한 경우는?

판단 기준을 질문 형태로 정리하면

이 작업을 소비할 곳이 하나뿐인가, 여러 팀/서비스인가?
하나뿐 → Sidekiq   |   여러 곳 → Kafka 후보
나중에 처음부터 다시 재생(replay)할 필요가 있는가?
없음 → Sidekiq   |   있음 → Kafka 후보
발행자·구독자가 서로 다른 배포 주기를 가진 독립된 서비스인가?
아니오(같은 앱) → Sidekiq   |   예 → Kafka 후보

세 질문에 전부 "여러 곳/필요/독립 서비스"로 답할 수 있어야 카프카를 쓸 이유가 뚜렷해진다. 하나라도 "하나뿐/불필요/같은 앱"이면, 십중팔구 Redis 기반 잡 큐로 충분하다.
