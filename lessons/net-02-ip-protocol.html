🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · 인터넷 네트워크 — IP가 무엇이고, 왜 한계가 있으며, 그 한계를 어떻게 극복하는지 이해한다

핵심 한 줄

IP(Internet Protocol)는 인터넷에서 내 컴퓨터의 논리적 주소이고, 데이터를 패킷(Packet) 단위로 목적지까지 전달하는 규칙이다. 그러나 IP만으로는 비연결성·비신뢰성·포트 구분 불가라는 세 가지 한계가 있어서, 그 위에 TCP가 올라가야 실용적인 통신이 된다.

IP란 무엇인가

앞 강의에서 TCP/IP 4계층의 ③ 인터넷 계층이 나왔다. IP는 바로 그 계층의 핵심 프로토콜이다.

  • 네트워크 주소 + 호스트 주소로 구성 — 전자는 같은 네트워크 구성원이 공유, 후자는 그 안에서 유일한 식별자
  • 실제 표현: 32비트 2진수 (IPv4). 예: 192.168.0.103 = 11000000.10101000.00000000.01100111
  • 약 232(약 43억) 개 생성 가능 — 이미 대부분 소진 → IPv6로 대체 진행 중
배포된 Rails 서버 URL https://api.example.com을 환경변수로 들고 다니는 것처럼, IP 주소는 인터넷 세계에서 그 컴퓨터를 찾아가기 위한 논리적 주소(로케이터)다. 물리적 위치(건물·랙)가 아니라 "이 네트워크에서 나는 이 번호"라는 약속.

패킷(Packet): 데이터를 전달하는 단위

IP는 데이터를 패킷(Packet) 단위로 쪼개서 전달한다. 큰 파일을 한 덩어리로 보내지 않고 여러 패킷으로 나누는 이유는, 중간 노드가 처리하기 쉽고 경로를 유연하게 선택할 수 있기 때문이다.

하나의 패킷 구성:

구성 요소 내용
Payload 실제 전송할 데이터 한 블록
주소지 정보 발신지 IP 주소 + 목적지 IP 주소
관리 정보 (Header) TTL(수명), 버전, 체크섬 등 전달에 필요한 메타데이터
클라이언트 — 패킷 생성 (Payload + 출발 IP + 목적 IP)
↓
인터넷 노드 A — 목적 IP를 보고 다음 노드 B로 전달 (라우팅)
↓
인터넷 노드 B → … → 노드 N (hop by hop)
↓
서버 — 패킷 수신, Payload 꺼냄

IP의 한계 3가지

IP만으로는 실용적인 통신을 하기에 부족하다. 세 가지 한계가 있다.

① 비연결성 (Connectionless)

IP는 패킷을 보내기 전에 "상대가 받을 수 있는 상태인지" 확인하지 않는다. 상대 서버가 꺼져 있거나 서비스 불가 상태여도 그냥 전송한다. 편지를 상대방이 집에 있든 없든 우편함에 넣고 가는 것과 같다.

② 비신뢰성 (Unreliable)

중간 노드에서 패킷이 유실될 수 있고(라우터 과부하 등), 도착 순서도 보장되지 않는다. 큰 파일을 패킷 1·2·3으로 나눠 보냈는데 3 → 1 → 2 순서로 도착하거나, 2번 패킷이 사라질 수 있다.

③ 포트 구분 불가 (Port Ambiguity)

같은 IP 주소를 가진 서버에 HTTP 서버, WebSocket 서버, DB 등 여러 앱이 동시에 떠 있을 수 있다. IP만으로는 "이 패킷이 어느 앱으로 가야 하는지" 구분할 수 없다. → 이 문제는 PORT(강의 7)가 해결한다.

프론트에서 fetch()가 네트워크 에러를 던질 때 재시도 로직(retry)을 직접 구현해야 했던 이유가 여기 있다. IP 자체는 "보냈다"만 알지, "잘 받았다"를 보장하지 않는다. TCP가 바로 이 "잘 받았는지 확인"을 IP 위에 추가하는 계층이다.

한계를 극복하는 방법: TCP/UDP

IP의 한계(비신뢰성·비연결성)를 보완하기 위해 트랜스포트 계층에 TCP 또는 UDP를 올린다. 자세한 내용은 다음 강의에서 다룬다.

정리

  • IP = 인터넷에서 컴퓨터를 찾는 논리적 주소 (32비트, IPv4 → IPv6 전환 중)
  • 데이터 전달 단위 = 패킷 (Payload + 주소지 정보 + 관리 정보)
  • IP의 한계: 비연결성(수신 여부 미확인), 비신뢰성(유실·순서 미보장), 포트 구분 불가
