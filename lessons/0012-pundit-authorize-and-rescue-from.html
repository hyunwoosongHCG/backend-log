🎯 미션 연결: "왜 403이 반환되는가"를 코드에서 추적할 수 있다 — MISSION.md

오늘의 출발점

review_template_policy.rb를 리뷰하다가 이런 의문이 생겼다: 

# app/policies/review_template_policy.rb
def update?
  show? && !record.reviews.kept.exists?
end

이 메서드가 false를 반환하면 403이 된다고 했는데 — 어떤 경로로? 컨트롤러에서 에러를 직접 던지는 코드는 없다. 그런데 403이 나온다. 이 레슨에서 그 경로를 코드 수준에서 완전히 추적한다.

등장인물 세 명

이름 위치 하는 일
Pundit gem (외부 라이브러리) "이 사람이 이 작업을 할 권한이 있나?" 를 확인하는 도구
Policy 클래스 app/policies/*.rb 권한 규칙이 실제로 정의된 곳

1단계: authorize — 권한 확인 명령

컨트롤러에서 권한 확인이 필요한 시점에 한 줄을 쓴다:

# app/controllers/v2/reviews/review_templates.rb
put ':id' do
  authorize review_template, :update?   # ← 이 한 줄
  # ...
end

authorize는 Pundit이 제공하는 메서드다. 두 개의 인자를 받는다:

  • 첫 번째: 검사 대상 객체 (review_template)
  • 두 번째: 어떤 권한인지 (:update?)
React에서 비유하면 이렇다:

authorize(reviewTemplate, 'update') → "이 리뷰 템플릿을 update할 권한이 있어?"

2단계: Policy 클래스 — 권한 규칙 정의

Pundit은 네이밍 컨벤션으로 Policy 클래스를 자동으로 찾는다. 규칙은 단순하다: 모델명 + Policy. 

authorize review_template, :update?
    ↓
review_template.class  →  ReviewTemplate
    ↓
ReviewTemplate + "Policy"  →  ReviewTemplatePolicy
    ↓
ReviewTemplatePolicy.new(pundit_user, review_template).update?

실제 Policy 파일:

# app/policies/review_template_policy.rb
class ReviewTemplatePolicy < ApplicationPolicy
  def show?
    !record.discarded? && admin?
  end

  def update?
    show? && !record.reviews.kept.exists?
  end
end

세 가지를 짚고 가자:

  • ApplicationPolicy를 상속 — Pundit 규칙. admin? 같은 공통 메서드가 여기 정의돼 있다.
  • record — 검사 대상 객체. 여기서는 review_template. Pundit이 자동으로 두 번째 인자를 record에 담는다.
  • user — 권한을 확인할 사람. 바로 아래에서 설명.

pundit_user: "누구" 기준으로 권한을 확인하나?

Pundit은 기본적으로 current_user를 policy에 넘긴다. 그런데 이 프로젝트는 다르게 설정돼 있다: 

# app/controllers/grape_base.rb
def pundit_user
  Member.find_by(workspace: workspace, user: current_user) || current_user
end

User가 아닌 Member를 넘긴다. 이 앱에서 권한은 "누구인가"가 아니라 "이 워크스페이스에서 어떤 역할인가"로 결정되기 때문이다. 그래서 admin? 체크도 Member 기준으로 동작한다.

3단계: authorize 내부 — false면 예외를 던진다

authorize는 내부적으로 이렇게 동작한다: 

# Pundit 내부 (실제 소스를 단순화한 것)
def authorize(record, query)
  policy = policy(record)               # Policy 클래스 찾기
  result = policy.public_send(query)    # .update? 호출
  raise Pundit::NotAuthorizedError unless result  # false면 예외 발생!
  true
end

중요한 점은 컨트롤러 코드가 아무것도 하지 않아도 Pundit이 예외를 던진다는 것이다. 컨트롤러에 별도의 if/else가 없어도 된다.

update?가 false가 되는 두 가지 경우:

조건 이유 결과
show? == false 삭제된 템플릿 이거나 어드민이 아님 예외 발생 → 403
reviews.kept.exists? == true 살아있는 리뷰가 존재함 예외 발생 → 403

4단계: rescue_from — 전역 예외 핸들러

예외가 던져졌다. 이걸 누가 잡는가? grape_base.rb에 미리 등록된 핸들러가 잡는다: 

# app/controllers/grape_base.rb
rescue_from Pundit::NotAuthorizedError, with: :user_not_authorized

rescue_from은 "이 예외가 발생하면 이 메서드로 처리해라"라고 등록하는 것이다. 한 번 등록해두면 이후 어느 엔드포인트에서 예외가 발생하든 자동으로 잡는다.

JS의 try/catch와 비교: 
// JS: 매번 직접 감싸야 함
try {
  await authorizeUpdate(template)
} catch (e) {
  if (e instanceof NotAuthorizedError) {
    return res.status(403).json({ errors: [...] })
  }
}
# Ruby/Grape: 한 번 등록 → 전체 자동 적용
rescue_from Pundit::NotAuthorizedError, with: :user_not_authorized
JS에서는 매 함수마다 try/catch를 써야 하지만, rescue_from은 전체 API에 걸쳐 작동하는 전역 에러 핸들러다. Express의 app.use(errorHandler)와 같은 역할이다.

user_not_authorized: 실제 403을 만드는 메서드

# app/controllers/grape_base.rb
def user_not_authorized(exception)
  policy_name = exception.policy.class.to_s.underscore
  errors = exception.record&.errors&.full_messages
  if errors.blank?
    errors = [I18n.t("#{policy_name}.#{exception.query}", ...)]
  end
  error!({ errors: errors }, 403)   # ← Grape 메서드: 즉시 응답 종료
end

error!는 Grape가 제공하는 메서드다. 호출되는 순간 요청 처리가 즉시 중단되고 지정한 코드(403)로 응답이 나간다. 이후 코드는 실행되지 않는다.

이 프로젝트에 등록된 전체 핸들러

# app/controllers/grape_base.rb
rescue_from ActiveRecord::RecordNotFound,        with: :not_found                # 404
rescue_from ActiveRecord::RecordInvalid,         with: :bad_request              # 400
rescue_from Pundit::NotAuthorizedError,          with: :user_not_authorized      # 403
rescue_from Grape::Exceptions::ValidationErrors, with: :parameter_validate_error # 400
rescue_from ActiveRecord::RecordNotUnique,       with: :conflict                 # 409

예외 종류별로 HTTP 상태코드를 매핑해두는 전역 에러 맵이다. 컨트롤러에서 에러 코드를 직접 쓰지 않아도 되는 이유가 바로 이것이다.

전체 흐름 한눈에

① PUT /review_templates/:id 요청 도착
↓ GrapeBase before 훅
② authenticate_user! — 로그인 여부 확인 (실패 시 401)
↓ put ':id' 블록 진입
③ authorize review_template, :update? 호출
↓ Pundit이 ReviewTemplatePolicy 찾아서 .update? 실행
④ update? → true: 아래 코드 계속 실행 / false: Pundit::NotAuthorizedError 발생
↓ (false인 경우) rescue_from 이 잡음
⑤ user_not_authorized 실행 → error!({…}, 403)
↓
⑥ 클라이언트에게 403 Forbidden 반환
