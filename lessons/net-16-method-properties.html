HTTP 메서드의 속성

네트워크 · HTTP 메서드 강의 19 08:55
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 메서드 — 안전·멱등·캐시 가능 세 속성을 이해하고, API 설계와 재시도 전략에 어떻게 쓰이는지 파악한다

핵심 한 줄

HTTP 메서드에는 안전(Safe), 멱등(Idempotent), 캐시 가능(Cacheable) 세 가지 공식 속성이 있고, 이 속성이 재시도 안전성·캐시 전략·서버 설계에 직결된다.

속성 1: 안전(Safe)

정의: 메서드를 호출해도 리소스가 변경되지 않는 성질.

  • 안전한 메서드: GET, HEAD, OPTIONS
  • 안전하지 않은 메서드: POST, PUT, PATCH, DELETE

중요한 점: "안전"의 범위는 해당 리소스에 한정된다. GET 요청도 서버가 로그를 기록하거나 접속 통계를 갱신할 수 있다. 하지만 GET이 요청한 리소스 자체는 변경되지 않으므로 "안전"하다고 본다.

속성 2: 멱등(Idempotent)

정의: 같은 요청을 한 번 하든 열 번 하든 결과가 동일한 성질.

메서드 멱등 여부 이유
GET 멱등 같은 리소스를 조회하면 항상 같은 결과
PUT 멱등 같은 내용으로 대체하면 몇 번을 해도 상태 동일
DELETE 멱등 이미 삭제된 것을 다시 삭제해도 "없는 상태" 동일
POST 비멱등 회원 등록을 2번 하면 회원이 2명 생긴다
PATCH 구현에 따라 다름 절댓값 설정이면 멱등, 상대값 증가이면 비멱등

멱등의 실무 의미: 자동 재시도

멱등성이 왜 중요한가? 네트워크 오류 후 자동 재시도를 안전하게 할 수 있느냐가 결정되기 때문이다. 

# GET, PUT, DELETE는 재시도 안전 — 결과가 달라지지 않는다
GET /members/1   → 실패 → 재시도 → 동일한 데이터 반환 (안전)
DELETE /members/1 → 실패 → 재시도 → 이미 삭제됐거나 없음 (안전)

# POST는 재시도 위험 — 중복 실행 가능
POST /members → 실패? → 재시도 → 회원이 2명 생길 수 있다 (위험)

멱등하지 않은 POST를 재시도하려면 별도로 중복 방지 키(Idempotency Key)를 사용해야 한다. 결제 API에서 흔히 쓰는 패턴이다.

PATCH는 왜 "구현에 따라 다름"인가?

# 멱등한 PATCH: 절댓값으로 설정
PATCH /members/1 { "age": 31 }   # 몇 번을 해도 age = 31

# 비멱등한 PATCH: 상대값으로 증가
PATCH /members/1 { "age_increment": 1 }   # 3번 하면 age가 3 증가

속성 3: 캐시 가능(Cacheable)

응답 결과를 브라우저·CDN·프록시 등이 저장해서 재사용할 수 있는 성질이다.

메서드 명세상 캐시 가능 실무에서 실제 사용
GET 가능 주로 사용 (URL이 캐시 키)
HEAD 가능 사용
POST 가능 거의 사용 안 함 (바디까지 캐시 키로 고려해야 해 구현 복잡)
PATCH 가능 거의 사용 안 함 (위와 같은 이유)
PUT, DELETE 불가 사용 안 함

결론적으로 실무에서 캐시는 GET, HEAD에서만 쓴다고 이해해도 무방하다.

세 속성 전체 요약 표

메서드 안전(Safe) 멱등(Idempotent) 캐시 가능(Cacheable)
GET 안전 멱등 가능
HEAD 안전 멱등 가능
OPTIONS 안전 멱등 불가
POST 안전하지 않음 비멱등 명세상 가능, 실무 불가
PUT 안전하지 않음 멱등 불가
PATCH 안전하지 않음 구현에 따라 다름 명세상 가능, 실무 불가
DELETE 안전하지 않음 멱등 불가
React Query / SWR에서 useQuery는 GET 요청에만 쓰고, useMutation은 POST/PUT/PATCH/DELETE에 쓰는 이유가 정확히 이 속성들 때문이다. useQuery는 캐싱·자동 재시도를 기본으로 켜지만, useMutation은 멱등성이 없으니 재시도를 신중하게 제어한다. 라이브러리 설계 자체가 HTTP 메서드의 속성을 그대로 반영한 것이다.

정리

  • 안전(Safe): 리소스 변경 없음. GET·HEAD·OPTIONS만 해당
  • 멱등(Idempotent): 반복 호출 = 같은 결과. GET·PUT·DELETE는 멱등, POST는 비멱등
  • 캐시 가능(Cacheable): 실무에서는 GET·HEAD만 사용한다고 이해하면 충분
  • 멱등성은 자동 재시도 안전성에 직결 — API 설계 시 반드시 고려
  • PATCH의 멱등 여부는 서버 구현에 달려 있다 (절대값 vs 상대값)
