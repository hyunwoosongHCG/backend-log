🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 메서드 활용 — 리소스 관리 방식(컬렉션·스토어·문서·컨트롤러)을 파악하고 실무 URI 설계 판단력을 기른다

핵심 한 줄

HTTP API 설계는 "리소스 URI는 명사, 행위는 HTTP 메서드"가 원칙이며, 리소스를 누가 관리하느냐(서버 vs 클라이언트)에 따라 컬렉션(Collection)과 스토어(Store)로 나뉜다. 이 두 패턴으로 해결이 안 될 때 컨트롤 URI(Control URI)를 쓴다.

리소스 설계의 원칙: 명사로 URI, 행위는 메서드로

URI를 설계할 때 가장 중요한 규칙은 리소스(Resource)와 행위(Action)를 분리하는 것이다. 리소스는 URI(경로)로 표현하고, 행위는 HTTP 메서드(GET/POST/PUT/PATCH/DELETE)로 표현한다.

나쁜 예 (행위가 URI에) 좋은 예 (행위는 메서드로)
POST /createMember POST /members
GET /getMember/1 GET /members/1
POST /deleteMember/1 DELETE /members/1
GET /memberList GET /members
React 컴포넌트로 치면: 컴포넌트 이름(리소스)과 props로 받는 동작(행위)을 분리하는 것과 같다. <UserCard onDelete={...} />에서 "UserCard"는 리소스(명사), "onDelete"는 행위(동사). URI 설계도 똑같이 /users/1은 리소스, DELETE는 행위다.

리소스 관리 방식 4가지 (아키텍처 패턴)

1. 문서(Document) — 단일 개념

파일 하나, 객체 인스턴스 하나, DB의 행(row) 하나처럼 단일 개념을 나타낸다. 컬렉션의 하위 리소스가 문서다. 

GET  /members/1        # 회원 1번 조회
GET  /files/star.jpg   # 파일 하나 조회

2. 컬렉션(Collection) — POST 기반, 서버가 URI 관리

서버가 리소스 URI를 생성하고 관리하는 방식이다. 클라이언트는 "등록해줘"만 요청하고, 서버가 새로 생성된 리소스의 식별자(id)를 결정한다. 회원 관리 API가 대표적인 예다. 

# 클라이언트: 어디에 저장될지 모름, 그냥 요청만
POST /members
{ "username": "hyunwoo", "email": "hw@example.com" }

# 서버: 직접 id를 생성하고 URI를 만들어 응답
HTTP/1.1 201 Created
Location: /members/100    ← 서버가 /members/100 을 만들었다고 알려줌

서버가 auto-increment id(DB의 AUTO_INCREMENT)로 식별자를 만든다. 클라이언트는 생성된 URI를 모르다가 201 Created + Location 헤더로 알게 된다.

동작 메서드 URI 설명
목록 조회 GET /members 회원 목록
등록 POST /members 새 회원 생성, URI는 서버가 결정
단건 조회 GET /members/{id} 특정 회원 조회
수정 PATCH /members/{id} 회원 정보 부분 수정
삭제 DELETE /members/{id} 회원 삭제

3. 스토어(Store) — PUT 기반, 클라이언트가 URI 관리

클라이언트가 리소스 URI를 알고 직접 관리하는 방식이다. PUT을 주로 사용한다 — 없으면 생성, 있으면 덮어쓴다. 파일 관리 시스템이 대표적인 예다. 파일 이름은 클라이언트가 직접 지정한다. 

# 클라이언트: 파일 이름(star.jpg)을 클라이언트가 직접 알고 지정
PUT /files/star.jpg
[파일 바이너리 데이터]

# 없으면 생성(201 Created), 있으면 덮어쓰기(200 OK)
HTTP/1.1 201 Created
동작 메서드 URI 설명
목록 조회 GET /files 파일 목록
단건 조회 GET /files/{filename} 특정 파일 조회
등록/수정 PUT /files/{filename} 클라이언트가 URI(파일명)를 직접 지정
삭제 DELETE /files/{filename} 파일 삭제
컬렉션 vs 스토어의 핵심 질문: "id(식별자)를 누가 만드나?"
컬렉션 = 서버가 만든다 (DB auto-increment, UUID 생성 등). React 앱에서 회원 가입 후 서버가 돌려준 { id: 100 }을 받아서 쓰는 게 이것.
스토어 = 클라이언트가 이미 알고 지정한다. 파일 업로드할 때 파일명(star.jpg)은 내가 정해서 서버에 PUT하는 것.

4. 컨트롤러/컨트롤 URI(Controller / Control URI) — 동사 사용

문서·컬렉션·스토어만으로는 표현하기 어려운 추가 프로세스(비즈니스 로직)를 실행할 때 쓴다. HTTP 메서드로 행위를 표현하기 어려울 때 예외적으로 URI에 동사(행위)를 사용한다.

특히 HTML Form은 GET과 POST만 지원하기 때문에, PUT/PATCH/DELETE 같은 행위를 Form으로 전송해야 할 때 컨트롤 URI를 설계한다. 

# 문서/컬렉션 패턴으로 표현하기 어려운 행위들
POST /members/{id}/resign     # 회원 탈퇴 처리 (단순 DELETE가 아닌 탈퇴 프로세스)
POST /members/{id}/activate   # 회원 활성화
POST /orders/{id}/cancel      # 주문 취소 (취소 로직 실행)
POST /orders/{id}/ship        # 배송 시작

# HTML Form 제약 때문에 사용하는 컨트롤 URI
GET  /members/{id}/edit        # 수정 폼 조회
POST /members/{id}/delete      # Form으로 DELETE를 못 보내니 POST로
POST /members/{id}/update      # Form으로 PATCH를 못 보내니 POST로

컨트롤 URI는 최대한 억제하고, 정말 HTTP 메서드로 해결이 안 될 때만 쓴다. Rails의 resources 라우팅과 Grape API 설계에서 member do로 추가하는 커스텀 액션이 바로 이 컨트롤 URI에 해당한다.

컬렉션 vs 스토어 한눈 비교

컬렉션 (Collection) 스토어 (Store)
주요 메서드 POST로 등록 PUT으로 등록/수정
URI 결정 주체 서버 클라이언트
대표 예시 회원 관리 API 파일 저장소
등록 응답 201 + Location 헤더 201 or 200
실무 빈도 매우 높음 (대부분의 API) 상대적으로 낮음

실무 설계 판단 흐름

이 API가 리소스를 다루는가?
↓
URI는 명사로 → /members, /orders, /files
↓
id를 누가 생성하나? 서버 → 컬렉션(POST) / 클라이언트 → 스토어(PUT)
↓
메서드로 행위 표현이 안 되나? → 컨트롤 URI (최후 수단)

정리

  • URI는 명사(리소스), 행위는 HTTP 메서드로 표현
  • 컬렉션(Collection): POST 기반, 서버가 URI 생성 → 대부분의 REST API
  • 스토어(Store): PUT 기반, 클라이언트가 URI 지정 → 파일 업로드 등
  • 문서(Document): 단일 리소스 (컬렉션 하위의 /members/1 같은 형태)
  • 컨트롤 URI: 동사 포함 URI — Form 제약이나 복잡한 프로세스에서만 사용
  • Rails resources 라우팅이 컬렉션 패턴을 자동으로 구현해준다
