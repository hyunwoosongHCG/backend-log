resources 한 줄 — 실제로 몇 가지 경우가 숨어있나

레슨 023 resources · canonical actions · collection/member/new ~20분
🎯 ppback config/routes/v1.rb의 resources :multisource_feedbacks do collection { get 'new' } end를 보고 "왜 :id 붙은 라우트까지 자동으로 생기지? 이 한 줄에 경우의 수가 몇 개나 숨어있는 거야?" 싶었던 것에서 시작 — MISSION.md

React Router와 다른 출발점

React Router라면 라우트를 하나하나 손으로 등록한다: 

<Route path="/photos" element={<PhotoList />} />
<Route path="/photos/:id" element={<PhotoDetail />} />
<Route path="/photos/new" element={<PhotoForm />} />
// ...생성/수정/삭제도 각각 등록

Rails는 정반대다. "REST 자원 하나 = CRUD 라우트 세트"라는 관습(convention)이 있어서, resources :photos 딱 한 줄이 저 여러 개를 자동으로 만들어낸다. 문제는 "자동"이라는 말 뒤에 실제로는 여러 겹의 조건 분기(경우의 수)가 숨어있다는 것 — 오늘은 그걸 다 펼쳐본다.

💡 프론트로 치면: resources :photos는 React Router의 개별 <Route> 여러 줄이 아니라, "CRUD 페이지 세트를 만들어주는 코드 제너레이터(scaffolding)"에 더 가깝다. 다만 이 제너레이터가 상황(api_only 여부, only/except, 블록 유무, 중첩 여부)에 따라 결과물을 다르게 뱉는다는 게 오늘 배울 "경우의 수"다.

경우의 수 ① — API 전용 모드냐 아니냐 (액션 개수: 5개 vs 7개)

Rails의 resources는 원래 아래 7개 액션을 기본으로 만든다: index, new, create, show, edit, update, destroy. 그런데 ppback은 config/application.rb에서 config.api_only = true로 설정되어 있다. API 전용 모드에서는 화면(HTML 폼)을 그리기 위한 액션인 new·edit이 애초에 의미가 없으므로, Rails 프레임워크 코드 자체가 이 둘을 자동으로 뺀다: 

# actionpack 소스 (action_dispatch/routing/mapper.rb)
def default_actions
  if @api_only
    [:index, :create, :show, :update, :destroy]              # 5개
  else
    [:index, :create, :new, :show, :update, :destroy, :edit]  # 7개
  end
end
Q1. config.api_only = true인 앱에서 resources :photos가 기본으로 만들지 않는 액션 두 가지는?

경우의 수 ② — 액션 이름이 URL에 노출되느냐 (canonical 여부)

5~7개 액션 모두가 "그 이름 그대로" URL에 나타나는 건 아니다. Rails는 index, create, new, show, update, destroy를 canonical action(관용적인 액션)으로 취급해서, 이 액션들은 URL에 이름을 붙이지 않고 "URL 모양 + HTTP 메서드"만으로 구분한다. 반면 edit은 이 목록에 없어서 URL에 이름이 그대로 붙는다. 

CANONICAL_ACTIONS = %w(index create new show update destroy)   # edit은 없다!

GET    /photos          → index    (이름 없음, GET+목록 경로로 구분)
GET    /photos/:id      → show     (이름 없음, GET+id 경로로 구분)
GET    /photos/:id/edit → edit     (canonical이 아니라서 "edit"이 그대로 URL에 붙음)
Q2. 다음 중 URL에 액션 이름이 그대로 노출되는 것은?

경우의 수 ③ — 어느 스코프에 속하느냐 (collection / member / new)

Rails 내부적으로 액션마다 URL "자리"가 정해져 있다. 이 자리를 스코프(scope)라고 부르고, 세 종류뿐이다:

스코프 URL 모양 여기 속하는 기본 액션
collection /photos (id 없음) index, create
member /photos/:id (id 있음) show, update, destroy, edit
new /photos/new new

이 세 스코프는 do...end 블록으로 직접 열어서 커스텀 라우트를 추가할 수도 있다. 예를 들어 "사진 미리보기" 같은, RESTful 7종에 없는 기능을 추가하고 싶다면: 

resources :photos do
  member do
    get 'preview'   # member 스코프 = id가 붙는 자리
  end
end
Q3. 위 코드처럼 member do get 'preview' end라고 쓰면 생기는 URL은?

경우의 수 ④ — only / except로 액션 자체를 줄이기

5개 기본 액션이 전부 필요하지 않을 때도 있다. ppback에서 실제로 본 예: 

# config/routes/v1.rb
resources :multisource_feedback_responses, only: [:show, :update, :destroy]

이렇게 쓰면 index와 create는 아예 라우트로 만들어지지 않는다. (이 리소스는 다른 API로 생성/목록 조회를 하고, 개별 항목 조회/수정/삭제만 이 라우트로 하는 경우다.) 반대로 except: [:destroy]처럼 "이것만 빼고 다 만들어라"도 가능하다.

Q4. resources :multisource_feedback_responses, only: [:show, :update, :destroy]일 때 생기지 않는 라우트는?

경우의 수 ⑤ — do...end 블록은 "추가"이지 "대체"가 아니다

처음 봤을 때 가장 헷갈렸던 부분. ppback 실제 코드: 

resources :multisource_feedbacks do
  collection do
    get 'new'
  end
end

이 코드를 보면 get 'new' 하나만 눈에 보여서, "그럼 show·update·destroy는 어디서 온 거지?" 라는 의문이 생긴다. 답은 — 이 블록은 기본 5개 액션 위에 얹는 "추가분"일 뿐이라는 것. Rails resources 내부 코드를 보면 순서가 이렇다: 

def resources(*resources, &block)
  ...
  resource_scope(Resource.new(...)) do
    yield if block_given?              # ① 사용자가 쓴 do...end 블록 — 딱 한 번 실행
                                        #    (지금 예시에선 get 'new' 하나)

    collection do                      # ② 여기부터는 블록 유무와 무관하게 무조건 실행
      get  :index  if ...
      post :create if ...
    end

    set_member_mappings_for_resource   # ③ show/update/destroy(=id 붙는 라우트)가 여기서 나옴
  end
end

즉 do...end 블록을 통째로 지워도 index·create·show·update·destroy는 그대로 생긴다. 블록이 있어야만 생기는 건 new 하나뿐이다 (원래 api_only라 자동 제외됐던 걸 누군가 수동으로 되살려놓은 것).

Q5. resources :photos do collection { get 'new' } end에서 show/update/destroy 라우트는 어디서 오나?

경우의 수 ⑥ — 중첩(nested)되면 URL 앞에 상위 자원이 붙는다

실제 라우트를 rails routes -c multisource_feedbacks로 찍어보면 URL이 /api/v1/workspaces/:workspace_id/multisource_feedbacks로, 앞에 workspaces/:workspace_id가 붙어 있다. 이건 resources :multisource_feedbacks가 resources :workspaces 블록 안에 중첩되어 있기 때문이다: 

resources :workspaces do
  resources :multisource_feedbacks    # 중첩 → 상위 workspaces의 :id가 URL 앞에 붙는다
end
# 최종: /workspaces/:workspace_id/multisource_feedbacks

"이 360피드백이 어느 workspace 소속인지"가 URL 자체에 박히기 때문에, 컨트롤러에서 params[:workspace_id]로 항상 상위 자원을 먼저 찾아 권한 검사를 할 수 있게 된다 (지난 PR 리뷰에서 봤던 workspace.all_multisource_feedbacks.where(...) 같은 보안 스코프의 근거이기도 하다).

Q6. resources :workspaces do resources :multisource_feedbacks end이면 최종 URL 접두어는?

마지막 한 조각 — "controller#action" 문자열의 정체

지금까지의 모든 경우의 수는 결국 라우팅 테이블에 "multisource_feedbacks#show" 같은 문자열을 등록하는 걸로 끝난다. 요청이 실제로 오면 Rails가 이 문자열을 # 기준으로 쪼갠다: 

"multisource_feedbacks#show"
 └ 왼쪽: "multisource_feedbacks" → 카멜케이스 + Controller → MultisourceFeedbacksController.new
 └ 오른쪽: "show"                → 그 인스턴스의 show 메서드를 호출

지금까지 배운 6가지 경우의 수는 전부 "이 문자열이 정확히 뭐가 되는지, 그리고 어떤 URL+메서드 조합에 붙는지"를 결정하는 규칙들이었다.

핵심 요약 — 경우의 수 총정리

경우의 수 갈리는 기준 ppback 예시
① api_only 여부 5개 액션 vs 7개 액션 config.api_only = true → new/edit 자동 제외
② canonical 여부 URL에 액션명 노출 안 함 vs 함 show/update는 숨김, edit은 노출
③ 스코프 종류 collection / member / new collection { get 'new' }
④ only / except 기본 액션 화이트/블랙리스트로 축소 multisource_feedback_responses, only: [...]
⑤ 커스텀 블록 기본 라우트에 "추가"(대체 아님) collection { get 'new' }
⑥ nested 상위 자원의 :id가 URL 접두어로 붙음 /workspaces/:workspace_id/multisource_feedbacks
