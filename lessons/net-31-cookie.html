쿠키

🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 일반 — Stateless HTTP에서 상태를 유지하는 메커니즘인 쿠키의 동작 원리·생명주기·보안 속성 전반을 파악한다

핵심 한 줄

HTTP는 기본적으로 상태가 없다(Stateless). 쿠키는 브라우저가 서버에서 받은 데이터를 저장했다가 이후 요청마다 자동으로 첨부해서 보내는 방식으로 상태를 유지하는 메커니즘이다.

왜 쿠키가 필요한가

HTTP는 요청-응답이 끝나면 서버가 클라이언트를 기억하지 않는다. 로그인을 했더라도 다음 요청은 서버 입장에서 완전히 새로운 요청이다.

이를 해결하는 가장 원시적인 방법은 매 요청에 로그인 정보를 직접 담아 보내는 것이다. 하지만 이 방식은 두 가지 문제가 있다.

  • 모든 요청마다 직접 정보를 담아야 하므로 구현이 번거롭다
  • URL이나 요청 파라미터에 노출되면 보안 위험이 크다

쿠키는 이 문제를 해결한다. 서버가 "이 데이터를 가지고 있어라"라고 응답에 담아주면, 브라우저가 저장해뒀다가 이후 같은 서버에 요청할 때마다 자동으로 첨부해서 보낸다.

쿠키 동작 흐름

클라이언트: 로그인 요청 (POST /login)
↓
서버: 인증 성공 → Set-Cookie: session_id=abc123; HttpOnly 응답
↓
브라우저: 쿠키 저장
↓
클라이언트: 다음 요청 시 자동으로 Cookie: session_id=abc123 포함
↓
서버: 쿠키를 읽어 "이 사용자는 로그인된 상태"임을 확인

쿠키의 용도

  • 세션 관리(Session Management): 로그인 상태 유지, 장바구니, 게임 스코어
  • 개인화(Personalization): 다크모드 테마, 언어 설정, 사용자 맞춤 설정
  • 트래킹(Tracking): 사용자 행동 분석, 광고 타겟팅

주의사항

쿠키는 설정된 서버에 대한 모든 요청에 자동으로 전송된다. 이미지 하나를 불러오는 요청에도 쿠키가 붙어 나간다는 뜻이다. 네트워크 트래픽이 늘어나므로 최소한의 정보만 쿠키에 저장해야 한다. 그리고 주민번호, 카드번호 같은 보안 민감 정보는 절대 쿠키에 저장하지 않는다.

Web Storage — 쿠키의 대안

서버로 전송할 필요가 없는 데이터는 Web Storage(웹 스토리지)를 사용한다. HTML5에서 도입됐으며, 같은 도메인(프로토콜·호스트·포트가 같은 경우) 내에서 공유된다.

구분 localStorage sessionStorage 쿠키
접근 window.localStorage window.sessionStorage document.cookie
유지 영구 (직접 삭제 전까지) 탭/브라우저 닫으면 삭제 만료 시간 설정 가능
서버 전송 안 함 안 함 자동 전송
용량 약 5MB 약 5MB 약 4KB
프론트 개발할 때 localStorage.setItem('theme', 'dark')처럼 UI 설정을 저장했던 경험이 있을 것이다. 이 데이터는 서버로 전송되지 않으니 서버가 전혀 모른다. 반면 로그인 세션은 서버가 알아야 하므로 쿠키(또는 쿠키에 담긴 세션 ID)를 쓴다. "서버가 알아야 하나?" → 쿠키. "브라우저만 알면 되나?" → localStorage/sessionStorage.

localStorage의 함정: 값은 항상 string이다. 객체를 그냥 저장하면 [object Object]가 된다. 반드시 JSON.stringify()로 직렬화해서 저장하고, 꺼낼 때는 JSON.parse()로 역직렬화해야 한다. 

// 잘못된 예
localStorage.setItem('user', { name: 'Kim' });
localStorage.getItem('user'); // "[object Object]"

// 올바른 예
localStorage.setItem('user', JSON.stringify({ name: 'Kim' }));
JSON.parse(localStorage.getItem('user')); // { name: 'Kim' }

쿠키 생명주기 (Expires / Max-Age)

쿠키에는 만료 시간을 설정할 수 있다.

Set-Cookie: session_id=abc123; Expires=Sat, 10-Jan-2026 04:39:21 GMT
Set-Cookie: session_id=abc123; Max-Age=3600
  • Expires: 만료 날짜를 절대 시각(GMT)으로 지정
  • Max-Age: 현재 시각으로부터 몇 초 후에 만료할지 상대적으로 지정. 0 또는 음수면 즉시 삭제된다

세션 쿠키(Session Cookie): Expires와 Max-Age를 모두 생략하면 브라우저를 닫을 때 삭제된다.
영속 쿠키(Persistent Cookie): 만료 날짜를 지정하면 브라우저를 닫아도 그 날짜까지 유지된다.

쿠키 도메인 (Domain)

domain 속성으로 쿠키를 전송할 도메인 범위를 지정한다.

  • domain 명시: 해당 도메인과 모든 서브도메인에 쿠키가 전송된다.
    domain=example.org → example.org와 dev.example.org 둘 다 전송
  • domain 생략: 현재 문서 도메인에만 전송된다. 서브도메인에는 전송 안 됨.

쿠키 경로 (Path)

path 속성이 지정된 경로와 그 하위 경로에서만 쿠키가 전송된다. 

Set-Cookie: token=abc; Path=/admin

위 설정이면 /admin, /admin/users에는 쿠키가 전송되지만 /나 /public에는 전송되지 않는다. 일반적으로는 Path=/로 설정해 모든 경로에서 쿠키를 전송한다.

쿠키 보안 속성

Secure

이 속성이 있으면 HTTPS 연결에서만 쿠키가 전송된다. HTTP(평문) 연결에서는 전송하지 않아 도청(스니핑)으로부터 보호한다. 

Set-Cookie: token=abc; Secure

HttpOnly

이 속성이 있으면 JavaScript에서 쿠키에 접근할 수 없다. 즉, document.cookie로 이 쿠키를 읽거나 수정하는 것이 불가능해진다. XSS(Cross-Site Scripting) 공격으로부터 쿠키를 보호하는 핵심 속성이다. 

Set-Cookie: session_id=abc; HttpOnly
Rails 세션 쿠키는 기본적으로 HttpOnly가 붙어 있다. 개발자 도구 콘솔에서 document.cookie를 입력해도 Rails 세션 쿠키가 보이지 않는 이유가 이것이다. 악성 JS가 주입돼도 세션 쿠키를 훔쳐갈 수 없도록 OS 수준에서 차단한다.

SameSite

쿠키가 어떤 출처의 요청에 함께 전송될지를 제어한다. CSRF(Cross-Site Request Forgery) 공격 방어가 목적이다.

  • Strict: 같은 사이트 요청에서만 전송. 외부 사이트에서 링크를 클릭해 이동할 때도 쿠키를 안 보냄
  • Lax: 외부에서 링크 클릭(GET 탐색)은 허용, 외부에서의 POST/PUT 등은 차단. 현재 대부분 브라우저의 기본값
  • None: 크로스 사이트 요청에도 쿠키 전송. 반드시 Secure와 함께 써야 함

보안 위협 — XSS와 CSRF

XSS (Cross-Site Scripting, 크로스 사이트 스크립팅)

공격자가 웹 페이지에 악성 JavaScript를 주입해서 다른 사용자의 세션·쿠키를 탈취하는 공격이다.

  • Stored XSS: 악성 스크립트를 DB에 저장해 (예: 게시글 내용) 다른 사용자가 조회 시 실행됨
  • Reflected XSS: URL 파라미터에 스크립트를 담아 피해자를 유도
  • DOM-based XSS: 클라이언트 JS가 URL/DOM을 그대로 렌더링하면서 발생

방어: HttpOnly 쿠키 사용 시 JS로 쿠키를 읽을 수 없어 탈취를 차단. 서버에서 사용자 입력을 이스케이프하거나 CSP(Content-Security-Policy) 헤더 설정도 필수.

CSRF / XSRF (Cross-Site Request Forgery, 크로스 사이트 요청 위조)

피해자가 이미 사이트 A에 로그인된 상태에서, 공격자가 만든 외부 페이지(사이트 B)가 사이트 A에 요청을 보내도록 유도하는 공격이다. 브라우저가 쿠키를 자동으로 첨부하기 때문에 발생한다. 

<!-- 사이트 B에 심어진 악성 코드 -->
<img src="https://bank.com/api/transfer?to=attacker&amount=1000000">

피해자가 사이트 B를 방문하기만 해도 위 이미지 태그가 bank.com에 GET 요청을 보내고, 브라우저가 자동으로 bank.com의 쿠키를 첨부해서 전송한다. 서버는 쿠키가 있으니 정상 요청으로 착각한다.

방어 방법:

  • SameSite 쿠키: 외부 사이트 요청에는 쿠키를 첨부하지 않도록 설정
  • CSRF 토큰: 서버가 폼마다 고유한 토큰을 발급하고, 요청 시 이를 검증. Rails는 protect_from_forgery로 기본 활성화됨
Rails에서 ApplicationController에 기본으로 들어있는 protect_from_forgery with: :exception이 바로 CSRF 토큰 검증이다. HTML 폼을 렌더링할 때 Rails가 자동으로 숨겨진 authenticity_token 필드를 넣어준다. API 모드(Grape, JSON 응답)에서는 대신 SameSite 쿠키와 Authorization 헤더 방식으로 방어한다.

정리

  • 쿠키: 서버가 Set-Cookie로 발급 → 브라우저 저장 → 이후 요청마다 Cookie 헤더로 자동 전송
  • 용도: 세션 관리(로그인), 개인화, 트래킹. 보안 민감 정보는 저장 금지
  • Web Storage: 서버 전송 불필요한 데이터는 localStorage(영구) / sessionStorage(탭 종료 시 삭제) 사용
  • 생명주기: Expires(절대 시각) 또는 Max-Age(상대 초) 설정. 생략 시 세션 쿠키
  • Secure: HTTPS에서만 전송
  • HttpOnly: JS에서 접근 불가 → XSS 방어
  • SameSite: 크로스 사이트 요청 시 쿠키 전송 제어 → CSRF 방어
