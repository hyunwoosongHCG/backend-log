카프카 Producer(발행자)

레슨 026 Producer · kafkajs · produce_async · Fire-and-forget ~15분
🎯 "메시지를 보냈는데 응답이 없다 — 실패인가?" — 발행이 요청/응답과 뭐가 다른지 이해한다 — MISSION.md

Producer란

Producer(프로듀서)는 토픽에 메시지를 써넣는 쪽이다. theplus-aws-lambda의 people-sync Lambda가 대표적인 예다. HR 시스템에서 새 멤버가 생성되면, 이 Lambda는 Postgres에 직접 write하는 것과 별개로 perpl-account 토픽에도 메시지를 하나 발행한다. 

// theplus-aws-lambda/people-sync/peopleModelLayer/config/kafka.js
class KafkaService {
  async sendKafkaMessage({ key, topic, payload, maxRetries = 2 }) {
    const producer = this.kafka.producer();
    await producer.connect();
    await producer.send({
      topic,
      messages: [{ key, value: JSON.stringify(payload) }],
    });
    await producer.disconnect();
  }
}

// userSynchronizer.js — 멤버 생성 시 호출
KafkaService.sendKafkaMessage({
  key: 'User',
  topic: 'perpl-account',
  payload: { uid, email, display_name, workspace_hash_id },
});

여기서 중요한 건, 이 코드가 "누가 이 메시지를 받는지 전혀 모른다"는 점이다. 실제로는 optimiz-system(Java 서비스)의 PerplAccountConsumer가 받아서 처리하지만, Lambda 쪽 코드에는 그 사실이 전혀 등장하지 않는다. 그냥 perpl-account라는 주소로 던질 뿐이다.

HTTP 요청과 비교하면 이해가 쉽다.

await fetch('/api/users', { method: 'POST', body })는 서버가 처리를 끝내고 응답(JSON, 상태 코드)을 줄 때까지 기다린다.

반면 카프카 발행은 window.dispatchEvent(new CustomEvent('user:created', { detail }))에 더 가깝다. 이벤트를 던지고 나면 그걸로 끝 — 누가 들었는지, 뭘 했는지 발행자는 신경 쓰지 않는다. "받았다"는 응답이 아니라 "브로커에 잘 적재됐다"는 확인만 받는다.
Q1. producer.send()가 성공적으로 끝났다는 것의 의미는?

같은 서비스가 Producer이면서 Consumer일 수 있다

"Producer/Consumer"는 앱 전체의 역할이 아니라 메시지 한 건 단위의 역할이다. theplus-back은 카프카를 구독하는 Consumer이지만, 동시에 자기도 이벤트를 발행한다. 예를 들어 멤버가 활성화되면 Produce::UserStatusProduce가 user-performance-module-status 토픽에 메시지를 쏜다: 

# theplus-back/app/services/produce/user_status_produce.rb (요약)
class Produce::UserStatusProduce
  def self.call(user:, type:)
    payload = { type: type, data: { uid: user.uid } }
    Karafka.producer.produce_async(
      topic: 'user-performance-module-status',
      payload: payload.to_json
    )
  end
end

즉 theplus-back은 record-hr-organization-code 같은 토픽에 대해서는 Consumer지만, user-performance-module-status에 대해서는 Producer다. 한 애플리케이션 안에 두 역할이 공존한다.

Q2. 한 Rails 앱이 카프카 토픽 A는 구독(consume)하면서, 처리 결과를 토픽 B에 발행(produce)할 수 있는가?

발행 당시 구독자가 없어도 괜찮다

자바스크립트 EventEmitter나 DOM 이벤트는 리스너가 등록돼 있지 않은 순간에 이벤트가 발생하면 그냥 유실된다. 카프카는 다르다 — 토픽은 메시지를 로그처럼 일정 기간 보관하기 때문에, Consumer가 나중에 붙어도(또는 재시작해서 다시 붙어도) 못 읽은 메시지부터 이어서 읽을 수 있다. ppback의 karafka.rb에 Consumer 라우팅이 주석 처리돼 있어도 Producer 코드(log_producer.rb)는 멀쩡히 도는 이유가 이거다 — 발행은 구독자의 존재 여부와 무관하다.

핵심 요약

개념 역할
Producer 토픽에 메시지를 써넣는 쪽. 누가 읽는지 몰라도 된다
produce_async / send "브로커가 받았다"까지만 보장. Consumer 처리 완료를 기다리지 않는다
역할의 단위 앱 전체가 아니라 토픽 단위. 한 앱이 A는 소비, B는 발행 둘 다 가능
