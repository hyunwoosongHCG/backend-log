핵심 한 줄

DNS(Domain Name System)는 도메인 이름 → IP 주소를 매칭해주는 전화번호부다. IP 주소는 외우기 어렵고 바뀔 수 있으므로, 사람이 기억하기 쉬운 이름으로 IP를 찾아주는 분산형 주소록 서비스가 필요하다.

왜 DNS가 필요한가

IP 주소를 직접 쓰면 두 가지 문제가 생긴다.

  • 외우기 어렵다: 142.250.196.46보다 google.com이 훨씬 기억하기 쉽다. 서비스가 늘어날수록 IP를 직접 관리하는 건 불가능에 가깝다.
  • IP가 바뀔 수 있다: 서버를 교체하거나 클라우드를 이전하면 IP가 달라진다. IP를 직접 쓰고 있었다면 모든 클라이언트의 설정을 바꿔야 한다. DNS에 등록된 매핑만 업데이트하면 도메인 이름은 그대로 유지된다.
프론트에서 API 서버 주소를 하드코딩하지 않고 환경변수로 빼두는 것과 같다. VITE_API_URL=https://api.example.com처럼 "이름 → 실제 주소" 매핑을 한 곳에서 관리하면, 서버 주소가 바뀌어도 코드 전체를 고칠 필요가 없다. DNS는 인터넷 전체를 대상으로 하는 그 환경변수 서버다.

DNS의 동작 원리

DNS는 Key(도메인 이름) / Value(IP 주소) 쌍을 저장하는 분산 데이터베이스다. google.com → 142.250.196.46 같은 레코드가 전 세계 DNS 서버에 분산 저장돼 있다.

1. 브라우저에 도메인 입력
사용자가 https://google.com을 입력
↓
2. DNS 서버에 질의
브라우저(또는 OS)가 DNS 서버에 "google.com의 IP가 뭐야?"를 물어봄
↓
3. DNS 서버가 IP 응답
DNS 서버가 매칭되는 IP 주소 142.250.196.46를 응답
↓
4. IP로 접속
브라우저가 받은 IP로 실제 HTTP 요청을 보냄

실무에서 DNS와 마주치는 순간들

1. 서버 배포 시 도메인 연결

Rails 앱을 AWS EC2에 배포하면 IP 주소(3.34.xx.xx)가 생긴다. 이 IP를 DNS 설정에서 도메인(api.myapp.com)과 연결(A 레코드 등록)하면, 이후 클라이언트는 도메인으로 접속할 수 있다.

2. 환경별 도메인 분리

api.myapp.com은 프로덕션 서버 IP, api-staging.myapp.com은 스테이징 서버 IP로 각각 등록하면, 코드 변경 없이 환경을 분리할 수 있다.

3. 서버 이전 시 도메인 유지

서버 IP가 바뀌어도 DNS 레코드만 새 IP로 업데이트하면, 클라이언트 코드 수정 없이 트래픽이 새 서버로 이동한다. 단, DNS 변경이 전 세계에 전파되는 데 시간이 걸린다(TTL 만료까지, 보통 수십 분~수 시간).

Git에서 브랜치 이름(main)이 특정 커밋 해시를 가리키듯, DNS 도메인(google.com)은 특정 IP를 가리킨다. 브랜치가 새 커밋으로 이동해도 이름은 그대로이듯, 서버 IP가 바뀌어도 도메인 이름은 그대로다. DNS 레코드 업데이트 = 브랜치 포인터 이동.

정리

  • DNS = 도메인 이름 → IP 주소를 매칭하는 분산 주소록 서비스
  • 필요한 이유: IP는 외우기 어렵고 변경될 수 있음 → 도메인으로 추상화
  • 동작: 브라우저 → DNS 질의 → IP 응답 → 그 IP로 HTTP 접속
  • 실무: 서버 배포 시 A 레코드 등록, 서버 이전 시 레코드 업데이트로 도메인 유지
  • DNS 변경은 전파에 시간이 걸린다 (TTL — Time To Live)
