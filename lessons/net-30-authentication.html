핵심 한 줄

클라이언트는 Authorization 헤더로 "나는 누구다"를 증명하고, 서버는 WWW-Authenticate 헤더로 "이렇게 증명해라"고 안내한다.

Authorization — 인증 정보 전달

클라이언트가 서버에 인증 자격증명(Credentials)을 전달하는 요청 헤더다. 인증 방식마다 값의 형식이 다르다. 

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Authorization: Basic dXNlcjpwYXNzd29yZA==
  • Bearer: JWT(JSON Web Token) 기반 인증에서 사용. 현대 API의 표준.
  • Basic: username:password를 Base64로 인코딩한 값. 오래된 방식이며 HTTPS 없이 쓰면 위험하다.
프론트에서 Axios나 fetch로 API를 호출할 때 매 요청마다 붙이는 바로 그 헤더다. axios.defaults.headers.common['Authorization'] = `Bearer ${token}`처럼 로그인 후 저장한 JWT를 모든 요청에 자동으로 첨부한 경험이 있을 것이다. 서버(Rails/Grape)는 이 헤더를 읽어 누가 요청했는지 확인한다. Grape API에서는 headers['Authorization']으로 접근하거나 Devise/JWT 미들웨어가 자동으로 파싱한다.

WWW-Authenticate — 인증 방법 안내

서버가 클라이언트에게 어떤 방식으로 인증해야 하는지를 알려주는 응답 헤더다. 반드시 401 Unauthorized 응답과 함께 전달된다. 

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="example", error="invalid_token"

이 헤더를 받은 클라이언트는 "아, Bearer 토큰 방식으로 인증해야 하는구나"를 알 수 있다. 브라우저는 이 헤더를 받으면 기본 인증 팝업을 띄우거나, JS 코드가 로그인 페이지로 리다이렉트하는 방식으로 처리한다.

인증 흐름 정리

클라이언트가 인증 없이 보호된 리소스 요청
↓
서버: 401 Unauthorized + WWW-Authenticate: Bearer realm="api"
↓
클라이언트가 로그인 후 토큰 획득
↓
재요청: Authorization: Bearer eyJhbG...
↓
서버: 200 OK + 요청한 리소스 반환

정리

  • Authorization: 요청 헤더 — 클라이언트가 인증 자격증명 전달 (Bearer, Basic 등)
  • WWW-Authenticate: 응답 헤더 — 401과 함께 인증 방법 안내
  • 인증 방식(Bearer/Basic 등)마다 Authorization 헤더 값의 형식이 다르다
