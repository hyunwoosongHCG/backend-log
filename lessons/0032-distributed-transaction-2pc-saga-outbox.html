레슨 24에서 남겨둔 질문

레슨 24에서 PR #1294를 보면서 이런 질문을 남겼었다: "트랜잭션 커밋 후에 카프카 이벤트를 발행하도록 순서를 바꿨는데, 그럼 커밋은 됐는데 발행 직전에 프로세스가 죽으면 어떻게 되는가?" 답은 "그 이벤트는 유실된다"이다. DB write와 메시지 발행은 서로 다른 두 시스템(Postgres, Kafka)에 대한 쓰기라서, 하나의 트랜잭션으로 묶을 수 없다 — 이걸 Dual Write Problem(이중 쓰기 문제)이라고 부른다.

이 문제, 그리고 더 넓게 "여러 서비스에 걸친 작업을 어떻게 안전하게 다룰 것인가"는 팀 시니어(juhoLee)와의 대화에서 이렇게 정리됐다: "2PC → Saga → Outbox 순서로 보시죠." 세 개는 경쟁 관계가 아니라 계보다 — 2PC의 한계를 Saga가 풀고, Saga를 실전에서 안전하게 쓰기 위한 배관이 Outbox다.

1. 2PC (Two-Phase Commit) — 가장 직관적이지만 MSA엔 안 맞는 방법

2PC는 여러 데이터베이스에 걸친 트랜잭션을 하나처럼 커밋하는 프로토콜이다. 코디네이터가 참여자 전원에게 "준비됐어?"(prepare)라고 묻고, 전원이 "그렇다"고 답해야만 "그럼 커밋해"(commit)라고 지시한다. 한 명이라도 실패를 답하면 전원 롤백한다.

① 코디네이터 → 모든 참여자: "커밋할 준비 됐어?" (Prepare)
↓ 전원 준비 완료 응답 시
② 코디네이터 → 모든 참여자: "그럼 커밋해" (Commit)
↓ 하나라도 실패하면
전원 롤백 (Abort)
프론트에서 여러 API를 동시에 호출하고 Promise.all로 "전부 성공해야 진행"하는 것과 비슷한 발상이다. 다만 2PC는 그보다 훨씬 엄격하다 — 응답을 기다리는 동안 참여자들이 자원(락)을 계속 붙잡고 있어야 하고, 코디네이터가 죽으면 참여자들이 영원히 "커밋해도 되나?" 상태로 멈춰버릴 수 있다.

이게 왜 MSA·카프카 환경에 안 맞는가: 2PC는 모든 참여자가 같은 프로토콜(보통 XA)을 지원해야 하는데, Kafka는 애초에 이런 분산 락 기반 프로토콜을 위해 설계되지 않았다. 그리고 서비스 하나가 느려지면 나머지 전부가 락을 붙잡은 채 대기해야 해서, MSA가 원하는 "서비스 간 느슨한 결합"과 정반대로 간다.

Q1. 2PC가 MSA·카프카 환경에서 잘 쓰이지 않는 가장 큰 이유는?

2. Saga — "묶어서 커밋" 대신 "실패하면 되돌리기"

Saga는 발상을 뒤집는다. 여러 서비스를 하나의 트랜잭션으로 묶으려 하지 않고, 각 서비스가 자기 몫의 로컬 트랜잭션만 책임지고, 끝나면 다음 단계를 트리거하는 이벤트를 발행한다. 중간 단계에서 실패하면, 이미 끝낸 이전 단계들을 보상 트랜잭션(compensating transaction)으로 하나씩 되돌린다 — DB 트랜잭션의 롤백처럼 자동은 아니고, "취소 로직"을 직접 짜야 한다.

① HR 서비스: 조직 변경 로컬 트랜잭션 커밋 → "조직 변경됨" 이벤트 발행
↓
② Performance(theplus-back): 이벤트 소비 → 조직 캐시 갱신 로컬 트랜잭션 커밋
↓ 만약 ②가 실패하면
보상 이벤트 발행 → HR 서비스가 ①의 변경을 되돌리는 보상 트랜잭션 실행

이 프로젝트의 실제 카프카 토픽 구조 — optimiz-async로 조직/발령 이벤트가 오고, theplus-back이 처리 후 user-performance-module-status 같은 이벤트를 다시 발행하는 구조 — 는 이미 Saga의 "각자 로컬 트랜잭션 + 이벤트로 다음 단계 트리거" 모양과 닮아 있다. 다만 지금은 실패했을 때 자동으로 되돌리는 보상 로직은 없다 — 실패하면 DLQ로 가서 사람이 확인하는 수준이다. "Saga를 도입했다"고 부르려면 이 보상 흐름까지 설계돼 있어야 한다.

Q2. Saga 패턴에서 중간 단계가 실패했을 때 이전 단계를 되돌리는 방법은?

3. Transactional Outbox — Saga를 안전하게 발행하는 배관

Saga가 굴러가려면 "로컬 트랜잭션 커밋 + 이벤트 발행"이 안전하게 짝지어져야 하는데, 이게 바로 레슨 24에서 남긴 Dual Write Problem이다. Outbox 패턴은 이렇게 푼다:

① 비즈니스 데이터 저장 + 이벤트를 outbox 테이블에 저장 — 같은 로컬 트랜잭션 안에서
↓ 트랜잭션 커밋 (둘 다 성공하거나 둘 다 실패)
② 별도 프로세스(relay/poller)가 outbox 테이블을 감시
↓ 새 행 발견 시
③ 카프카에 발행 → 성공하면 outbox 행에 "발행 완료" 표시

핵심은 "이벤트를 발행하는 행위"를 카프카에 직접 하지 않고, 같은 DB 트랜잭션으로 보장되는 테이블에 대신 적어두는 것이다. 비즈니스 데이터가 커밋됐다면 outbox 행도 반드시 함께 커밋돼 있다 — 그러면 relay 프로세스가 언제 도든 그 행을 읽어서 "언젠가는" 카프카에 발행할 수 있다. 프로세스가 발행 도중 죽어도, outbox 테이블에 기록이 남아있으니 재시작하면 다시 시도하면 된다 — 이벤트가 유실되지 않는다(단, 중복 발행 가능성은 남아서, 수신 측은 레슨 31의 멱등성이 결국 다시 필요하다).

프론트의 낙관적 업데이트(optimistic update) + 재시도 큐와 비슷하다. 서버에 저장은 로컬 스토리지에 "동기화 대기 중"으로 먼저 기록해두고, 별도 백그라운드 로직이 그 대기열을 훑으며 실제 서버 전송을 재시도하는 구조 — 전송 자체가 사용자 액션과 같은 순간에, 같은 성공/실패로 묶이지 않아도 되게 만든다.
Q3. Outbox 패턴이 Dual Write Problem을 해결하는 핵심 아이디어는?

지금 이 프로젝트는 어디쯤 와 있나

레이어 상태
서비스 내부 로컬 트랜잭션 PR #1294에서 개선 완료 — 저장 트랜잭션 커밋 후 이벤트 발행, 뱅 메서드로 원자성 확보
Dual Write 안전성 (Outbox) 아직 없음 — 커밋 후 발행 사이에 죽으면 이벤트 유실 가능. 다음 개선 후보
분산 트랜잭션 전략 (Saga) 이벤트 체인은 있지만, 실패 시 자동 보상 트랜잭션은 없음 — juhoLee가 짚은 "아직 논의가 필요한 부분"

"MSA인데 분산 트랜잭션 전략이 없어서 한 것"이라는 juhoLee의 말이 이 표의 3번째 줄이다. PR #1294는 1번째 줄(로컬 원자성)을 확실히 고쳤고, 2·3번째 줄은 여전히 열린 과제로 남아있다 — "다 고쳤다"가 아니라 "제일 급한 층부터 고쳤다"로 이해하는 게 정확하다.
