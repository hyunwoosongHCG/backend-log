Entity라는 단어가 두 가지 의미로 쓰인다

레슨 002 Grape::Entity · Model · 직렬화 ~10분
🎯 미션 연결: 코드를 보고 각 레이어가 무슨 역할인지 구분할 수 있다 — MISSION.md

혼란의 출발점

백엔드를 공부하다 보면 "Entity"라는 단어를 두 곳에서 만나게 된다. 같은 단어인데 전혀 다른 것을 가리킨다. 구분하지 못하면 코드를 읽을 때마다 헷갈린다.

의미 1 — Java/Spring에서 말하는 Entity

MVC나 3레이어 아키텍처를 Java 기준으로 배우면 이런 코드가 나온다: 

@Entity          // ← "이 클래스는 DB 테이블과 연결됐다"는 선언
@Table(name = "users")
public class User {
    @Id
    private Long id;
    private String name;
}

여기서 Entity는 DB 테이블과 1:1로 매핑되는 클래스다. users 테이블이 있으면 User Entity가 있다. DB에 값을 저장하고 꺼내는 주체다.

의미 2 — 이 프로젝트의 Grape::Entity

이 프로젝트에서 Entity라고 부르는 건 완전히 다르다: 

class Entities::Workspaces::OneOnOneSettingEntity < Grape::Entity
  expose :use_required_template
  expose :use_privacy_comment
  expose :use_inactive_alert
end

DB와 무관하다. 모델 객체를 받아서 응답 JSON에 어떤 필드를 내보낼지 선별하는 도구다. expose에 적힌 필드만 JSON 응답에 포함되고, 나머지는 숨겨진다.

TypeScript로 치면 API 응답 타입을 정의하고 필요한 필드만 골라 리턴하는 함수와 같다: 
function toResponse(record: OneOnOneSetting) {
  return {
    use_required_template: record.use_required_template,
    use_privacy_comment: record.use_privacy_comment,
    // 내부 전용 필드는 여기 안 적으면 노출 안 됨
  }
}
expose 한 줄 = 응답에 포함할 필드 한 개.

Rails에서는 역할이 어떻게 나뉘나

Java/Spring 용어 Rails + Grape 하는 일
Entity (@Entity) Model (ActiveRecord) DB 테이블 매핑, 데이터 저장/조회
DTO / ResponseBody Entity (Grape::Entity) 응답 JSON 직렬화, 노출 필드 선별

Rails에서 DB와 연결된 것은 Model이다. Grape의 Entity는 이름이 같을 뿐, DB를 전혀 모른다.

왜 Entity라는 이름을 썼을까

Grape 라이브러리가 자신의 직렬화 도구에 "Entity"라는 이름을 붙였기 때문이다. "표현 가능한 단위"라는 의미로 썼지만, Java 생태계에서 Entity가 다른 뜻으로 굳어져 있어서 혼란이 생긴다. 프레임워크마다 같은 단어를 다르게 쓰는 일은 백엔드에서 자주 있다.

이 프로젝트 구조 다시 보기

Grape Endpoint (Controller) — 요청 받기, 인증, present 호출
↓
ActiveRecord Model — DB에서 데이터 꺼내기 (Java의 Entity 역할)
↓
Grape::Entity — 응답 JSON으로 변환 (Java의 DTO 역할)
↓
{ "data": { "use_required_template": false, … } }
