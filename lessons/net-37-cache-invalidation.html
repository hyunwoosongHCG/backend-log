캐시 무효화

🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 캐시 — 절대 캐시되면 안 되는 자원을 확실히 막는 방법을 배운다

핵심 한 줄

통장 잔고처럼 반드시 최신값이어야 하는 응답에는 Cache-Control: no-cache, no-store, must-revalidate와 Pragma: no-cache를 조합해야 오래된 데이터가 절대 표시되지 않는다.

왜 캐시 무효화가 필요한가

지금까지 배운 캐시는 "자원을 저장해두고 재사용해 빠르게"가 목적이었다. 그런데 캐시가 있으면 안 되는 자원도 있다.

  • 통장 잔고 — 1초 전 금액이 표시되면 큰일
  • 로그인 세션 정보 — 로그아웃 후에도 이전 응답이 남으면 보안 문제
  • 재고 수량, 주문 상태 — 항상 실시간 값이어야 함

이런 자원에 아무 캐시 헤더도 안 붙이면 될까? 브라우저나 프록시 캐시가 임의로 캐시하는 경우가 있어서 그것만으로는 부족하다. 명시적으로 "캐시하지 말라"고 알려줘야 한다.

캐시 무효화 완전 처방

Cache-Control: no-cache, no-store, must-revalidate
Pragma: no-cache

네 가지를 모두 쓰는 이유를 하나씩 살펴본다.

no-store — 저장 자체를 금지

응답을 캐시 저장소에 절대 저장하지 말라는 지시다. 메모리 캐시, 디스크 캐시 모두 포함된다. 이것이 가장 강력한 지시어지만, 단독으로는 하위 호환이 불완전하다.

no-cache — 저장돼 있어도 반드시 검증

만약 어떤 이유로 캐시가 저장됐더라도, 사용 전 반드시 Origin 서버에 검증하라는 지시다. no-store와 조합하면 저장도 안 되고, 혹시 저장됐더라도 못 쓴다.

must-revalidate — Origin 장애 시에도 오래된 캐시 차단

이것이 핵심이다. no-cache만 있을 때 Origin 서버가 일시 장애라면 프록시가 오래된 캐시를 200 OK로 내보낼 수 있다. must-revalidate는 그 경우 반드시 504 Gateway Timeout을 강제한다.

클라이언트: 잔고 조회 요청
↓
프록시: Origin 서버에 검증 시도 → 하지만 Origin 장애!
↓
no-cache만일 때: 프록시가 오래된 캐시를 200으로 반환 → 잘못된 잔고 표시
↓ 또는
must-revalidate 있을 때: 504 Gateway Timeout → 오류는 나지만 오래된 값은 절대 표시 안 됨

Pragma: no-cache — HTTP 1.0 하위 호환

Cache-Control은 HTTP 1.1에서 도입됐다. HTTP 1.0만 이해하는 구형 프록시 서버는 이 헤더를 무시할 수 있다. Pragma: no-cache는 HTTP 1.0 시대의 캐시 금지 지시어다. 현대 환경에서 사실상 거의 필요 없지만, 완전한 무효화를 위해 함께 보내는 것이 표준 관행이다.

캐시 전략 전체 맵

자원 종류 권장 Cache-Control 이유
정적 파일 (해시 파일명) public, max-age=31536000, immutable 파일명이 바뀌면 URL이 바뀌므로 무효화는 URL이 담당
index.html (SPA 진입점) no-cache 항상 최신 JS·CSS 파일 목록을 가져와야 함
API 응답 (공개 데이터) public, max-age=60 수십 초 단위 캐시 허용
API 응답 (개인 데이터) private, max-age=60 프록시에 저장 금지
민감 데이터 (잔고, 세션 등) no-cache, no-store, must-revalidate 절대 캐시 금지
Rails에서 민감한 API 응답에 캐시를 막으려면 컨트롤러에서 이렇게 쓴다. 
# app/controllers/accounts_controller.rb
def balance
  response.headers['Cache-Control'] = 'no-cache, no-store, must-revalidate'
  response.headers['Pragma'] = 'no-cache'
  render json: { balance: current_user.balance }
end
반대로 정적 자원에는 public/assets/에서 자동으로 긴 max-age가 붙는다. Rails의 config.public_file_server.headers에서 이 동작을 제어할 수 있다. 프론트에서 fetch API로 API를 호출할 때 Network 탭에서 응답 헤더의 Cache-Control 값을 확인하면 서버가 어떤 캐시 전략을 쓰는지 바로 알 수 있다.

정리

  • no-store: 캐시 저장 자체 금지
  • no-cache: 저장은 하더라도 사용 전 반드시 Origin 검증
  • must-revalidate: Origin 장애 시에도 오래된 캐시 반환 금지, 504 강제
  • Pragma: no-cache: HTTP 1.0 구형 프록시 하위 호환
  • 민감 데이터에는 네 가지 조합을 함께 써야 완전한 무효화가 보장된다
