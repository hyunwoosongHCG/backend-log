캐시와 조건부 요청 헤더

🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 캐시 — 캐시를 제어하는 Cache-Control 지시어들을 한데 모아 정리한다

핵심 한 줄

캐시 관련 헤더는 크게 캐시 지시어(Cache-Control)와 조건부 요청 헤더 쌍(검증·조건) 두 종류로 나뉜다. 각 지시어가 무엇을 제어하는지 명확히 알아야 캐시 전략을 설계할 수 있다.

Cache-Control 지시어 총정리

캐시 수명 제어

지시어 위치 의미
max-age=N 응답 캐시 유효 시간(초). 그 안에는 서버에 묻지 않고 캐시 사용.
s-maxage=N 응답 프록시·CDN 같은 공유 캐시 서버의 유효 시간. max-age보다 우선 적용.
max-stale=N 요청 만료된 캐시라도 N초 이내라면 사용해도 된다. (클라가 허용 범위를 넓힘)
min-fresh=N 요청 유효기간이 N초 이상 남아 있는 캐시만 사용하겠다.

캐시 저장·검증 정책

지시어 의미
no-cache 캐시에 저장은 하지만, 사용 전 반드시 Origin 서버에 검증을 받아야 한다. (프록시 캐시가 검증을 생략하고 오래된 자원을 200으로 줄 수 있는 것을 막기 위해)
no-store 캐시 저장 자체를 금지한다. 민감한 응답(토큰, 잔고 등)에 사용.
must-revalidate 캐시 만료 후 사용 전에 반드시 Origin 서버에 검증해야 한다. Origin 서버에 접근할 수 없으면 504 Gateway Timeout을 반환해야 한다.
proxy-revalidate must-revalidate와 같지만, 프록시 캐시에만 적용된다.

캐시 공개 범위

지시어 의미
public 응답을 프록시·CDN 같은 공유 캐시 서버에도 저장해도 된다.
private 해당 사용자의 브라우저 캐시에만 저장. 공유 캐시(프록시)에는 저장 금지. (개인 데이터 보호)

no-cache vs must-revalidate — 왜 같이 쓰나?

이 둘의 차이는 프록시 캐시가 Origin 서버에 접근하지 못하는 상황에서 드러난다.

상황 no-cache만 적용 시 must-revalidate 적용 시
Origin 서버 정상 서버에 검증 후 사용 (동일) 서버에 검증 후 사용 (동일)
Origin 서버 일시 장애 프록시가 오래된 캐시를 200 OK로 반환할 수 있다 반드시 504 Gateway Timeout 반환

캐시 무효화(강의 40)에서 두 지시어를 함께 쓰는 이유가 바로 이것이다. 통장 잔고처럼 절대 오래된 값이 나오면 안 되는 경우에는 no-cache만으로는 부족하고 must-revalidate도 같이 써야 한다.

조건부 요청 헤더 쌍 요약

검증 헤더 (응답) 조건부 요청 헤더 (재요청) 비교 방식
Last-Modified If-Modified-Since 날짜 비교 — 수정 시각이 이후면 200, 아니면 304
ETag If-None-Match 태그 비교 — 다르면 200, 같으면 304

둘 다 있을 때는 ETag / If-None-Match가 우선 적용된다.

React 앱 빌드 전략과 대응시키면 이렇다. 정적 파일(JS·CSS·이미지)엔 Cache-Control: public, max-age=31536000, immutable — 1년 캐시, 내용이 바뀌면 파일명 해시가 바뀌므로 무효화는 파일명이 담당한다. index.html엔 Cache-Control: no-cache — 항상 Origin에서 최신 버전을 확인해야 최신 해시 파일명 목록을 알 수 있기 때문이다. Rails도 public/assets/ 아래 컴파일된 파일에 동일한 전략을 쓴다.

정리

  • max-age: 캐시 유효 시간(초)
  • no-cache: 저장은 하되 사용 전 매번 Origin 서버 검증 필수
  • no-store: 캐시 저장 자체 금지
  • must-revalidate: 만료 후 Origin 검증 필수, 실패 시 504 강제
  • public / private: 공유 캐시 허용 여부
  • 조건부 요청 헤더 쌍: Last-Modified ↔ If-Modified-Since, ETag ↔ If-None-Match
