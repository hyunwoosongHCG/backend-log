핵심 한 줄

Host, Location, Allow, Retry-After는 "어느 도메인으로 가야 하나", "리다이렉트는 어디로", "허용 메서드는 무엇", "언제 다시 시도하나"처럼 특수 상황을 처리하기 위한 헤더다.

Host — 요청한 호스트(도메인) 헤더

클라이언트가 요청을 보낼 때 어떤 도메인을 대상으로 하는지를 명시하는 헤더다. HTTP/1.1부터 필수(required) 헤더로 지정됐다. 

GET /search HTTP/1.1
Host: www.google.com

왜 필수인가 — 가상 호스팅(Virtual Hosting)

현대 서버 인프라에서는 하나의 IP 주소에 여러 도메인을 올리는 것이 일반적이다. 예를 들어 같은 IP(192.168.0.1)에 aaa.com과 bbb.com이 함께 있을 수 있다.

클라이언트가 aaa.com 접속 시도
↓
DNS 조회 → IP 192.168.0.1 확인
↓
TCP 연결 → HTTP 요청 전송
↓
서버는 Host: aaa.com 헤더를 보고 어느 사이트로 라우팅할지 결정

Host 헤더가 없으면 서버는 이 요청이 aaa.com을 원하는지 bbb.com을 원하는지 알 수 없다. 그래서 필수 헤더다.

React 앱을 Vercel에 배포할 때 같은 Vercel 서버에 수천 개의 프로젝트가 올라가 있지만, 각 도메인으로 접속하면 자신의 프로젝트만 보인다. Vercel이 Host 헤더를 읽어 어느 프로젝트를 서빙할지 분기하기 때문이다. Rails에서도 request.host로 이 값을 읽을 수 있다.

Location — 리다이렉션 주소

응답 헤더로 사용하며, 두 가지 상황에서 쓰인다.

  • 3xx 리다이렉션: 브라우저가 이 헤더의 URL로 자동 이동한다. 
    HTTP/1.1 301 Moved Permanently
Location: https://www.new-site.com/
  • 201 Created: 새로 생성된 리소스의 URI를 알려준다. 
    HTTP/1.1 201 Created
Location: /posts/42
Rails에서 리소스를 생성한 뒤 상세 페이지로 이동시키는 패턴이다. redirect_to @post를 쓰면 Rails가 자동으로 Location: /posts/42 헤더를 담아 302 응답을 보내고, 브라우저가 그 주소로 이동한다. 프론트에서 router.push('/posts/42')를 호출하는 것과 결과는 같지만, 서버가 브라우저에게 "여기로 가라"고 지시하는 방식이다.

Allow — 허용 HTTP 메서드

서버가 허용하는 HTTP 메서드 목록을 응답 헤더로 전달한다. 405 Method Not Allowed 응답과 함께 사용한다. 

HTTP/1.1 405 Method Not Allowed
Allow: GET, POST, DELETE

클라이언트가 PUT /posts/1을 요청했는데 서버가 PUT을 지원하지 않는다면, 405와 함께 위 헤더를 보내 "이 엔드포인트는 GET, POST, DELETE만 된다"고 알려준다.

Retry-After — 다음 요청까지 대기 시간

서버가 잠시 사용 불가(503)거나 요청이 너무 많은(429) 경우, 얼마 후에 다시 시도하면 되는지를 알려주는 응답 헤더다. 날짜 형식과 초(second) 형식 두 가지를 쓸 수 있다. 

HTTP/1.1 503 Service Unavailable
Retry-After: Fri, 31 Dec 1999 23:59:59 GMT   ← 날짜 형식

HTTP/1.1 503 Service Unavailable
Retry-After: 120                               ← 120초 후에 재시도

이론상 있는 헤더지만, 실무에서는 사용이 드물다. 서버 복구 시점을 정확히 예측하기 어렵고, 잘못된 값을 주면 클라이언트가 너무 이르거나 너무 늦게 재시도하기 때문이다.

정리

  • Host: 요청 헤더, 필수 — 어느 도메인 대상인지 명시 (가상 호스팅 핵심)
  • Location: 응답 헤더 — 3xx 리다이렉트 주소 또는 201 Created 리소스 URI
  • Allow: 응답 헤더 — 405와 함께 허용 메서드 목록 전달
  • Retry-After: 응답 헤더 — 503/429와 함께 재시도 가능 시점 안내 (실무 사용 드묾)
