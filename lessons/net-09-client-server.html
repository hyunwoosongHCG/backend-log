핵심 한 줄

클라이언트는 요청(Request)만, 서버는 응답(Response)만 담당한다. 둘을 명확히 분리함으로써 각자 자신의 역할에 집중하고, 각자 독립적으로 발전할 수 있다.

클라이언트와 서버 — 역할 분리

HTTP의 가장 기본적인 특징은 클라이언트-서버 구조(Client-Server Architecture)다.

  • 클라이언트(Client): 요청을 보내는 쪽. 브라우저, 모바일 앱, 다른 서버 등
  • 서버(Server): 요청을 받아 처리하고 응답을 돌려주는 쪽
클라이언트 — "이 데이터 줘" (Request)
↓
서버 — 요청을 처리하고 결과를 만든다
↓
클라이언트 — "200 OK + 데이터" (Response)

중요한 점은 서버가 먼저 클라이언트에게 데이터를 보내지 않는다는 것이다. 클라이언트가 요청을 보내야만 서버가 응답한다. (WebSocket이나 SSE는 이 규칙에서 벗어난 특수한 경우다.)

왜 분리가 좋은가?

클라이언트와 서버를 분리하지 않던 시절, 서버가 화면 렌더링·비즈니스 로직·데이터 저장을 모두 담당했다. 문제 하나가 생기면 모든 것이 얽혀 있었다.

분리하면 각자의 역할이 명확해진다.

역할 클라이언트가 집중할 것 서버가 집중할 것
UI/UX 화면 렌더링, 사용자 인터랙션 관심 없음
데이터 받은 데이터를 표시 데이터 저장·조회·비즈니스 로직
장애 클라이언트 이슈는 클라이언트에서 해결 서버 이슈는 서버에서 해결
발전 서버 내부 몰라도 UI 독립 개발 가능 클라이언트 몰라도 API 독립 개발 가능
지금 하는 프론트엔드 작업이 바로 이 구조다. React 앱(클라이언트)과 Rails 서버(서버)는 별도 레포, 별도 배포로 완전히 분리돼 있다. 프론트는 서버 DB 구조를 몰라도 되고, 서버는 React 컴포넌트를 몰라도 된다. 둘의 계약은 오직 "어떤 URL에 어떤 데이터를 요청하면 어떤 JSON이 온다"는 API 명세뿐이다.

정리

  • HTTP = 클라이언트가 요청, 서버가 응답하는 구조
  • 역할 분리 → 각자 독립적으로 개발·배포·확장 가능
  • 장애가 생겨도 해당 역할(클라이언트 or 서버)에서만 원인 파악
  • 프론트·백 분리 개발이 당연해진 이유가 바로 이 구조
