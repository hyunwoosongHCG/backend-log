🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 헤더 - 캐시 — 왜 캐시가 필요한지, 어떻게 동작하는지 원리부터 잡는다

핵심 한 줄

캐시(Cache)란 서버에서 받은 자원을 브라우저에 저장해두고, 유효 시간(max-age) 안에는 서버에 다시 요청하지 않고 저장된 자원을 바로 꺼내 쓰는 것이다.

캐시가 왜 필요한가

이미지·영상·JS 번들 같은 정적 자원은 용량이 크고, 자주 바뀌지 않는다. 그런데 캐시가 없다면 페이지를 열 때마다 같은 파일을 서버에서 처음부터 다시 받아야 한다.

  • 네트워크 비용 낭비 — 이미 받은 파일을 또 받는다
  • 응답 지연 — 서버 왕복 시간만큼 느려진다
  • 서버 부하 — 클라이언트가 늘수록 동일한 파일 요청이 폭증한다

캐시는 이 문제를 해결한다. 서버가 응답 헤더에 Cache-Control: max-age=60을 담아 보내면, 브라우저는 "60초 동안은 다시 요청하지 않아도 된다"는 뜻으로 해석하고 자원을 로컬 캐시 저장소에 저장한다.

캐시 동작 흐름

최초 요청 — 캐시 없음

브라우저 → 서버: GET /logo.png
↓
서버 → 브라우저: 200 OK + 바디(이미지) + Cache-Control: max-age=60
↓
브라우저: 응답을 캐시에 저장, 유효기간(60초) 함께 기록
↓
화면에 이미지 표시

두 번째 요청 — 캐시 유효 시간 이내

브라우저: "캐시에 있고 유효기간이 남았네" → 서버 요청 없이 캐시 자원 사용
↓
화면에 이미지 표시 (네트워크 0바이트 소비)

세 번째 요청 — 캐시 만료 이후

브라우저: "캐시가 있지만 유효기간이 지났다" → 서버에 재요청
↓
서버: 자원을 다시 전송 + 새 Cache-Control: max-age=60
↓
브라우저: 캐시 갱신, 새 유효기간으로 다시 저장

Cache-Control 헤더

캐시의 핵심은 서버가 응답할 때 담는 Cache-Control 헤더다.

지시어 의미
max-age=N N초 동안 캐시 유효. 그 안에는 서버 요청 없이 캐시 자원을 사용한다.
no-cache 캐시에 저장은 하지만, 사용 전 매번 서버에 검증을 받아야 한다.
no-store 캐시 자체를 저장하지 않는다. 민감한 데이터에 사용.

이 강의에서는 max-age만 다룬다. no-cache와 no-store의 정확한 차이는 강의 38·40에서 자세히 본다.

프론트에서 vite build를 하면 main.a3f9c2.js처럼 파일명에 해시가 붙는다. 이건 캐시를 최대한 활용하기 위한 전략이다 — 내용이 같으면 파일명도 같아서 캐시 히트가 되고, 내용이 바뀌면 파일명도 바뀌어서 브라우저가 새 파일을 받는다. Cache-Control: max-age=31536000(1년)을 걸어두고 파일명 해시로 무효화를 제어하는 것이 정적 자원 캐시의 표준 패턴이다.

브라우저 개발자 도구에서 확인하기

Chrome DevTools → Network 탭을 열고 페이지를 새로고침하면 캐시 동작을 직접 볼 수 있다.

  • Status: 200 — 서버에서 새로 받은 응답
  • Status: 200 (from disk cache) — 로컬 디스크 캐시에서 꺼낸 것
  • Status: 200 (from memory cache) — 메모리 캐시에서 꺼낸 것 (더 빠름)
  • Size: 0 B 혹은 (disk cache) — 네트워크를 전혀 쓰지 않았다는 의미

정리

  • 캐시는 서버 자원을 브라우저에 저장해두고 유효 시간 안에는 재사용한다
  • 서버는 응답 헤더 Cache-Control: max-age=N으로 유효 시간을 알려준다
  • 유효 시간이 지나면 브라우저는 서버에 재요청한다
  • 재요청 때 자원이 그대로라면 다시 받는 것은 낭비 → 다음 강의에서 검증으로 해결
