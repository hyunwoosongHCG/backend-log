클라이언트에서 서버로 데이터 전송

🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 메서드 활용 — 클라이언트가 서버에 데이터를 전달하는 두 가지 방식과 실무 전송 유형 4가지를 파악한다

핵심 한 줄

클라이언트가 서버에 데이터를 보내는 방법은 크게 두 가지다 — URL 쿼리 파라미터(Query Parameter)로 보내거나, HTTP 메시지 바디(Message Body)에 담아 보내거나. 언제 어느 방법을 쓰는지는 "조회냐, 변경이냐"로 갈린다.

데이터 전달 방식 2가지

1. 쿼리 파라미터(Query Parameter)를 통한 전송

URL 끝에 ?key=value 형태로 데이터를 붙여 보내는 방식이다. GET 요청에서 주로 사용한다. 메시지 바디 없이 URL 자체에 데이터가 드러나기 때문에, 검색어·정렬 조건·필터처럼 "어떤 데이터를 달라"는 조회 조건을 전달할 때 적합하다. 

GET /search?keyword=ruby&sort=latest&page=2 HTTP/1.1
Host: api.example.com
  • URL에 데이터가 그대로 노출된다 → 민감 정보(비밀번호 등)는 절대 사용 금지
  • 북마크·공유가 가능하다 → 검색 결과 URL을 그대로 공유할 수 있는 이유
  • 브라우저 히스토리·서버 로그에 남는다

2. 메시지 바디(Message Body)를 통한 전송

HTTP 요청의 본문(body)에 데이터를 담아 보내는 방식이다. POST, PUT, PATCH 요청에서 사용한다. URL에 데이터가 드러나지 않으며, 대용량 데이터와 민감 정보 전송에 적합하다. 회원 가입, 게시글 등록, 주문 처리처럼 "서버에 리소스를 등록하거나 수정"할 때 쓴다. 

POST /members HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "username": "hyunwoo",
  "email": "hw@example.com",
  "password": "secret123"
}
useSearchParams()로 URL 쿼리를 읽고 쓰는 것이 바로 쿼리 파라미터 방식이다. 검색·필터 UI 상태를 URL에 반영할 때 ?keyword=ruby&sort=latest처럼 붙이는 그 값들. 반면 fetch('/api/members', { method: 'POST', body: JSON.stringify(form) })처럼 body에 담아 보내는 것이 메시지 바디 방식이다. 비밀번호가 URL에 뜨면 안 되는 이유가 바로 여기 있다.

실무 전송 유형 4가지

두 가지 방식이 실제로 어떤 시나리오로 쓰이는지 4가지 유형으로 정리한다.

유형 1. 정적 데이터 조회 (Static Data)

이미지·HTML·CSS·JS 파일처럼 쿼리 파라미터 없이 경로만으로 리소스를 가져오는 경우다. 데이터를 따로 전달할 필요가 없다. 

GET /images/logo.png HTTP/1.1

유형 2. 동적 데이터 조회 (Dynamic Data, 쿼리 파라미터)

검색, 정렬, 필터처럼 조건에 따라 결과가 달라지는 조회다. 쿼리 파라미터로 조건을 전달한다. 

GET /articles?category=backend&sort=newest&page=3 HTTP/1.1

유형 3. HTML Form 전송

브라우저의 <form> 태그로 데이터를 전송하는 방식이다. HTML Form은 GET과 POST만 지원한다 — PUT, PATCH, DELETE는 form으로 직접 전송 불가.

method Content-Type 바디 형태 용도
GET — 쿼리 파라미터로 변환 조회 조건 전달
POST application/x-www-form-urlencoded key=value&key2=value2 텍스트 데이터 전송
POST multipart/form-data 파일 + 텍스트 혼합 파일 업로드 
<!-- application/x-www-form-urlencoded -->
<form action="/login" method="POST">
  <input name="email" type="email">
  <input name="password" type="password">
  <button type="submit">로그인</button>
</form>

<!-- multipart/form-data (파일 포함 시 enctype 필수) -->
<form action="/upload" method="POST" enctype="multipart/form-data">
  <input name="file" type="file">
  <input name="description" type="text">
  <button type="submit">업로드</button>
</form>

유형 4. HTTP API 전송 (JSON)

현대 웹 개발에서 가장 흔한 방식이다. 서버-서버 통신, 앱(iOS/Android)-서버, JavaScript AJAX(fetch/axios) 모두 이 유형에 해당한다. Content-Type: application/json으로 JSON 데이터를 바디에 담아 보낸다. GET, POST, PUT, PATCH, DELETE 모두 사용 가능하다. 

// 프론트엔드에서 하던 바로 그 방식
const response = await fetch('/api/members', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    username: 'hyunwoo',
    email: 'hw@example.com'
  })
});
const data = await response.json();

Rails + Grape API 조합에서 클라이언트가 보내는 방식이 바로 이것이다. Grape 컨트롤러에서 params[:username]으로 받는 그 값이 클라이언트가 바디에 담아 보낸 JSON이다.

요약 비교표

전달 방식 HTTP 메서드 데이터 위치 주요 용도
쿼리 파라미터 GET (주로) URL 뒤 검색·필터·정렬 조건
메시지 바디 (form-urlencoded) POST 바디 HTML Form 텍스트 전송
메시지 바디 (multipart) POST 바디 HTML Form 파일 업로드
메시지 바디 (JSON) POST/PUT/PATCH 바디 API 통신, AJAX, 앱-서버

정리

  • 조회 조건은 쿼리 파라미터(Query Parameter) — URL에 드러나도 되는 값만
  • 등록·수정·민감 정보는 메시지 바디(Message Body) — POST/PUT/PATCH
  • HTML Form은 GET과 POST만 지원 → 나머지 메서드는 API 호출로
  • 실무 API 통신의 표준은 application/json + 메시지 바디
  • Content-Type 헤더가 서버에게 "바디를 어떻게 해석할지" 알려주는 열쇠다
