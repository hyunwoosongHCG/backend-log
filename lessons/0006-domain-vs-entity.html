🎯 미션 연결: 코드에서 비즈니스 로직이 어디에 있어야 하는지 판단할 수 있다 — MISSION.md

질문의 출발점

선배 개발자가 이런 코드를 보여줬다. Service 메서드가 딱 세 줄인데, 삭제 로직은 어디 있을까? 

// HrAppointmentWorkDeleteCommandService.java

public void execute(AppointmentWorkDeleteCommand command) {
    HrAppointmentWork appointmentWork = repository.findById(command.getAppointmentWorkId())
        .orElseThrow(() -> new NotFound("조회된 발령 작업이 없습니다."));

    appointmentWork.delete(hrEmpManager.getLoginId(), hrEmpManager.getClientId());

    hrWorkLogManager.saveWorkLogs(command.getAppointmentWorkId(), WorkType.APPOINTMENT, WorkLogType.DELETED);
}

appointmentWork.delete(...)가 한 줄인데 실제 삭제 로직은 도메인 객체 안에 있다. Service는 "누구한테 시킬지"만 알고, 세부 사항은 모른다. 이것이 도메인(Domain)의 핵심이다.

3레이어 아키텍처에서 Entity는 빈 그릇

전통적인 3레이어(Controller → Service → Repository) 방식에서 Entity는 이렇게 생겼다:

@Entity
public class AppointmentWork {
    private Long id;
    private String status;
    private LocalDateTime deletedAt;

    // getter, setter만 있음 — 비즈니스 로직 없음
    public Long getId() { return id; }
    public void setStatus(String status) { this.status = status; }
}
React로 치면 type AppointmentWork = { id: number; status: string; deletedAt: Date | null } — 데이터 타입 정의만 있고 동작은 없는 것과 같다. 컴포넌트가 아니라 props 타입이다.

이 경우 "삭제 가능한지 검증"과 "자식 데이터 삭제" 로직은 전부 Service에 들어간다. Service가 비대해지고, 비슷한 검증 로직이 여러 Service에 흩어지기 쉽다.

DDD에서 Entity는 자기 책임을 가진 도메인 객체

위 코드에서 실제 delete 메서드를 보면:

// HrAppointmentWork.java (Domain Entity)

public void delete(String modUserId, String clientId) {
    HrAppointmentWorkDeletionPolicy.validate(this); // ① 삭제 가능한지 스스로 검증

    ZonedDateTime now = ZonedDateTime.now();

    this.appointmentWorkDetails.forEach(          // ② 자식 데이터도 스스로 정리
        detail -> detail.deleteByParent(modUserId, clientId, now));
    this.appointmentWorkTargets.forEach(
        target -> target.deleteByParent(modUserId, clientId, now));
    this.appointmentWorkTypes.forEach(
        type -> type.deleteByParent(modUserId, clientId, now));

    this.modUserId = modUserId;                   // ③ 자신의 필드를 직접 변경
    this.clientId = clientId;
    this.deletedAt = now;                         // soft delete
}

Entity 자신이 검증하고, 자식 데이터를 정리하고, 자신의 상태를 바꾼다. ORM(JPA)이 이 변경을 감지해서 자동으로 DB 쿼리를 날린다. Service는 이 세부 과정을 전혀 모른다.

React 컴포넌트가 자기 state와 이벤트 핸들러를 직접 가지는 것과 같다. 부모 컴포넌트는 <AppointmentWork onDelete={...} />만 호출하면 되고, 내부가 어떻게 동작하는지 알 필요 없다.

비교표

3레이어 Entity DDD 도메인 Entity
데이터 보유 O O
비즈니스 로직 X (Service에 있음) O (스스로 가짐)
Service 역할 두껍다 — 로직 집중 얇다 — 조율만
비유 데이터 타입 정의 스스로 동작하는 객체

왜 DDD 방식이 더 나을 수 있나

선배 개발자가 한 말: "한 곳에 책임을 몰빵하면 안 좋다. 거기 문제 생기면 어디가 문제인지 다 확인해야 한다."

삭제 검증 로직이 Service에만 있으면, 다른 Service에서 같은 Entity를 삭제할 때 검증을 빠뜨릴 수 있다. 도메인 객체가 스스로 검증하면 어디서 호출하든 규칙이 지켜진다.

Rails에서는?

Rails의 ActiveRecord Model은 이미 DDD 도메인 엔티티에 가깝다:

# Rails Model — 자기 검증과 콜백을 직접 가짐
class OneOnOneSetting < ApplicationRecord
  validates :workspace_id, presence: true     # 스스로 검증

  before_destroy :check_can_delete            # 삭제 전 규칙을 직접 처리

  def deactivate!
    update!(active: false)                    # 상태 변경 로직을 직접 가짐
  end

  private

  def check_can_delete
    throw :abort if meetings.exists?          # 자식 데이터 있으면 삭제 거부
  end
end

Java 3레이어에서 Service가 하던 일을 Rails에서는 Model이 직접 한다. 이것이 Rails의 "Fat Model, Skinny Controller" 철학이다.

3레이어: Controller → 두꺼운 Service (비즈니스 로직 집중) → 빈 Entity
vs
DDD / Rails: Controller → 얇은 Service → 로직 가진 Domain Entity
