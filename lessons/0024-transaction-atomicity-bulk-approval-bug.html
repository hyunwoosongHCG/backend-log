실제로 본 에러

Sentry에 이런 에러가 반복해서 쌓였다:

NoMethodError: undefined method 'each' for nil

    change_data.each do |key, value|
               ^^^^^
  from app/services/objective_revision_diff.rb:17:in `call`

change_data는 @objective.pending_objective_revision&.data&.deep_symbolize_keys인데, 이게 nil이라는 뜻이었다. 즉 목표(objective)의 상태(stage)는 여전히 "승인 대기중"인데, 정작 승인할 대기 데이터(revision)는 이미 사라진 상태였다. 같은 유저가 재시도할 때마다 매번 같은 자리에서 죽었다 — 우연한 사고가 아니라 영구히 고장난 채로 멈춰있는 데이터였다는 뜻이다.

트랜잭션과 원자성(Atomicity)이란

DB 트랜잭션의 ACID 중 A(Atomicity, 원자성)는 "여러 개의 쓰기 작업을 하나의 작업 단위로 묶어서, 전부 성공하거나 전부 실패하거나 둘 중 하나만 있게 한다"는 원칙이다. Rails에서는 이걸 ActiveRecord::Base.transaction do ... end로 표현한다. 

ActiveRecord::Base.transaction do
  revision.update!(status: :accepted)   # ①
  objective.update!(stage: :open)       # ②
end
# ①까지 성공했는데 ②에서 예외가 나면?
# → 트랜잭션 전체가 롤백되어 ①도 취소된다. "반쯤 성공"은 없다.

반대로 이 블록으로 묶지 않으면, ①과 ②는 각각 독립된 SQL 문으로 즉시 커밋된다. ①이 끝나고 ②가 실패하면, ①은 이미 영구히 반영된 채로 남는다.

Q1. transaction do 블록으로 묶지 않은 두 개의 update!가 있다. 첫 번째는 성공하고 두 번째에서 예외가 났다. 결과는?

이 버그는 왜 생겼나 — 트랜잭션 없이 두 번의 UPDATE

목표 수정요청을 승인하는 코드(app/models/objective.rb:872)를 보면: 

def accept_update_and_create_history(current_user_id, diff_history)
  objective_revisions.where(status: :pending).update_all(status: :accepted)  # ① revision → accepted
  update(stage: :open)                                                       # ② objective → open
  create_history('UpdateAccepted', current_user_id, diff_history)            # ③ 히스토리 기록
end

이 메서드를 호출하는 단건 승인 API(PUT :id/update_approval, app/controllers/v2/objectives/objectives.rb:820)도, 그 전에 호출되는 ObjectiveApproveUpdate.call도 어디에도 transaction do로 감싸져 있지 않았다. 즉 ①과 ②는 서로 다른 트랜잭션(각자 즉시 커밋)이다. ①이 끝난 직후, 서버가 재시작되거나 타임아웃이 나거나 ②에서 유효성 검사가 실패하면 — revision은 이미 accepted로 확정되고, objective의 stage는 영원히 pending_update로 남는다.

프론트에서 여러 개의 fetch를 await로 순서대로 호출했는데, 첫 번째는 서버에 이미 반영됐고 두 번째에서 네트워크 에러가 났다고 생각해보자. 두 번째 실패를 캐치해서 사용자에게 에러를 보여줘도, 첫 번째 요청이 서버에 만든 부작용(side effect)은 이미 되돌릴 수 없다. Promise.all로 묶어도 이 문제는 똑같다 — Promise.all은 "동시에 실행"만 해줄 뿐, "하나 실패하면 나머지도 롤백"해주는 기능은 없다. DB 트랜잭션은 서버 쪽에서 이 롤백을 보장해주는 장치다.

update vs update! — 조용히 실패하는 함정

더 나쁜 건, accept_update_and_create_history의 ②가 update!가 아니라 update(느낌표 없음)라는 점이다. update!는 유효성 검사 실패 시 ActiveRecord::RecordInvalid 예외를 던지지만, update는 예외 없이 그냥 false만 반환한다. 

update(stage: :open)   # 검증 실패해도 예외 안 남, false만 반환
create_history(...)    # 반환값을 안 보니 이 줄은 그냥 실행된다

실제로 Objective 모델엔 이런 검증이 걸려있다(app/models/objective.rb:306-308): 

validate :started_time, :ended_time     # 시작일/종료일이 없으면 실패
validate :individual_validation         # 개인 목표가 아닌데 담당팀이 없으면 실패

이 조건 중 하나에 걸리면 ②는 에러 로그도, Sentry 알림도 없이 조용히 실패한다. 문제는 그 순간엔 아무도 모른다는 것 — 나중에 그 목표를 다시 승인 시도할 때가 되어서야 전혀 다른 에러(NoMethodError)로 터진다. 원인과 증상이 시간적으로도, 코드 위치로도 멀리 떨어져 있어서 추적이 어려웠던 이유다.

Q2. user.update(age: -1)가 유효성 검사에 걸려 실패했다. 반환값을 확인하지 않고 다음 줄을 실행하면?

그런데 일괄승인(bulk_accept)은 왜 안전했나

처음엔 "여러 개를 한 번에 승인하는 일괄 처리 중 하나만 실패한 케이스"라고 추측했는데, 확인해보니 오히려 반대였다. 일괄승인 로직(app/services/objective/admin/bulk_update_service.rb:92)은 이렇게 되어 있다: 

def bulk_accept
  ActiveRecord::Base.transaction do   # ← 루프 전체가 하나의 트랜잭션
    objectives.each do |obj|
      # ... revision accept, stage update 등
    end
  end
end

루프 전체가 트랜잭션 하나로 묶여 있어서, 여러 목표 중 하나를 처리하다 예외가 나면 배치 전체가 롤백된다 — 부분 성공이 없다. 즉 배치 API는 이미 원자성을 지키고 있었다. 문제는 트랜잭션이 없는 단건 승인 API 쪽이었고, 거기서 한번 고장난 데이터가 이후 배치 승인에서 다시 읽힐 때 크래시를 일으킨 것이다. "여러 개 중 하나 실패"가 아니라 "서로 다른 두 API의 원자성 보장 수준이 달랐던 것"이 진짜 원인이었다.

고치는 방법

단건 승인 경로에도 배치와 동일하게 트랜잭션을 씌우면 된다:

def accept_update_and_create_history(current_user_id, diff_history)
  ActiveRecord::Base.transaction do
    objective_revisions.where(status: :pending).update_all(status: :accepted)
    update!(stage: :open)   # update! 로 바꿔서 실패 시 확실히 예외로 드러나게
    create_history('UpdateAccepted', current_user_id, diff_history)
  end
end

핵심은 두 가지다: ① 여러 개의 관련된 쓰기를 트랜잭션으로 묶기, ② update 대신 update!를 써서 실패를 조용히 넘기지 않기. 이 둘이 같이 있어야 "일부만 반영된 상태"가 만들어질 수 없다.
