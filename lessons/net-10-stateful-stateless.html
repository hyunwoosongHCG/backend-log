Stateful, Stateless

네트워크 · HTTP 기본 강의 13 11:32
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 기본 — HTTP가 왜 Stateless인지, 그 설계의 이점과 한계를 이해한다

핵심 한 줄

HTTP는 무상태(Stateless) 프로토콜이다. 서버가 이전 요청을 전혀 기억하지 않으며, 클라이언트가 매 요청마다 필요한 정보를 모두 직접 담아 보내야 한다. 덕분에 서버를 무한히 늘릴 수 있다.

Stateful(상태 유지) vs Stateless(무상태)

Stateful — 서버가 상태를 기억한다

서버가 이전 대화를 기억하는 구조다. 마치 편의점 단골처럼, 지난번에 뭘 샀는지 직원이 기억하고 있어 "저번 거 또 주세요"만 말해도 된다. 

// Stateful 예시 — 서버가 상태를 들고 있다
// 요청 1
클라이언트: "노트북 사고 싶어요"
서버: (기억: 노트북) "어떤 모델이요?"

// 요청 2 (상태가 유지된 같은 서버에서 처리)
클라이언트: "맥북이요"
서버: (기억: 노트북 + 맥북) "몇 개요?"

// 요청 3
클라이언트: "2개요"
서버: (기억: 노트북 + 맥북 + 2개) "결제하시겠어요?"

문제는 중간 서버가 바뀌면 상태가 사라진다는 것이다. 서버 A가 기억하던 것을 서버 B는 모른다. 서버 A가 장애 나면 대화가 끊긴다.

Stateless — 서버가 상태를 기억하지 않는다

서버는 아무것도 기억하지 않는다. 클라이언트가 매 요청마다 맥락을 전부 담아서 보낸다. 

// Stateless 예시 — 클라이언트가 상태를 들고 다닌다
// 요청 1
클라이언트: "노트북 맥북 2개 사고 싶어요. 결제는 카드로."
서버: "결제 완료입니다." (이후 기억 안 함)

// 요청 2 (다른 서버가 처리해도 OK)
클라이언트: "노트북 맥북 2개 반품하고 싶어요. 주문번호 12345."
서버: "반품 처리했습니다." (이후 기억 안 함)
React 컴포넌트의 순수 함수(pure function)와 같다. 같은 입력(props)을 넣으면 항상 같은 출력이 나온다. 이전 호출을 기억하지 않는다. 서버도 마찬가지 — 이전 요청을 기억하지 않고, 요청에 담긴 정보만으로 응답을 결정한다. Authorization 헤더에 토큰을 매번 담아 보내는 것이 바로 이 때문이다.

Stateless의 핵심 장점 — 수평 확장(Scale Out)

서버가 상태를 갖지 않으면 어떤 서버든 어떤 요청이든 처리할 수 있다. 트래픽이 폭발하면? 서버를 그냥 더 추가하면 된다.

수많은 클라이언트 요청
↓
로드 밸런서(Load Balancer) — 요청을 분산
↓
서버 A · 서버 B · 서버 C … (무한 추가 가능)

서버 A가 요청 1을 처리했어도, 서버 B·C가 요청 2·3을 처리해도 아무 문제없다. 이것을 스케일 아웃(Scale Out) = 수평 확장이라 한다. 상태가 없기 때문에 가능한 설계다.

Stateless의 한계와 현실적 해결책

하지만 현실에서는 "상태"가 필요한 경우가 있다. 대표적인 예가 로그인이다.

상황 문제 해결책
로그인 유지 매 요청마다 아이디·비번을 보낼 수 없다 쿠키(Cookie) + 세션(Session) 또는 JWT 토큰
장바구니 이전에 담은 상품을 서버가 모른다 요청에 장바구니 ID를 담아 보냄
결제 진행 중 단계별 입력 내용을 서버가 모른다 클라이언트가 단계별 데이터를 들고 다님

핵심은 상태를 서버가 아닌 클라이언트나 별도 저장소(DB·Redis)에 두는 것이다. 서버 자체는 Stateless를 유지하면서, 상태는 클라이언트 쿠키·토큰이나 외부 저장소에 위임한다.

정리

  • Stateful: 서버가 이전 상태를 기억 → 서버 교체 불가, 장애에 취약
  • Stateless: 서버가 아무것도 기억 안 함 → 어떤 서버든 처리 가능, 수평 확장 쉬움
  • HTTP는 기본적으로 Stateless 설계
  • 로그인 같은 상태 유지는 쿠키·세션·토큰으로 해결 (서버를 Stateful로 만드는 게 아님)
