카프카 토픽(Topic)이란

레슨 025 Topic · Kafka · Pub/Sub · 토픽 네이밍 컨벤션 ~15분
🎯 "서비스 A가 서비스 B를 몰라도 데이터를 주고받을 수 있는 이유는?" — 카프카가 서비스 사이를 어떻게 갈라놓는지 이해한다 — MISSION.md

왜 토픽이 필요한가

theplus-back(Performance Plus 백엔드)은 HR 정보가 바뀌었다는 걸 어떻게 알까? HR 시스템에 직접 물어보러 가지 않는다. 대신 record-hr-organization-code라는 이름의 토픽(Topic)을 구독하고 있다가, 거기에 메시지가 오면 반응한다.

토픽은 카프카에서 메시지가 쌓이는 이름 붙은 채널이다. 발행자(Producer)는 "이 토픽에 이런 이벤트가 있었다"고 써넣고, 구독자(Consumer)는 자기가 관심 있는 토픽만 구독해서 읽는다. 발행자와 구독자는 서로의 존재를 몰라도 된다 — 오직 토픽 이름이라는 약속만 공유한다.

Kafka 토픽은 REST API 엔드포인트가 아니라 이벤트 이름에 가깝다.

window.addEventListener('user:created', handler)처럼, 누가 dispatchEvent를 호출했는지 몰라도 'user:created'라는 이벤트 이름만 알면 반응할 수 있다.
토픽 이름 = 이벤트 이름. 구독자는 발행자의 코드를 import할 필요가 없다.

실제 예시 — theplus-back의 토픽들

theplus-back/karafka.rb에는 이런 토픽들이 등록되어 있다 (일부만 발췌): 

# theplus-back/karafka.rb
routes.draw do
  consumer_group :perpl do
    topic :'optimiz-async' do
      consumer OptimizAsyncConsumer
    end
    topic :'record-hr-organization-code' do
      consumer HrmOrganizationInfoConsumer
    end
    topic :'user-system-module-status' do
      consumer UserSystemModuleStatusConsumer
    end
    # ... 20여 개 더
  end
end

토픽 하나 = 이벤트 종류 하나. optimiz-async엔 사람/조직/발령 관련 이벤트가, record-hr-organization-code엔 "조직 코드가 바뀌었다"는 이벤트만 온다. REST처럼 "리소스"로 나누는 게 아니라, "무슨 일이 일어났는지"로 나눈다.

Q1. 카프카 토픽을 가장 잘 설명한 것은?

토픽 이름은 곧 컨벤션이다

이 프로젝트에서 특히 흥미로웠던 건 토픽 이름에 "누가 발행했는지"를 박아넣는 규칙이었다. 사용자의 모듈 활성/비활성 이벤트를 예로 들면:

토픽 이름 발행 주체
user-performance-module-statusPerformance(theplus-back / people-sync)가 발행
user-system-module-statusSystem(optimiz-system)이 발행
user-time-module-statusTime(optimiz-time)이 발행

처음엔 이름이 너무 비슷해서 오타나 네이밍 버그처럼 보였다. 실제로는 optimiz-system의 CONST_SYSTEM_TOPIC.java에 이 셋이 나란히 상수로 정의돼 있었다 — 의도된 컨벤션이었다. 토픽 이름만 보고 "누가 이걸 썼는지" 유추할 수 있게 설계한 것.

Q2. user-performance-module-status와 user-system-module-status가 둘 다 존재하는 이유는?

토픽과 "-fail" 짝

theplus-back의 토픽은 대부분 -fail이 붙은 짝을 가진다 (optimiz-async / optimiz-fail, record-hr-organization-code / record-hr-organization-code-fail). 이건 실패한 메시지를 격리하는 DLQ(Dead Letter Queue) 패턴인데, 자세한 내용은 레슨 29에서 다룬다. 지금은 "토픽은 실패 처리용으로도 하나 더 만들 수 있다"는 것만 기억하면 된다.

핵심 요약

개념 역할
Topic 이름 붙은 이벤트 채널. 발행자/구독자를 서로 몰라도 되게 만드는 접점
토픽 분리 기준 리소스가 아니라 "무슨 일이 일어났는지"(이벤트 종류) 기준
네이밍 컨벤션 이 프로젝트는 토픽 이름에 발행 모듈명을 넣는다 (예: user-{모듈}-module-status)
