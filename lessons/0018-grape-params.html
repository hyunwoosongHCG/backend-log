출발점: 두 가지 400 경로

레슨 12에서 rescue_from 목록을 봤다:

rescue_from ActiveRecord::RecordInvalid,         with: :bad_request  # ← 레슨 14에서 완성
rescue_from Grape::Exceptions::ValidationErrors, with: :parameter_validate_error  # ← 오늘

RecordInvalid는 모델 레이어의 validates에서 온다 (레슨 14). ValidationErrors는 Grape가 API 입력을 검증하다 실패할 때 발생한다. 이 레슨에서 그 경로를 완성한다.

params do 블록 — API 파라미터 선언

Grape는 각 엔드포인트에 어떤 파라미터를 받는지 선언하는 DSL을 제공한다. 라우트 핸들러가 실행되기 전에 Grape가 이 선언대로 자동으로 검증한다. 

# app/controllers/v2/reviews/review_templates.rb
post do
  params do
    requires :name,        type: String,  desc: '템플릿 이름'
    requires :category_id, type: Integer, desc: '카테고리 ID'
    optional :description, type: String,  desc: '설명 (선택)'
    optional :status,      type: String,  default: 'draft', values: %w[draft active]
  end
  template = ReviewTemplate.create!(declared(params))
  present template, with: ReviewTemplateEntity
end
TypeScript의 인터페이스로 함수 인자를 타입 검사하는 것과 같다:

// TS: 함수 선언에 타입을 붙임
function createTemplate(args: { name: string; categoryId: number; description?: string }) { ... }

# Grape: 라우트에 파라미터 스펙을 붙임
params do
  requires :name,        type: String
  requires :category_id, type: Integer
  optional :description, type: String
end
둘 다 "이 인자가 없거나 타입이 다르면 에러"라는 규칙을 선언하는 방식이다.

requires vs optional

키워드 의미 없으면
requires 필수 파라미터 ValidationErrors 예외 발생 → 400
optional 선택 파라미터 nil 또는 default 값

주요 옵션들:

requires :age,    type: Integer                    # 타입 강제 (문자열 "3" → 정수 3 자동 변환)
optional :status, type: String, default: 'draft'  # 기본값
optional :role,   type: String, values: %w[admin member]  # 허용값 목록
requires :email,  type: String, regexp: /.+@.+/   # 정규식 검사
Q1. requires :name, type: String으로 선언된 파라미터를 빠뜨리고 요청하면?

declared(params) — 선언한 것만 통과시키기

params에는 클라이언트가 보낸 모든 값이 담긴다. declared(params)는 params do에서 선언한 키만 걸러낸 것이다. 

# 클라이언트가 보낸 것
{ name: "월간 리뷰", category_id: 3, hacker_field: "DROP TABLE users" }

# declared(params) — 선언한 것만 남음
{ name: "월간 리뷰", category_id: 3 }
# hacker_field는 선언하지 않았으므로 자동으로 제거됨

Rails의 Strong Parameters (params.require(:user).permit(:name, :email))과 같은 역할이다. 선언하지 않은 파라미터는 자동으로 걸러내서 의도치 않은 대입을 막는다.

Q2. declared(params)를 쓰는 이유는?

전체 400 에러 경로 정리

이제 400이 나오는 두 경로가 모두 완성됐다:

경로 A: 필수 파라미터 누락 또는 타입 불일치
→ Grape::Exceptions::ValidationErrors → rescue_from → 400
또는
경로 B: 파라미터는 맞지만 모델 validates 실패
→ ActiveRecord::RecordInvalid (save! 호출 시) → rescue_from → 400

두 검증의 차이: Grape params 검증은 API 경계에서 (입력 형식), validates는 모델 레이어에서 (비즈니스 규칙). 둘 다 있으면 두 겹으로 보호된다.

Q3. Grape::Exceptions::ValidationErrors와 ActiveRecord::RecordInvalid의 차이는?

핵심 요약

개념 역할
params do 엔드포인트가 받을 파라미터 선언
requires 필수 — 없으면 ValidationErrors → 400
optional 선택 — 없으면 nil 또는 default
declared(params) 선언한 파라미터만 필터링해서 반환
