.kept는 레슨 13에서 배운 where도 아니고 find_by도 아니다. 이것이 scope다.

두 번째 — 레슨 12의 rescue_from 목록:

rescue_from ActiveRecord::RecordInvalid, with: :bad_request  # 400

RecordInvalid는 어디서 발생하는가? 바로 validates에서 온다. 이 레슨에서 두 가지를 모두 완성한다.

파트 1: Scope — 자주 쓰는 where 조건에 이름을 붙이다

where(discarded_at: nil) 같은 조건을 매번 쓰는 건 번거롭다. scope는 이 조건에 이름을 붙여서 메서드처럼 호출하는 기능이다. 

# 정의
class Review < ApplicationRecord
  scope :kept,      -> { where(discarded_at: nil) }
  scope :discarded, -> { where.not(discarded_at: nil) }
  scope :submitted, -> { where(status: :submitted) }
end

# 호출
Review.kept                    # discarded_at이 nil인 것만
Review.kept.submitted          # 체이닝 가능 (Relation 반환이므로)
record.reviews.kept.exists?    # 레슨 12에서 봤던 그것

-> { ... }는 람다(lambda)다. "호출될 때 이 블록을 실행해라"는 의미인데, 지금은 그냥 "조건을 감싸는 껍데기"로 이해해도 충분하다.

JS에서 자주 쓰는 필터를 함수로 추출하는 것과 같다:

const activeUsers = users => users.filter(u => u.discardedAt === null)

scope도 "자주 쓰는 where 조건을 이름 붙인 것"이다. 차이는 scope는 Relation을 반환하므로 뒤에 계속 체이닝할 수 있다는 점.

실무에서: Discard gem의 .kept와 .discarded

이 프로젝트에서 레코드를 "삭제"할 때 실제로 DB에서 지우지 않는다. 대신 discarded_at 컬럼에 삭제 시각을 기록한다. 이를 소프트 삭제(Soft Delete)라고 한다.

scope 의미 SQL 조건
.kept "살아있는" 레코드 (삭제 안 됨) WHERE discarded_at IS NULL
.discarded "삭제된" 레코드 WHERE discarded_at IS NOT NULL

이 두 scope는 Discard gem이 자동으로 만들어 준다. 레슨 12의 !record.discarded?, reviews.kept.exists?가 모두 이 gem에서 오는 것이다.

파트 2: validates — 저장 전에 데이터를 검사한다

모델에 "이 컬럼은 반드시 있어야 한다", "이 값은 유일해야 한다" 같은 규칙을 선언한다. ActiveRecord는 save나 create 전에 이 규칙을 자동으로 검사한다. 

class ReviewTemplate < ApplicationRecord
  validates :name, presence: true
  validates :name, uniqueness: { scope: :workspace_id }
  validates :status, inclusion: { in: %w[draft active archived] }
end

자주 쓰는 검사 종류:

검사 의미
presence: true nil이거나 빈 문자열이면 안 됨
uniqueness: true 테이블 전체에서 중복 불가
inclusion: { in: [...] } 지정한 값 목록 안에 있어야 함
length: { maximum: 100 } 길이 제한

save vs save! — 검사 실패시 어떻게 되나

저장 메서드가 두 가지 버전으로 존재한다. 느낌표(!)가 있는 버전과 없는 버전이다. 

template = ReviewTemplate.new(name: nil)  # presence: true 위반

# ! 없는 버전 — 조용히 실패
template.save    # → false 반환, 예외 없음
template.errors.full_messages  # → ["Name can't be blank"]

# ! 있는 버전 — 예외를 발생시킨다
template.save!   # → ActiveRecord::RecordInvalid 예외 발생!

그래서 레슨 12의 rescue_from ActiveRecord::RecordInvalid, with: :bad_request가 잡는 예외가 바로 이것이다. 컨트롤러에서 record.save!나 Model.create!를 쓰면, validates가 실패했을 때 예외가 발생하고 → rescue_from이 잡아 → 400 Bad Request로 응답한다.

① 클라이언트가 name 없이 POST 요청
↓
② 컨트롤러에서 ReviewTemplate.create!(params)
↓ validates :name, presence: true 실패
③ ActiveRecord::RecordInvalid 예외 발생
↓ rescue_from 이 잡음
④ bad_request 메서드 → error!({ errors: [...] }, 400)
↓
⑤ 클라이언트에게 400 Bad Request 반환
프론트에서 form validation을 생각하면 된다:

if (!name) return setError('이름은 필수입니다')

validates는 이 검사를 모델 레이어에서 선언적으로 하는 것이다. save!는 "검사 통과 못 하면 throw해라"는 옵션이다.

연결: scope + validates + 레슨 12

이제 레슨 12의 Policy 코드를 완전히 읽을 수 있다:

def show?
  !record.discarded? &&  # Discard gem: discarded_at IS NOT NULL 이면 false
  admin?
end

def update?
  show? &&
  !record.reviews           # has_many 연관관계 (레슨 8)
               .kept        # scope: discarded_at IS NULL 인 것만 (오늘 배움)
               .exists?     # 존재 여부 체크 (레슨 13)
end

그리고 400이 나오는 경로:

# validates 위반 → save! → RecordInvalid → rescue_from → 400
rescue_from ActiveRecord::RecordInvalid, with: :bad_request

핵심 요약

개념 역할 실패하면
scope 자주 쓰는 where에 이름 붙이기 — (빈 Relation 반환)
validates 저장 전 데이터 규칙 검사 save → false / save! → RecordInvalid
RecordInvalid validates 실패 + ! 메서드 rescue_from → 400 Bad Request
