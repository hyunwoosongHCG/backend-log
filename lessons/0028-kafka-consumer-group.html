Consumer Group이란

theplus-back의 karafka.rb를 보면 모든 토픽이 하나의 consumer_group :perpl 블록 안에 등록돼 있다: 

# theplus-back/karafka.rb
routes.draw do
  consumer_group :perpl do
    topic :'optimiz-async' do
      consumer OptimizAsyncConsumer
    end
    topic :'record-hr-organization-code' do
      consumer HrmOrganizationInfoConsumer
    end
    # ... 20여 개 더
  end
end

Consumer Group은 "같은 이름표를 단 Consumer 프로세스들의 묶음"이다. 카프카 브로커는 그룹 이름(perpl) 단위로 "이 그룹이 각 토픽을 어디까지 읽었는지"(오프셋)를 기록한다. theplus-back 서버가 한 대든 여러 대든, 전부 perpl이라는 그룹으로 접속하면 브로커가 메시지를 중복 없이 나눠서 배분해준다.

일반적인 이벤트 발행/구독(Pub/Sub)과 헷갈리기 쉬운 지점이다.

window.addEventListener를 두 군데서 걸어두면, 이벤트가 발생했을 때 둘 다 실행된다 (fan-out). 카프카의 Consumer Group은 정반대다 — 같은 그룹에 속한 Consumer들은 서로 경쟁해서 메시지를 나눠 가진다 (하나의 메시지는 그룹 안에서 딱 한 번만 처리된다). 마치 여러 명의 작업자가 하나의 작업 큐에서 서로 다른 티켓을 꺼내가는 것과 같다 — 같은 티켓을 두 사람이 동시에 처리하지 않는다.
Q1. theplus-back 서버 프로세스가 3대 떠 있고 모두 Consumer Group perpl에 속해 있다. optimiz-async 토픽에 메시지 하나가 도착하면?

같은 토픽을 여러 그룹이 각자 읽을 수도 있다

반대로, 다른 그룹 이름을 쓰면 완전히 독립적으로 같은 토픽 전체를 읽을 수 있다. 예를 들어 theplus-back이 perpl 그룹으로 optimiz-async를 읽어서 실제 DB를 갱신하는 동안, 만약 별도로 perpl-analytics라는 그룹을 만들어 같은 토픽을 구독하면 그 그룹은 자기 페이스대로 전체 메시지를 처음부터 다시 읽을 수도 있다. "그룹이 다르면 서로 다른 시점(오프셋)에서 독립적으로 읽는다"가 핵심이다.

Q2. 같은 토픽을 서로 다른 두 Consumer Group이 구독하면?

이 프로젝트에서는 왜 그룹이 하나뿐인가

theplus-back은 지금 perpl 그룹 하나로 20여 개 토픽 전부를 소비한다. karafka.rb의 config.concurrency = 5는 "그룹 안에서 몇 개의 워커 스레드가 동시에 메시지를 처리할지"를 정하는 값이다 — 여러 그룹을 만드는 것과는 별개로, 한 그룹 안에서도 병렬 처리 정도를 조절할 수 있다. 지금 구조는 "하나의 애플리케이션 = 하나의 그룹" 형태라, 그룹을 여러 개로 쪼갤 필요가 아직 없었던 것으로 보인다 (트래픽이 늘거나, 토픽별로 배포 주기를 분리해야 할 때 나눌 수 있다).

핵심 요약

개념 역할
Consumer Group 같은 이름표를 단 Consumer들의 묶음. 브로커가 이 단위로 오프셋을 추적
그룹 내부 경쟁 소비 — 메시지 하나는 그룹 안에서 한 번만 처리 (중복 방지)
그룹 사이 완전 독립 — 서로 다른 그룹은 같은 토픽을 각자 페이스로 읽을 수 있다
