MSK란

MSK(Amazon Managed Streaming for Apache Kafka)는 AWS가 카프카 브로커를 대신 운영해주는 서비스다. 브로커 서버를 직접 띄우고, 패치하고, 스케일 조정하는 대신 AWS가 관리하는 클러스터에 접속만 하면 된다.

RDS/Aurora가 Postgres에게 해주는 것과 똑같은 일을 MSK가 카프카에게 해준다.

MSK : Kafka = RDS : Postgres. 프로토콜과 클라이언트 코드는 동일하고, "누가 서버를 관리하느냐"만 다르다. 로컬 Docker로 띄운 Postgres와 운영 Aurora가 SQL 쿼리 코드는 똑같듯이, 로컬 Kafka 컨테이너와 운영 MSK도 Producer/Consumer 코드는 똑같다 — 접속 설정만 바뀐다.

실제 예시 — 같은 코드, 다른 설정

theplus-back의 karafka.rb는 환경에 따라 접속 대상을 분기한다:

# theplus-back/karafka.rb
config.kafka = {
  'bootstrap.servers':
    Rails.env.development? ? 'host.docker.internal:9092'   # 로컬 Docker Kafka
                            : Server.credentials['kafka_broker_url']  # AWS MSK 엔드포인트
}

unless Rails.env.development?
  config.kafka['security.protocol'.to_sym] = 'SASL_SSL'
  config.kafka['sasl.mechanisms'.to_sym] = 'OAUTHBEARER'
  config.oauth.token_provider_listener = OauthTokenRefresher.new
end

로컬에서는 host.docker.internal:9092 — 인증 없이 접속 가능한 컨테이너 안의 카프카. 운영에서는 MSK 엔드포인트에 SASL_SSL + OAUTHBEARER(AWS IAM 기반 인증)로 접속한다. 같은 Karafka::App, 같은 Consumer 코드가 환경 변수 하나로 완전히 다른 인프라에 붙는다.

Q1. 로컬 개발 환경에서는 SASL_SSL 인증 없이 카프카에 붙는데, 운영에서는 왜 필요한가?

Lambda 쪽도 같은 인증 방식을 쓴다

theplus-aws-lambda의 people-sync도 MSK에 붙는다 — Ruby 대신 Node.js의 kafkajs + aws-msk-iam-sasl-signer-js를 쓰지만, 인증 방식(IAM 기반 OAUTHBEARER)은 theplus-back과 동일한 컨벤션이다. 언어가 달라도(Ruby ↔ Node.js) "같은 MSK 클러스터에 붙는다"는 규칙은 공유된다 — MSK가 인증 방식을 강제하기 때문이다. 

// theplus-aws-lambda/people-sync/peopleModelLayer/config/kafka.js
const kafka = new Kafka({
  brokers: process.env["kafka_broker_url"].split(','),
  ssl: true,
  sasl: {
    mechanism: "oauthBearer",
    oauthBearerProvider: () => this.oauthBearerTokenProvider("ap-northeast-2"),
  },
});
Q2. Ruby(theplus-back)와 Node.js(people-sync Lambda)가 서로 다른 언어인데 같은 MSK 클러스터에 붙을 수 있는 이유는?

핵심 요약

개념 역할
MSK AWS가 운영해주는 관리형 카프카 클러스터 (RDS의 카프카 버전)
bootstrap.servers 접속할 브로커 주소. 환경별로 로컬 Docker ↔ MSK 엔드포인트로 갈아끼움
SASL_SSL / OAUTHBEARER MSK 접속 시 필요한 IAM 기반 인증 방식. 언어와 무관하게 프로토콜만 맞으면 접속 가능
