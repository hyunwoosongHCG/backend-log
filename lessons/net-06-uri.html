핵심 한 줄

URI(Uniform Resource Identifier)는 인터넷 상의 자원을 식별하는 통일된 방법이며, 우리가 매일 쓰는 URL은 그 방법 중 "위치(Location)로 자원을 찾는" 형태다.

URI · URL · URN — 세 개념의 관계

이 세 용어는 혼용되는 경우가 많지만, 포함 관계가 명확하다.

  • URI (Uniform Resource Identifier) — 자원을 식별하는 상위 개념. URL과 URN을 모두 포함한다.
  • URL (Uniform Resource Locator) — 자원의 위치(Location)로 식별. 가장 흔하게 쓰는 형태.
  • URN (Uniform Resource Name) — 자원에 이름(Name)을 부여해 식별. 위치가 바뀌어도 이름은 유지된다는 장점이 있지만, 실무에서는 거의 쓰이지 않는다.

위치는 변할 수 있다 — 서버가 이전되거나 경로가 바뀌면 URL이 깨진다. URN은 그 문제를 해결하려 했지만, 인터넷 표준으로 자리잡지 못했다. 결과적으로 오늘날 "URI"를 말하면 사실상 "URL"을 뜻하는 경우가 대부분이다.

React에서 컴포넌트를 찾는 방법이 두 가지라고 상상해보자. 하나는 "파일 경로(src/components/Button.tsx)"로 찾는 것(→ URL), 다른 하나는 "컴포넌트 이름(Button)"으로 찾는 것(→ URN). 실무에서는 파일 경로(URL 방식)를 쓰지, 이름만으로 찾는 방식(URN)은 거의 안 쓴다.

URL 구조 완전 분해

예시 URL 하나를 기준으로 각 파트를 분해한다.

https://google.com/search?q=hello&hl=ko

전체 형식(Format)은 다음과 같다.

scheme://[userinfo@]host[:port][/path][?query][#fragment]

scheme — 어떤 방법으로 접근할지

주로 프로토콜(protocol)을 지정한다. http, https, ftp 등이 있으며, https는 http에 보안 계층(TLS/SSL)을 더한 것이다.

userinfo — 사용자 인증 정보

형식: user:password@. URL에 사용자 정보를 직접 포함하는 것은 보안상 좋지 않아 거의 쓰이지 않는다. GitLab이나 SSH 접속 시 드물게 볼 수 있는 정도다.

host — 어느 컴퓨터로 보낼지

도메인명(google.com) 또는 IP 주소(192.168.0.1)를 쓴다. DNS가 도메인을 IP로 변환해준다 — 이전 강의(IP 프로토콜)에서 배운 내용과 이어진다.

port — 그 컴퓨터의 어떤 앱으로 보낼지

IP가 컴퓨터의 논리 주소라면, 포트(port)는 그 컴퓨터에서 실행 중인 앱의 주소다. http의 기본 포트는 80, https는 443이다. 기본 포트는 URL에서 생략할 수 있다 — https://google.com:443과 https://google.com은 동일하다. Rails 개발 서버는 보통 3000번 포트를 쓴다.

path — 그 앱 안에서 어떤 자원인지

리소스의 경로를 계층 구조로 표현한다. 

/departments/rnd/members/catsbi

Rails 라우팅(resources :members)이 이 구조와 직결된다. GET /members/42 같은 경로가 바로 이 path 부분이다.

query — 추가 조건을 전달할 때

?로 시작하고, &로 여러 항목을 이어붙인다. key=value 쌍의 문자열로 이루어지며, query parameter 또는 query string이라고도 부른다. 서버에서는 이 값을 파싱해 검색 조건, 필터, 정렬 기준 등으로 활용한다. 

?q=hello&hl=ko
↑ q(검색어)=hello, hl(언어)=ko 두 파라미터
fetch('/api/members?page=2&sort=name')처럼 API 호출 시 URL에 붙이는 파라미터가 바로 query다. axios라면 params: { page: 2, sort: 'name' } 옵션이 내부적으로 이 형식으로 변환된다. Rails 컨트롤러에서는 params[:page], params[:sort]로 꺼내 쓴다.

fragment — 페이지 내 위치

#같이_보기처럼 HTML 문서 내의 특정 위치(앵커)를 가리킨다. 서버로 전송되지 않는다 — 브라우저가 로컬에서 스크롤 위치를 처리한다. 따라서 서버 로그에는 fragment가 보이지 않는다.

React Router의 해시 라우팅(/#/about)이나, 마크다운 문서의 #section-name 링크가 바로 fragment다. "서버로 안 간다"는 특성 때문에 SPA 초기에 클라이언트 라우팅 방법으로 해시를 많이 썼다.

전체 구조 한눈에 보기

파트 역할 예시 생략 가능?
scheme 접근 프로토콜 https 불가
userinfo 인증 정보 user@ 가능 (거의 생략)
host 대상 컴퓨터 google.com 불가
port 대상 앱 포트 :443 가능 (기본값 사용)
path 자원 경로 /search 가능
query 추가 파라미터 ?q=hello 가능
fragment 페이지 내 위치 #section 가능 (서버 미전송)

정리

  • URI ⊃ URL ⊃ (실무에서 URL = URI로 혼용)
  • URL 형식: scheme://[userinfo@]host[:port][/path][?query][#fragment]
  • port = 컴퓨터 안 앱의 주소 (http=80, https=443, Rails dev=3000)
  • query = ?key=value&key2=value2 — 서버에 추가 조건 전달
  • fragment = #anchor — 브라우저 로컬 처리, 서버에 전송되지 않음
