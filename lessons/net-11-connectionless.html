비 연결성 (connectionless)

네트워크 · HTTP 기본 강의 14 08:43
🎯 김영한 「모든 개발자를 위한 HTTP 웹 기본 지식」 · HTTP 기본 — 연결을 유지하지 않는 HTTP의 설계와 지속 연결로 해결한 이유를 이해한다

핵심 한 줄

HTTP는 요청-응답이 끝나면 즉시 연결을 끊는다. 연결 유지 자체에도 서버 자원이 든다는 이유다. 단, 자원마다 연결·종료를 반복하는 낭비는 지속 연결(Persistent Connections)로 해결했다.

연결 유지(Connection) vs 비 연결성(Connectionless)

연결 유지 방식 — 연결을 계속 열어 둔다

서버가 클라이언트와의 연결을 요청이 없어도 계속 유지하는 방식이다. 연결 상태를 유지하는 동안 서버 자원(소켓·메모리·스레드)이 소비된다. 동시 접속자가 10만 명이라면, 10만 개의 연결이 계속 살아 있다. 

// 연결 유지 — 서버가 연결을 붙잡고 있다
클라이언트 A  ←— 연결 유지 중 ——→  서버 (자원 점유)
클라이언트 B  ←— 연결 유지 중 ——→  서버 (자원 점유)
클라이언트 C  ←— 연결 유지 중 ——→  서버 (자원 점유)
// 요청을 하지 않는 시간에도 자원이 묶인다

비 연결성 — 요청-응답 후 즉시 끊는다

HTTP의 기본 설계: 요청을 처리하고 응답을 돌려주면 바로 연결을 끊는다. 서버는 수천 명이 동시 접속해도 실제 요청을 처리하는 순간에만 자원을 쓴다. 사용자 대부분은 어떤 순간에도 "대기 중"이기 때문에 이 방식이 훨씬 효율적이다. 

// 비 연결성 — 요청-응답 후 바로 끊는다
클라이언트 A → [요청] → 서버 → [응답] → 연결 종료
클라이언트 B → [요청] → 서버 → [응답] → 연결 종료
// 처리하는 순간에만 자원을 쓴다
콜센터 전화 상담과 같다. 상담원이 전화를 끊지 않고 계속 연결을 유지하면, 수백 명이 동시에 대기해도 전화를 받은 상담원은 한 명을 붙잡고 있는 동안 다른 고객을 받을 수 없다. 반면 질문-답변 후 전화를 끊으면, 같은 상담원이 훨씬 많은 고객을 처리할 수 있다. HTTP의 비 연결성이 바로 이 논리다.

비 연결성의 단점 — 자원마다 연결을 새로 맺는다

연결을 끊는다는 것은, 다음 요청 때 다시 연결을 맺어야 한다는 뜻이다. TCP 연결을 맺으려면 3-way handshake(SYN → SYN-ACK → ACK)가 필요하다. 웹 페이지 하나를 불러올 때 필요한 자원을 생각해보자. 

// 비 연결성의 단순한 모델
GET /index.html  → [3-way handshake] → 응답 → 연결 종료
GET /style.css   → [3-way handshake] → 응답 → 연결 종료
GET /app.js      → [3-way handshake] → 응답 → 연결 종료
GET /logo.png    → [3-way handshake] → 응답 → 연결 종료
// 자원 4개에 handshake 4번 — 낭비!

자원마다 연결을 새로 맺으면 handshake 오버헤드가 누적된다. 특히 HTML·CSS·JS·이미지가 수십 개인 현대 웹에서는 심각한 성능 문제가 된다.

해결책 — HTTP 지속 연결 (Persistent Connections)

HTTP/1.1부터 지속 연결(Persistent Connections, Keep-Alive)이 기본값이 됐다. 한 번 연결을 맺고, 필요한 자원들을 그 연결로 모두 요청·응답한 뒤, 마지막에 한 번만 끊는다. 

// 지속 연결 (Persistent Connections / Keep-Alive)
[3-way handshake — 한 번만]
  GET /index.html → 응답
  GET /style.css  → 응답
  GET /app.js     → 응답
  GET /logo.png   → 응답
[연결 종료 — 한 번만]
// handshake는 처음 한 번. 자원 4개를 같은 연결로 처리
연결 수립 (3-way handshake) — 1회
↓
HTML 요청·응답
↓
CSS · JS · 이미지 … 요청·응답 (같은 연결 재사용)
↓
연결 종료 — 1회

HTTP/2는 여기서 더 나아가 한 연결에서 여러 요청을 동시에 보내는 멀티플렉싱(Multiplexing)을 지원한다.

정리

  • 비 연결성: 요청-응답 완료 후 즉시 연결 종료 → 서버 자원 절약
  • 단점: 요청마다 3-way handshake 필요 → 자원이 많을수록 낭비
  • 지속 연결(Keep-Alive): 한 연결에서 여러 자원을 주고받고 마지막에 종료
  • HTTP/1.1부터 지속 연결이 기본값, HTTP/2는 멀티플렉싱으로 더 발전
