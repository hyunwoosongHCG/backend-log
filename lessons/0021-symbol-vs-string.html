Ruby Symbol — :foo가 도대체 뭔가

레슨 021 Symbol · String · 인터닝(interning) ~15분
🎯 ppback `talent_sessions/start_worker.rb`에서 %i[feedbackpool multisource_feedbacks]를 보고 "이거 그냥 문자열 아닌가?" 싶었던 것에서 시작 — MISSION.md

실제로 본 코드

ppback 리뷰 중 이런 코드를 봤다:

talent_session_reference_data: { reference_data: %i[feedbackpool multisource_feedbacks] },

:feedbackpool, :multisource_feedbacks — 콜론이 붙은 이 값들이 Symbol이다. JS만 해온 입장에서는 낯설다. 문자열이랑 뭐가 다른가?

Symbol과 String, 겉모습은 비슷하다

:foo      # Symbol
"foo"     # String

둘 다 "foo"라는 이름/텍스트를 나타내는 것처럼 보인다. 하지만 역할이 다르다. String은 화면에 보여줄 텍스트, 사용자 입력, 바뀔 수 있는 데이터에 쓰고, Symbol은 코드 내부에서만 쓰는 고정된 이름표(해시 키, enum 값, 메서드 이름)에 쓴다.

핵심 차이 — 인터닝(interning)

Symbol은 불변(immutable)이고, 코드 어디에 몇 번을 쓰든 항상 완전히 같은 객체다. String은 리터럴을 새로 쓸 때마다 별개의 객체가 생긴다. object_id로 직접 확인하면 차이가 드러난다: 

:foo.object_id == :foo.object_id     # => true  (항상 같은 객체)
"foo".object_id == "foo".object_id   # => false (매번 새 객체)

같은 객체라는 건 비교(==)가 문자를 한 글자씩 훑을 필요 없이 정수 비교급으로 빠르다(O(1))는 뜻이다. 반복적으로 비교하거나 분기(case, include?)에 쓰기에 String보다 유리한 이유다.

Q1. :status.object_id == :status.object_id의 결과는?
⚠️ JS의 Symbol()과 이름은 같지만 동작은 정반대다:

// JS: 호출할 때마다 유일한 값을 만든다
Symbol('foo') === Symbol('foo')   // false

# Ruby: 항상 같은 값을 재사용한다 (인터닝)
:foo == :foo                      # true
이름만 같고 정반대로 동작하는 개념이라 그냥 별개로 기억하는 게 낫다. 굳이 비유하자면 Ruby Symbol은 const STATUS = { ACTIVE: 'active' }처럼 고정된 enum 값을 언어가 알아서 인터닝(캐싱)해주는 것에 더 가깝다.

축약 문법 두 가지 뜯어보기

맨 처음 코드로 돌아가면, 사실 축약 문법이 두 겹으로 겹쳐 있었다: 

talent_session_reference_data: { reference_data: %i[feedbackpool multisource_feedbacks] },
#                             ↑ 해시 키 축약형        ↑ 심볼 배열 리터럴 축약형
# 1. 해시 키 축약형 — key: value는 :key => value의 축약
talent_session_reference_data: value
# 동일:
:talent_session_reference_data => value

# 2. 심볼 배열 리터럴 — %i[...]는 [:a, :b]의 축약
%i[feedbackpool multisource_feedbacks]
# 동일:
[:feedbackpool, :multisource_feedbacks]

%i[...]는 공백으로만 구분하면 되고 따옴표·콜론·쉼표를 안 붙여도 된다. "고정된 식별자 목록"을 나열할 때 자주 쓰는 문법이다.

Q2. %i[a b c]와 완전히 같은 표현은?

왜 String이 아니라 Symbol을 쓰나

feedbackpool, multisource_feedbacks는 사용자 입력이나 화면에 보여줄 텍스트가 아니라, "어떤 참조 데이터 종류를 같이 복사할지"를 코드 내부에서만 구분하는 고정된 태그다. 이런 경우 Symbol이 관용적이다:

  • 고정된 내부 식별자(enum 성격) — 값이 바뀔 일이 없다
  • 반복 비교·분기에서 String보다 빠르고 메모리도 덜 쓴다
  • 불변이라 실수로 값이 바뀌는 사고가 안 난다

단, JSON으로 직렬화되는 순간 Symbol은 String으로 바뀐다. 그래서 Symbol은 프로세스 내부통신에서만의미가 있고, API 응답처럼 프로세스 경계를 넘는 데이터는 보통 String을 쓴다.

Q3. 다음 중 Symbol을 쓰는 게 더 적절한 경우는?

핵심 요약

항목 Symbol (:foo) String ("foo")
가변성 불변(immutable) 가변(mutable)
동일 리터럴 재사용 항상 같은 객체 (인터닝) 매번 새 객체
비교 속도 O(1) 글자 수만큼
주 용도 해시 키, enum 값, 메서드 이름 사용자 입력, 화면 표시 텍스트, API 페이로드
